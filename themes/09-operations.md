# 第9章: 運用支援サービス

## 座学

### CloudWatch

- AWSが提供する**モニタリング・可観測性（オブザーバビリティ）**サービス。AWSリソース（EC2、RDS、Lambdaなど）やアプリケーションの状態を**収集・可視化・通知**するための統合サービス。
- 例えるなら、**工場の中央制御室**。各機械（AWSリソース）の稼働状況を計器（メトリクス）で監視し、異常があれば警報（アラーム）を鳴らし、運転ログ（ログ）を保存し、決まった時刻に点検（イベント）を自動実行する。
- CloudWatchは大きく **CloudWatch Metrics / CloudWatch Alarm / CloudWatch Logs / CloudWatch Events（現EventBridge）** の4つの機能から構成される。以下で1つずつ見ていく。

#### CloudWatch Metrics（メトリクス）

- CloudWatchにおける **「数値データの収集・保存」** を担う機能。AWSリソースやアプリから送られてくる数値（CPU使用率、ネットワーク転送量、ディスクI/O、リクエスト数など）を**時系列データ**として記録する。
- メトリクスは **名前空間（Namespace）** / **ディメンション（Dimension）** / **タイムスタンプ** で管理され、"どのリソースの" "どの値が" "いつ" 記録されたかを識別できる。
- CloudWatch Metricsに送信されるメトリクスには **「標準メトリクス」** と **「カスタムメトリクス」** の2種類がある。

##### 標準メトリクス

- **CloudWatch Metricsの1種類**。AWSが**自動的に収集して**CloudWatch Metricsに送信してくれるメトリクス。EC2のCPU使用率、RDSの接続数、ELBのリクエスト数など、**追加設定なしで利用できる**。
- **重要な注意点**: EC2の **メモリ使用率** や **ディスク使用率** は標準メトリクスには**含まれない**。これはOSの内部情報のため、ハイパーバイザー側（AWS側）から見えないことが理由。これらを取得するには、次のカスタムメトリクスを使う必要がある。

##### カスタムメトリクス

- **CloudWatch Metricsの1種類**。利用者が**独自に定義**してCloudWatch Metricsへ送信するメトリクス。
- **CloudWatch Agent**をEC2にインストールすれば、メモリ使用率・ディスク使用率・プロセス数などのOSレベルの値をカスタムメトリクスとして送れる。
- アプリケーション独自の指標（注文数、エラー数、平均処理時間など）もカスタムメトリクスとして送信できる。
- **試験で問われるポイント**: 「EC2のメモリ使用率を監視したい」→ **CloudWatch Agent**をインストールしてカスタムメトリクスとして送信する。

#### CloudWatch Alarm（アラーム）

- CloudWatchにおける **「閾値監視と通知」** を担う機能。**CloudWatch Metricsで収集された値**を監視し、**設定した閾値を超えた（または下回った）ときに通知やアクションを実行**する。
- 例: "EC2のCPU使用率が70%を5分以上継続したらSNSトピック経由でメール通知" "RDSの空き容量が10%を下回ったらAuto Scalingを発動"
- **アラームの状態**は3種類。
  - `OK`: 閾値以内で正常
  - `ALARM`: 閾値を超過して異常
  - `INSUFFICIENT_DATA`: データ不足で判断できない
- **アラームから実行できるアクション**: SNSトピック経由の通知（メール・SMS・スマホプッシュ）、Auto Scalingのスケールイン/アウト、EC2の停止・再起動、Lambda関数の実行など。
- CloudWatch Metrics（測る）とCloudWatch Alarm（閾値で通知する）はセットで使うのが基本形。

#### CloudWatch Logs（ログ）

- CloudWatchにおける **「ログの収集・保存・検索」** を担う機能。AWSリソースやアプリケーションの**テキストログ**を一元管理する。
- 対象となるログ例: EC2のアプリログ、Lambdaの実行ログ、VPCフローログ、RDSのスロークエリログ、API Gatewayのアクセスログなど。
- **ロググループ**: CloudWatch Logsにおけるログの分類単位（例: `/aws/lambda/my-function`）。
- **ログストリーム**: 同一ソースからのログの系列（例: EC2インスタンスごとのログの流れ）。
- **メトリクスフィルター**: CloudWatch Logsに取り込まれたログから特定パターン（"ERROR" という文字列など）を抽出して**CloudWatch Metricsのメトリクスに変換**する機能。変換したメトリクスをCloudWatch Alarmで監視することで、"ログにエラーが3回出たら通知" のような仕組みが作れる。
- **CloudWatch Logs Insights**: ログに対して**SQLライクなクエリ言語**で検索・集計できる機能。
- **保持期間**: デフォルトは**無期限保存**。コスト削減のため用途に応じて1日〜10年の範囲で設定可能。

#### CloudWatch Events

- CloudWatchにおける **「AWSリソースの状態変化やスケジュールをトリガーに自動アクションを実行する仕組み」**。
- 例えるなら、**「〇〇が起きたら△△する」という自動化ルール**の集合体。
- 旧名称。現在は後述の **Amazon EventBridge** に統合・リブランドされており、マネジメントコンソールからはEventBridge側で操作する。試験では両方の名称で出題される可能性あり。
- 構成要素として **イベントソース / ターゲット / イベント / スケジュール** の4つを押さえる。

##### イベントソース

- **CloudWatch Eventsにおける「イベントの発生源」**。何をきっかけにルールを動かすかを指定する。
- 例: EC2の状態変化（起動・停止）、S3へのオブジェクトアップロード、CodeCommitへのプッシュ、AWS API呼び出しなど。

##### ターゲット

- **CloudWatch Eventsにおける「イベント発生時に実行される処理の宛先」**。アクションとして何を呼び出すかを指定する。
- 例: Lambda関数の実行、Step Functionsワークフロー起動、SNSトピックへの通知、SQSへのメッセージ送信、EC2インスタンスの再起動など。

##### イベント

- **CloudWatch Eventsにおける「ソースから発生する1つ1つの出来事」**。JSON形式で表現され、「何が・いつ・どこで起きたか」の情報を持つ。ターゲットにはこのJSONがそのまま渡される。
- **混同注意**: 後述の **CloudTrailの「イベント」はAPI呼び出し履歴の1件1件** を指す別物。CloudWatch Eventsのイベントは「何かが起きた」を示す通知メッセージで、**CloudTrailのイベント**は「誰がいつ何のAPIを呼んだか」の記録。名前は同じだが役割が違うので試験でも要注意。

##### スケジュール

- **CloudWatch Eventsにおける「時間ベースのトリガー」**。**cron式**または**rate式**で定期実行を定義する。
- 例: "毎日午前3時にバックアップ用Lambdaを実行" "5分ごとに死活監視関数を起動"。
- Linuxのcronと同じ感覚で、AWSのサーバーレスな定期実行を実現できる。

#### EventBridge

- **CloudWatch Eventsを包含・拡張**したサーバーレスイベントバスサービス。CloudWatch Eventsの全機能に加えて、以下の強化機能を持つ。
  - **イベントバス**: AWS公式 / SaaS / カスタムアプリケーションの3種類のバスを使い分けられる
  - **SaaSイベントソース**: Zendesk、PagerDuty、DatadogなどのSaaSアプリからのイベントを直接受け取れる
  - **スキーマレジストリ**: イベントのスキーマを自動検出・登録してコード生成に活用できる
  - **入力変換**: イベント内容を加工してターゲットに渡せる
- **試験で問われるポイント**: 「cronで定期処理を実行したい」→ **EventBridge**（旧CloudWatch Events）。「SaaSアプリからのイベントを受け取って自動処理したい」→ **EventBridge**。

### CloudTrail

- AWSアカウント内で実行された **API呼び出し（誰が・いつ・何をしたか）を記録**する監査ログサービス。セキュリティ監査、コンプライアンス対応、トラブルシューティングの基盤となる。
- 例えるなら、**オフィスの入退室記録 + 金庫の開閉ログ**。「誰が・いつ・何をしたか」を後から追跡できる仕組み。
- AWSアカウントを作成すると**自動的に有効化**され、**過去90日間**のイベント履歴が無料で確認できる。**長期保存**やS3エクスポート、組織全体でのログ集約には**証跡（Trail）**の作成が必要。
- **CloudWatch Logsとの違い**: CloudWatch Logsは「アプリが動いた結果のログ」。CloudTrailは「**利用者（人間やシステム）がAWSに対して行った操作**のログ」。視点の向きが違う。
- CloudTrailで記録されるイベントは **管理イベント** と **データイベント** の2種類に分類される。

#### 管理イベント

- **CloudTrailのイベント種別の1つ**。AWSリソースの **管理操作**（作成・変更・削除・設定変更）を記録するイベント。いわゆる**コントロールプレーン**の操作。
- 例: EC2の起動、IAMユーザーの作成、S3バケットの作成、セキュリティグループの変更など。
- **デフォルトで記録される**ので、特別な設定なしに監査ログとして利用できる。

#### データイベント

- **CloudTrailのイベント種別の1つ**。S3オブジェクトの読み書き（GetObject/PutObject）、Lambda関数の実行（Invoke）、DynamoDBのアイテム操作など、**データそのものへのアクセス**を記録するイベント。いわゆる**データプレーン**の操作。
- 大量に発生するため **デフォルトでは記録されない**。明示的に有効化する必要があり、有効化すると**追加料金**が発生する。
- **試験で問われるポイント**: 「S3バケットへのPutObject/GetObjectを記録して監査したい」→ CloudTrailの**データイベント**を有効化する。

### AWS Config

- AWSリソースの**設定状態（構成）を記録**し、**望ましい設定に準拠しているかを継続的に評価**するサービス。コンプライアンスとガバナンスの中核。
- 例えるなら、**社内ルール遵守チェッカー**。「全社員がIDカードを装着しているか」「書類が所定の場所に保管されているか」を定期巡回してチェックし、違反があれば報告する。
- リソースの**構成変更の履歴**もタイムラインとして保存される。「このS3バケットはいつ公開設定になったか」などを遡って確認できる。
- **CloudTrailとの違い**: CloudTrailは「**誰が・何をしたか**」の操作ログ（監査）。Configは「**今の設定が正しいか**」の評価＋構成履歴（コンプライアンス）。試験では両者の役割分担がよく問われる。

#### Configルール

- **AWS Configにおける「評価ルール」**。リソースの設定が**望ましい状態にあるか**を評価するための条件定義。
- 評価結果は **COMPLIANT（準拠）** または **NON_COMPLIANT（非準拠）** で記録される。
- 例: "全S3バケットは暗号化されているか" "全EC2インスタンスに必須タグが付いているか" "セキュリティグループで22番ポートが全開放（0.0.0.0/0）されていないか"。
- Configルールは **マネージドルール** と **カスタムルール** の2種類から選べる。

##### マネージドルール

- **Configルールの1種類**で、AWSが**事前定義**して提供しているルール集。100種類以上が用意されており、選んで有効化するだけで使える。
- 例: `s3-bucket-public-read-prohibited`（S3バケットが公開されていないこと）、`restricted-ssh`（SSHの全開放禁止）、`required-tags`（必須タグの存在） など。
- 一般的なセキュリティ・コンプライアンス要件の多くはマネージドルールでカバーできる。

##### カスタムルール

- **Configルールの1種類**で、利用者が**独自のロジック**で定義するルール。**Lambda関数**または**AWS Config Guard**（Guard言語）で評価ロジックを記述する。
- マネージドルールではカバーできない**社内独自の要件**に対応したいときに使う。

#### 修復アクション（Remediation）

- **AWS Configの機能**で、Configルールで**非準拠が検出されたときに自動実行される修復処理**。違反状態を自動的に正しい状態に戻せる。
- 例: "公開設定になったS3バケットを自動で非公開に戻す" "必須タグが欠落したEC2に自動でタグを付与する" "セキュリティグループの0.0.0.0/0の22番ポートを自動で閉じる"。
- 修復処理の実行基盤として、次の **Systems Manager Automation** を利用する。

##### Systems Manager Automation

- **Systems Manager（SSM）の機能の1つ**で、AWSリソースに対する**運用作業をドキュメント化して自動実行**する仕組み。
- 事前定義された**オートメーションドキュメント（Runbook）**を使って、EC2パッチ適用、AMI作成、インスタンス再起動、S3バケットの暗号化有効化などの定型作業を自動化できる。
- **AWS Configとの関係**: Configの修復アクションは、内部的にこのSSM Automationを呼び出して修復処理を実行する。つまり **「Configルール違反 → SSM Automationが自動修復」** という運用自動化の流れが組める。Config単体ではなく、SSM Automationと組み合わせることで"検知から修復まで"が完結する点を押さえる。

### CloudFormation

- AWSリソースを**コードで定義し、自動的に構築・更新・削除**できるサービス。いわゆる **IaC（Infrastructure as Code）** をAWS公式に実現するツール。
- **IaC（Infrastructure as Code）とは**: VPC・サブネット・EC2・セキュリティグループなどの**インフラ構成をコード（テキストファイル）として記述し、ソフトウェア開発と同じやり方でインフラを管理する考え方**。マネジメントコンソールでのクリック操作や、Excelの手順書に依存した構築から脱却し、**コードを"唯一の構成情報源（Single Source of Truth）"** として扱う。
  - **主なメリット**: ① **再現性** — 同じコードから何度でも同じ環境を作れる（開発/本番の構成ズレがなくなる） ② **バージョン管理** — Gitで変更履歴・差分・レビューが可能 ③ **自動化** — CI/CDに組み込んで変更を自動反映 ④ **ドキュメント化** — コード自体が最新の構成ドキュメントになる（属人化の解消）。
  - **IaCツールの代表例**: **CloudFormation**（AWS公式・YAML/JSON）、**Terraform**（HashiCorp社製・マルチクラウド対応でデファクト）、**AWS CDK**（TypeScript/Python/Javaなどのプログラミング言語でインフラ記述）。AWSの試験で「IaC」と来たらまず **CloudFormation** を選ぶ。
- 例えるなら、**建築設計図**。同じ設計図を渡せば、何度でも同じ建物（環境）を作れる。手作業による構築ミスを減らし、環境の再現性を高められる。
- **どんな場面で使う？**: 開発/ステージング/本番の3環境を同じ構成で用意したい、災害復旧用に別リージョンへ同じ環境を短時間で再構築したい、インフラの変更履歴をGitでバージョン管理したい、など。
- CloudFormationを理解するには **テンプレート** と **スタック** の2つの概念を押さえる。

#### テンプレート

- **CloudFormationにおける「AWSリソースの構成を記述した設計ファイル」**。**YAML**または**JSON**形式で書く（YAMLの方が可読性が高く主流）。
- 主な構成要素：
  - **Parameters**: スタック作成時に外部から渡す値（インスタンスタイプ、キーペア名など）
  - **Resources**: 実際に作成するAWSリソース定義（EC2、VPC、S3など）。**このセクションのみ必須**
  - **Outputs**: スタック作成後に取得できる値（作成したEC2のパブリックIPなど）
  - **Mappings** / **Conditions**: リージョン別の設定切替や条件分岐
- テンプレートを編集して再適用することで、**差分だけ**を安全に更新できる。**変更セット（Change Set）**を使えば、実際に適用する前に変更内容をプレビューできる。

#### スタック

- **CloudFormationにおける「1つのテンプレートから作成されたリソースのグループ」**。テンプレートが設計図なら、スタックは設計図から建てた**1軒の建物**に相当する。
- スタックを削除すると、そのテンプレートで作成されたリソースが**一括で削除**される。個別にリソースを削除する手間がなく、**クリーンアップが簡単**なのが大きなメリット。検証環境の作成・破棄が楽になる。
- 発展機能:
  - **ネストされたスタック**: 親スタックから子スタックを呼び出して、共通部品を再利用
  - **StackSets**: 複数アカウント・複数リージョンへの**一括展開**
- **試験で問われるポイント**: 「同じ構成の環境を複数リージョンに短時間で展開したい」→ **CloudFormation**。「インフラ構成をコードで管理・バージョン管理したい」→ **CloudFormation**。

### Systems Manager（SSM）

- EC2やオンプレミスサーバーを含む**AWSリソース全般の運用管理**を統合するサービス群。パッチ適用、リモート操作、構成管理などを一元的に行える。
- 例えるなら、**大規模IT環境の運用ツールボックス**。複数の運用作業を1つの画面から一括管理できる。
- SSMには多数の機能があるが、ここでは主要4つ（**Run Command / Patch Manager / Session Manager / Inventory**）を扱う。なお、Configの修復で使う **SSM Automation** も SSMの機能の1つ。
- **前提条件**: 管理対象のEC2には **SSM Agent** がインストールされている必要がある（Amazon Linux 2023などでは標準搭載）。さらにEC2のIAMロールに **`AmazonSSMManagedInstanceCore`** ポリシーが付与されている必要がある。

#### Run Command

- **Systems Managerの機能の1つ**で、複数のEC2インスタンスに対して**コマンドをリモート実行**する機能。SSHで1台ずつログインして実行する必要がなく、数百台に同じコマンドを**一括送信**できる。
- 例: "全EC2で `yum update` を実行" "特定のタグを持つインスタンスだけで設定ファイルを再読み込み"。
- 実行結果は**一元的に確認**でき、どのインスタンスで成功/失敗したかをマネジメントコンソールで把握できる。

#### Patch Manager

- **Systems Managerの機能の1つ**で、EC2やオンプレミスサーバーの**OSパッチ適用を自動化**する機能。パッチベースラインを定義し、スケジュールに基づいて一括適用できる。
- **メンテナンスウィンドウ**と組み合わせることで、業務影響の少ない時間帯（深夜など）に自動実行できる。
- コンプライアンス要件（例: "全サーバーは30日以内に最新パッチを適用すること"）への対応に使う。

#### Session Manager

- **Systems Managerの機能の1つ**で、EC2インスタンスに**ブラウザまたはAWS CLIからセキュアに接続**できる機能。**SSH鍵・踏み台サーバー・インバウンドポート開放が不要**。
- セキュリティ上のメリット：
  - SSHポート（22）を開ける必要がない → **攻撃面の削減**
  - **キーペアの管理が不要**
  - セッションログをCloudWatch LogsやS3に**自動記録** → 監査対応
- **試験で問われるポイント**: 「SSHポートを開けずに、踏み台サーバーなしでEC2にアクセスしたい」→ **Session Manager**。

#### Inventory

- **Systems Managerの機能の1つ**で、管理対象サーバーの**ソフトウェア構成情報を自動収集**する機能。
- 収集できる情報: インストール済みアプリケーション、Windows更新プログラム、ネットワーク構成、実行中のサービス、カスタム項目 など。
- どのサーバーに何が入っているかを一覧化・検索できるので、**脆弱性調査**や**資産管理**、**ライセンスコンプライアンス**の用途に使える。

### コスト管理サービス

- AWSでは従量課金の特性上、**コストの可視化・予算管理・最適化**のための専用サービス群が用意されている。ここでは代表的な4サービスを押さえる。
- 例えるなら、**家計管理アプリ**。先月いくら使ったか（Cost Explorer）、今月の予算を超えそうか（Budgets）、無駄遣いしている契約はないか（Compute Optimizer / Trusted Advisor）を見える化する。
- ⚠️ **名称の注意**: **Cost Explorer** と **Budgets** は独立したサービスではなく、**「AWS Billing and Cost Management」サービスの中の機能**。コンソールでも「Billing and Cost Management」から開く。ただし試験・ドキュメント・実務会話ではそれぞれ「Cost Explorer」「Budgets」という機能名で呼ぶのが一般的。

#### Cost Explorer（Billing and Cost Managementの機能）

- Billing and Cost Management 内の **コスト可視化機能**。AWS利用料金を**サービス別・日別・月別・タグ別などに可視化**できる分析ツール。過去12ヶ月〜最長14ヶ月分の利用状況をグラフと表で確認できる。
- **どんな場面で使う？**: 「先月のコストがなぜ急増したのか調べたい」「どのサービスが一番コストを使っているか知りたい」「特定プロジェクト（タグ）のコストを集計したい」など、**実績の振り返り**が主目的。
- **RI（リザーブドインスタンス）や Savings Plans の推奨事項**も確認できる。
- 事前に**有効化が必要**（初回のみ）。有効化後、データが反映されるまで最大24時間かかる。

#### Budgets（Billing and Cost Managementの機能）

- Billing and Cost Management 内の **予算管理機能**。AWSコストや使用量に**予算を設定**し、閾値を超えたときに**アラート通知**を受け取れる。
- **どんな場面で使う？**: 「月額予算を$100に設定して、$80（80%）で警告、$100（100%）到達で即通知したい」のように、**コスト超過の事前検知**が主目的。
- 予算タイプ: **コスト予算**（ドル単位）、**使用量予算**（EC2時間数など）、**RI/Savings Plans利用率予算**など。
- アラートはSNS経由でメール通知やLambda連携ができる。
- **Cost Explorerとの違い**: Cost Explorerは **振り返り**（過去の分析）、Budgetsは **予防**（未来の超過警告）。いずれも同じ Billing and Cost Management サービス内の別機能。

#### Compute Optimizer

- 機械学習で稼働中のリソース（EC2・EBS・Lambda・Auto Scalingグループなど）の使用状況を分析し、**最適なリソースサイズを推奨**してくれるサービス。
- **どんな場面で使う？**: "過剰スペックのEC2を適正サイズにダウンサイズしてコスト削減したい" "Lambdaのメモリ割り当てを最適化したい" など。
- 推奨事項例: 「このt3.largeはCPU使用率が常時10%未満なので、t3.smallにダウンサイズすると月額$50削減できます」。
- **試験で問われるポイント**: 「EC2の過剰プロビジョニングを検出したい」→ **Compute Optimizer**。

#### Trusted Advisor

- AWSベストプラクティスに照らしてアカウント全体を自動チェックし、**コスト・パフォーマンス・セキュリティ・耐障害性・サービス制限**の5カテゴリで推奨事項を提示するサービス。
- **コスト最適化**の観点での推奨例:
  - 使用率の低いEC2インスタンス
  - 関連付けされていないEIP（未使用なのに課金されている）
  - 使用率の低いEBSボリューム
  - アイドル状態のロードバランサー
- **サポートプラン**によって使える機能が異なる。ベーシック/デベロッパープランでは一部のチェック（セキュリティ・サービス制限の主要項目）のみ、ビジネス/エンタープライズプランでは**全チェック項目**が利用可能。
- **Compute Optimizerとの違い**: Compute Optimizerは**機械学習による詳細なリソースサイジング推奨**に特化。Trusted Advisorは**ベストプラクティスに基づく幅広いチェック**（コスト以外にセキュリティ等も含む）。

#### 第9回（4/18）ハンズオン①：CloudFormationでインフラ自動構築

##### 想定ユースケース
新規プロジェクトで**開発・ステージング・本番の3環境を同じ構成で用意する**シナリオです。手作業でマネジメントコンソールから3回構築すると、設定漏れや微妙な差異が混入し、"本番だけ動かない" "本番だけセキュリティ設定が違う" といった事故が起きやすい。CloudFormationを使えば、1つのYAMLテンプレートから何度でも**同じ構成を再現**できます。災害復旧時に別リージョンへ同じ環境を短時間で立ち上げる場面や、検証環境を作って終わったら一括削除する場面でも広く使われる典型パターンです。本ハンズオンでは、VPC + パブリックサブネット + EC2の最小構成をテンプレートから一発で構築し、スタック更新で安全に変更を加え、最後にスタック削除で全リソースを一括クリーンアップする流れを体験します。

##### このハンズオンで学ぶこと
- CloudFormation YAMLテンプレートの構造（Parameters/Resources/Outputs）の理解
- テンプレートによるVPC + EC2環境の自動構築と依存関係の管理
- スタック更新によるインフラ変更と変更セットのプレビュー
- スタック削除による全リソースの一括クリーンアップ

##### リソース状況
- **新規作成**: handson-cfn-stack（CloudFormationスタック）、CFn-VPC、CFn-Public-Subnet、InternetGateway、RouteTable、SecurityGroup、CFn-WebServer

##### ステップ1: テンプレートの内容を確認

以下のYAMLテンプレートを `cfn-template.yaml` として保存:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Hands-on VPC and EC2 Stack

Parameters:
  InstanceType:
    Type: String
    Default: t2.micro
    AllowedValues:
      - t2.micro
      - t2.small
    Description: EC2 instance type

Resources:
  HandsonVPC:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: 10.2.0.0/16
      EnableDnsSupport: true
      EnableDnsHostnames: true
      Tags:
        - Key: Name
          Value: CFn-VPC

  PublicSubnet:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref HandsonVPC
      CidrBlock: 10.2.1.0/24
      AvailabilityZone: !Select [0, !GetAZs '']
      MapPublicIpOnLaunch: true
      Tags:
        - Key: Name
          Value: CFn-Public-Subnet

  InternetGateway:
    Type: AWS::EC2::InternetGateway

  AttachGateway:
    Type: AWS::EC2::VPCGatewayAttachment
    Properties:
      VpcId: !Ref HandsonVPC
      InternetGatewayId: !Ref InternetGateway

  RouteTable:
    Type: AWS::EC2::RouteTable
    Properties:
      VpcId: !Ref HandsonVPC

  PublicRoute:
    Type: AWS::EC2::Route
    DependsOn: AttachGateway
    Properties:
      RouteTableId: !Ref RouteTable
      DestinationCidrBlock: 0.0.0.0/0
      GatewayId: !Ref InternetGateway

  SubnetRouteTableAssociation:
    Type: AWS::EC2::SubnetRouteTableAssociation
    Properties:
      SubnetId: !Ref PublicSubnet
      RouteTableId: !Ref RouteTable

  SecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: Allow HTTP
      VpcId: !Ref HandsonVPC
      SecurityGroupIngress:
        - IpProtocol: tcp
          FromPort: 80
          ToPort: 80
          CidrIp: 0.0.0.0/0

  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      InstanceType: !Ref InstanceType
      ImageId: ami-023e4e3bd8109d6a3
      SubnetId: !Ref PublicSubnet
      SecurityGroupIds:
        - !Ref SecurityGroup
      UserData:
        Fn::Base64: |
          #!/bin/bash
          dnf install -y nginx
          systemctl start nginx
          echo "<h1>Hello from CloudFormation!</h1>" > /usr/share/nginx/html/index.html
      Tags:
        - Key: Name
          Value: CFn-WebServer

Outputs:
  PublicIP:
    Description: Public IP of the web server
    Value: !GetAtt WebServer.PublicIp
  VpcId:
    Description: VPC ID
    Value: !Ref HandsonVPC
```

※ `ImageId` はリージョンによって異なります。講師が事前に最新のAmazon Linux 2023のAMI IDを確認してください。

#### 第9回（4/18）ハンズオン②：CloudWatchアラーム + Systems Manager

##### 想定ユースケース
ECサイトを運営する企業で、**本番EC2のCPU使用率を24時間自動監視し、異常があれば運用担当者のスマホへ即時通知する**仕組みを構築するシナリオです。さらに、セキュリティ強化のため**SSHポート(22)を開けず、踏み台サーバーも置かずに**EC2の保守ができる仕組みも必要です。本ハンズオンでは、CloudWatchアラーム → SNSトピック → スマホメール通知、という運用監視の一連の流れを実際に発火させて体験します。あわせてSystems Manager セッションマネージャーでブラウザから直接EC2にログインし、Run Commandで複数台に一括コマンド実行する運用作業も体験します。これらは実際の運用現場で日常的に使われる、AWSの定番運用パターンです。

##### このハンズオンで学ぶこと
- CloudWatchアラームによるCPU使用率の監視としきい値通知の設定
- アラーム発火 → SNSトピック → スマホでアラート通知を受信する一連の運用監視フローの体験
- Systems Manager セッションマネージャーによるSSH不要のEC2接続
- Run Commandによるリモートコマンドの一括実行
- 運用監視と管理の自動化の基本パターンの理解

##### リソース状況
- **新規作成**: EC2-SSM-Role（IAMロール）、SSM-Test（EC2インスタンス）、High-CPU-Alarm（CloudWatchアラーム）、cpu-alert（SNSトピック）

##### ステップ1: EC2インスタンスを起動（SSM用IAMロール付き）

   - ロール名: `EC2-SSM-Role`
2. EC2コンソール →「インスタンスを起動」

3. 画面右側の「インスタンスを起動」ボタンをクリック

##### ステップ2: CloudWatchアラームを作成

1. コンソール上部の検索バーに「CloudWatch」と入力 →「CloudWatch」を選択
2. 左メニュー「アラーム」をクリック（展開される場合はさらにアラーム一覧ページに遷移）→ 画面右上の **「アラームの作成」** ボタンをクリック

   - これまでの設定内容（メトリクス／条件／通知／アラーム名）が一覧表示される → ざっと確認
   - 画面右下の **「アラームの作成」** ボタンをクリック
8. メールの受信トレイで確認メールが届く →「Confirm subscription」をクリック
   - **スマホでも確認**: スマホのメールアプリで同じ確認メールが受信できていることをチェック

##### ステップ3: Systems Manager セッションマネージャーで接続

1. コンソール上部の検索バーに「Systems Manager」と入力 →「Systems Manager」を選択
2. 左メニュー「**セッションマネージャー**」（カタカナ表記）をクリック
3. ページ右上の **「セッションを開始する」** ボタンをクリック
4. **「Specify target」** 画面:
   - 「理由」（オプション）: 空欄のままでOK
   - **ターゲットインスタンス** 一覧で `SSM-Test` の行にチェックを入れる
   - 画面右下の **「セッションを開始」** ボタンをクリック
5. **確認**: ブラウザ上でターミナルが開き、Linuxシェルのプロンプトが出る。以下のコマンドを打って動作を体感する
   ```bash
   whoami       # → ssm-user （SSMが自動的に作成する専用ユーザー）
   hostname     # → インスタンスのホスト名（例: ip-10-0-xx-xx.ap-northeast-1.compute.internal）
   cat /etc/os-release  # → Amazon Linux 2023 であることを確認
   ```
6. **このステップの学習ポイント**:
   - **SSHキーペアを配っていない**のに、インスタンスの中でコマンドが実行できている
   - **セキュリティグループの22番ポート（SSH）は開けていない**（SG設定を別タブで確認すると良い）
   - それでも接続できるのは、EC2内の **SSM Agent** が **アウトバウンド通信でAWS側にポーリング** しており、マネジメントコンソールとの通信路をSSM側が確立してくれるため
   - メリット: キーペアの管理不要、踏み台サーバー不要、22番ポートを開けなくて済む（攻撃面の削減）、接続履歴がCloudTrailに残る（監査可能）

##### ステップ4: Run Commandでコマンドを一括実行

1. 左メニュー「**Run Command**」をクリック
2. ページ右上の **「Run Command」** ボタン（オレンジ色）をクリック → **「コマンドの実行」** 画面が開く
3. **「コマンドドキュメント」** セクション:
   - 一覧にはデフォルトで `AWS-ApplyAnsiblePlaybooks` などAnsible/Chef系のドキュメントが並んでいる
   - 検索ボックスに `AWS-RunShellScript` と入力してフィルタ
   - 表示された **`AWS-RunShellScript`** の行のラジオボタンを選択
   - → 下の「コマンドのパラメータ」セクションが **Commands** という入力欄に切り替わる
4. **「コマンドのパラメータ」** セクションの **Commands** 欄に以下を入力:
   ```bash
   hostname
   uptime
   df -h
   ```
   - **各コマンドの意味**:
     - `hostname` … サーバーのホスト名（コンピューター名）を表示するLinuxコマンド
     - `uptime` … サーバーの起動からの経過時間・現在のログインユーザー数・直近1/5/15分のロードアベレージ（CPU待ち行列の平均長）を表示
     - `df -h` … ディスク（ファイルシステム）の使用量をサイズ・使用率付きで表示。`-h` はhuman-readable（人間に読みやすい単位、例: 1.2G, 500M）の意味
   - いずれも運用現場で **サーバーの基本状態を素早く確認する** ときに使う定番コマンド。Run Commandでこれらを一斉実行できる＝何十台ものサーバーに1クリックでヘルスチェックを流せる、という体験がこのステップのポイント
5. **「ターゲット」** セクション:
   - **「インスタンスを手動で選択する」** のラジオボタンを選択
   - 下に表示されるインスタンス一覧で `SSM-Test` にチェック
6. その他の項目（レート制御・出力オプション・CloudWatchアラーム・SNS通知）は変更せずそのまま
7. 画面右下（一番下までスクロール）の **「実行」** ボタンをクリック
8. コマンド実行の詳細画面に遷移 → 「全体的なステータス」が `進行中` → `成功` になるのを待つ（数秒〜十数秒）
9. **「ターゲットと出力」** セクションに移動:
   - 表内の `SSM-Test` インスタンスの行にチェックを入れる → 右上の **「出力の表示」** ボタンをクリック
   - （または、インスタンスIDの行のリンクをそのままクリックしても同じ画面に遷移できる）
10. 出力詳細画面で **「出力」** セクションを開く（折りたたまれている場合があるので矢印アイコンをクリックして展開）
11. **確認**: Standard output content（標準出力）に以下のような内容が表示される
    - `hostname` の結果（例: `ip-172-31-xx-xx.ap-northeast-1.compute.internal`）
    - `uptime` の結果（稼働時間とロードアベレージ）
    - `df -h` の結果（ディスク使用量の表）

##### ステップ5: アラームを発火させてスマホで通知を受信

ここまでで監視の仕組みは整いました。実際にCPU負荷をかけてアラームを発火させ、SNS経由でスマホに通知が届くことを確認します。

1. 再び左メニュー「**Run Command**」をクリック → ページ右上の **「Run Command」** ボタン（オレンジ色）をクリック → 「コマンドの実行」画面が開く
2. **「コマンドドキュメント」** セクションで検索ボックスに `AWS-RunShellScript` と入力 → 表示された **`AWS-RunShellScript`** のラジオボタンを選択
3. **「コマンドのパラメータ」** セクションの **Commands** 欄に以下を入力（`yes` コマンドでCPUを意図的に占有。t2.microは1vCPUなので2並列で十分100%近くまで上がる）:
   ```bash
   nohup yes > /dev/null 2>&1 &
   nohup yes > /dev/null 2>&1 &
   ```
   - **コマンドの意味** (1行ずつ分解):
     - `yes` … ひたすら `y` と文字を出力し続けるLinuxコマンド（本来はシェルスクリプトの自動応答用）。無限ループでCPUを使い続けるため、**負荷テストの定番**
     - `> /dev/null` … 標準出力を **捨てる**（`/dev/null` はLinuxの"ゴミ箱"。ここに書き込むと消える）。画面が `y` で埋め尽くされるのを防ぐ
     - `2>&1` … 標準エラー出力も標準出力と同じ先（=ここでは`/dev/null`）に流す。念のためエラーも捨てる
     - `&` … コマンドを **バックグラウンド実行**。これがないとRun Commandがコマンド完了を待ち続けてタイムアウトする
     - `nohup` … Run Commandのセッションが切れても **プロセスを生かし続ける**（no hang up）ための保険
     - これを **2行=2プロセス** 立ち上げるので、1vCPUのt2.microではCPU使用率が100%に張り付く
4. **「ターゲット」** セクション:「インスタンスを手動で選択する」→ `SSM-Test` にチェック
5. 画面右下（一番下までスクロール）の **「実行」** ボタンをクリック
6. **CPU使用率が上がっていく様子を観察**:
   - EC2コンソール → `SSM-Test` を選択 →「モニタリング」タブを開く
   - `CPU utilization (%)` のグラフが右肩上がりになる
   - ⚠️ t2.microは基本モニタリング（5分間隔）なので反映に時間がかかる。数分待ってからグラフを更新すること
7. CloudWatchコンソール →「アラーム」→ `High-CPU-Alarm` を開く
8. **確認**: アラームの状態が `OK` → `ALARM` に変わる
   - ⏱️ **発火まで約8〜10分かかる** のでゆっくり待つ
   - **なぜ即座に鳴らないか**:
     - t2.microは **基本モニタリング=5分間隔** でCloudWatchにCPU使用率を送る
     - アラームの設定は **5分間のAverage（平均）≥ 70%** 。負荷を投入した直後の5分は「前半=0%、後半=100%」と平均されて例えば24%程度にしかならず、閾値70%に届かない
     - **次の5分間のデータポイント**（負荷投入から5〜10分後）でようやく平均が90〜100%に達し、アラームが `ALARM` に遷移する
     - さらに、データポイントの時刻（例: 12:52）から **アラーム状態の更新（例: 13:02）までCloudWatch側の集計・評価ラグで数分遅れる** ことも多い
   - 💡 **ブラウザで待つときの注意**: アラーム画面は **自動更新されない**。閾値を超えても画面を **リロード（Cmd/Ctrl + R）** するまでグラフも状態表示も古いまま。定期的にリロードして状態を確認すること
   - 💡 実運用では **Statistic=Maximum** にしたり **詳細モニタリング（1分間隔）** を有効化することで、もっと早くアラームを鳴らす設計もできる（ただし追加料金発生）
9. **スマホで確認**: ステップ2で登録したメールアドレスにアラート通知メールが届く
   - 件名例: `ALARM: "High-CPU-Alarm" in Asia Pacific (Tokyo)`
   - 本文にメトリクス名・閾値・発火時刻が記載されている
   - スマホのメールアプリで通知バナーが出ることを確認
   - ⏱️ メールは **アラームが `ALARM` 状態に遷移した後に送信開始** → さらに1〜2分遅れて着信することがある
10. CPU負荷を停止する（セッションマネージャーかRun Commandから）:
    - **事前にプロセスが居ることを確認**:
      ```bash
      pgrep -a yes
      ```
      → `2815 yes` `2816 yes` のようにPIDとプロセス名が2行表示されれば動いている
    - **停止コマンドを実行**:
      - セッションマネージャー経由 → `sudo pkill yes`（`sudo` 必須）
      - Run Command経由 → `pkill yes`（Run Commandはroot権限で動くので `sudo` 不要）
    - **停止後にプロセスが消えたことを確認**:
      ```bash
      pgrep -a yes
      ```
      → 何も出力されなければ成功（終了コード1で帰ってくる）
    - ⚠️ セッションマネージャーで `sudo` なしで `pkill yes` を打つと `Operation not permitted` エラーになる。`yes` プロセスはRun Commandがroot権限で起動したため、非rootユーザー（`ssm-user`）からは停止できない
11. 数分後にアラームの状態が `ALARM` → `OK` に戻ることを確認

> **運用監視の基本パターン体験**: **Metrics（計測） → Alarm（閾値監視） → SNS（通知配信） → 受信** の一連の流れが実際に機能することを確認できました。実務でもこの基本形を起点に、必要に応じてEventBridgeやLogs Insightsなどを組み合わせて監視を強化していきます。

##### リソース削除

1. CloudWatchアラーム `High-CPU-Alarm` を削除
2. SNSトピック `cpu-alert` を削除
3. EC2インスタンス `SSM-Test` を終了
4. IAMロール `EC2-SSM-Role` を削除

## 練習問題

### 問題1
ある企業では、EC2インスタンス上で動作するWebアプリケーションを運用しています。運用チームから「EC2インスタンスのメモリ使用率が高騰してアプリケーションが不安定になることがある」との報告を受けました。メモリ使用率を継続的に監視し、閾値を超えた場合に運用チームへ通知する仕組みを構築する必要があります。

この要件を満たすために、最も適切な対応はどれでしょうか。

#### 選択肢
A. EC2インスタンスの詳細モニタリングを有効化して、CloudWatchの標準メトリクスでメモリ使用率を監視し、CloudWatchアラームを設定する。

B. EC2インスタンスにCloudWatchエージェントをインストールして、メモリ使用率をカスタムメトリクスとしてCloudWatchに送信し、CloudWatchアラームを設定する。

C. AWS Configルールを作成して、EC2インスタンスのメモリ使用率を評価し、非準拠の場合にSNSトピックへ通知する。

D. CloudTrailでEC2インスタンスのメモリ使用率イベントを記録し、CloudWatch Logsに送信してアラームを設定する。

### 問題2
あるソリューションアーキテクトは、本番環境のEC2インスタンスが予期せず終了されたことを調査しています。「誰が・いつ・どのAPIを呼び出してインスタンスを終了したのか」を特定する必要があります。

この調査に最も適したAWSサービスはどれでしょうか。

#### 選択肢
A. Amazon CloudWatch Logsでインスタンスのシステムログを確認する。

B. AWS CloudTrailのイベント履歴でTerminateInstances APIの呼び出し記録を確認する。

C. AWS Configのリソースタイムラインで構成変更履歴を確認する。

D. Amazon EventBridgeのイベントバスでEC2状態変化イベントを確認する。

### 問題3
あなたはソリューションアーキテクトとして、AWS上でアプリケーションの開発およびテストを行っています。その際は、作業効率を高めるために、AWS環境のインフラ構成を一括にデプロイするような方法が求められています。それによって、開発環境やテスト環境を迅速にプロビジョニングしたり、容易に削除したりできるようにすることを目指しています。

この要件を満たす最も適切なAWSサービスの設定はどれでしょうか。

### 問題9
あなたはソリューションアーキテクトとして、複数のEC2インスタンスを活用して社内システムを構築しています。このシステムの運用においては、多数のEC2インスタンスに対する中央集中ロギングによる管理が必要とされています。その際、EC2インスタンスに対するネットワークトラフィックに関するログや、内部OSに関するステータスログが必要となります。

#### 選択肢

B. EC2インスタンスのネットワークインターフェースに対してAWS Config を有効化して、リソース構成に関するログを収集する。複数インスタンスのログを取りまとめるために、単一のCloudWatch Logsにログを送信する。

C. EC2インスタンスのネットワークインターフェースに対してVPCフローログを有効化する。複数のVPCフローログを取りまとめるために、単一のCloudWatch Logsにログを送信する。

D. EC2インスタンスにCloudWatchエージェントをインストールして、インスタンスの詳細ログを取得する。これらのログを取りまとめるために、単一のCloudWatch Logsにログを送信する。

E. EC2インスタンスの詳細モニタリングを有効化して、インスタンスの詳細ログを取得する。これらのログを取りまとめるために、単一のCloudWatch Logsにログを送信する。

### 問題10
ある企業では、AWS Configルールを使用してセキュリティグループの設定を継続的に評価しています。「SSHポート（22番）が0.0.0.0/0で全開放されているセキュリティグループ」が検出された場合に、自動的にそのルールを削除して修復する仕組みを構築したいと考えています。

C. AWS CloudTrailでセキュリティグループの変更イベントを検知し、SNS経由で運用チームに通知して手動で修正する。

D. AWS Trusted Advisorのセキュリティチェックで検出された結果に基づいて、手動でセキュリティグループを修正する。
