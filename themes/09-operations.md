# 第9章: 運用支援サービス

## 座学

## 全体の導入（なぜ運用支援サービスが必要なのか）

今回のテーマは **運用支援サービスとコスト管理** です。ここまで第1回から第8回にかけて、VPC、EC2、ストレージ、データベース、ネットワーク、アプリケーション、分析……とAWSで **システムを作る** ための主要サービスを一通り見てきました。

でもシステムは作って終わりじゃないですよね。**作った後に"運用"するフェーズ**が本番です。運用フェーズで直面する課題を考えてみましょう。

- 本番のEC2で急にCPU使用率が跳ね上がった。**誰が気づきますか？** 24時間張り付いて見ているわけにはいかない
- 社内ルールで "全S3バケットは暗号化必須" と決めた。**誰が毎日全バケットをチェックしますか？**
- 月末にAWS請求書を見たら予想の3倍だった。**もっと早く気づきたい**ですよね

こうした "運用の課題" に対して、AWSは専用のサービス群を用意しています。今日はこれらを **「監視・自動化・統制・コスト管理」** の4つの切り口で整理していきます。

1. **監視する** → CloudWatch（メトリクス・ログ・アラーム・イベント）
2. **監査する・統制する** → CloudTrail（誰が何をしたか）、AWS Config（設定が正しいか）
3. **インフラ・運用を自動化する** → CloudFormation（IaC）、Systems Manager（運用管理）
4. **コストを管理する** → Cost Explorer、Budgets、Compute Optimizer、Trusted Advisor

作るフェーズから運用フェーズへの頭の切り替えだと思ってください。

あともう一つ、この回の内容について少しお話しさせてください。エンジニアと一口に言っても、得意分野や歩んできたキャリアは本当に人それぞれですよね。バリバリとコードを書いてプロダクトを作っていくのが得意な人もいれば、そこが自分の強みだとは感じていない人もいると思います。

今日学ぶ **運用支援サービス** の領域は、そういう "開発でゴリゴリ戦うのとはちょっと違う形で現場に価値を出したい" という人にとって、キャリアの新しい選択肢になり得る分野だと私は思っています。システムを安定して動かし続ける、監視の仕組みを整える、コストを最適化する、ガバナンスを効かせる――これらはどれも **確かなスキルが身につけば現場から強く頼られる仕事** です。AWSのサービスの使い方と運用のベストプラクティスをきちんと押さえていくことで、"運用を任せられるエンジニア" という専門性をつくっていけます。

なので今日の内容は、単に試験のために覚えるというよりも、**自分のキャリアの幅を広げる1日** だと思って聞いてもらえると嬉しいです。ここで学ぶことは、どの現場に行っても必ず求められる力になりますので、一緒にじっくり見ていきましょう。

## CloudWatch

### 導入（監視の必要性）

まず最初は **CloudWatch**。**AWSのモニタリング・可観測性サービス**です。

さっき"EC2のCPUが跳ね上がったら誰が気づく？"という話をしましたよね。答えは **"CloudWatchが気づいてくれる"** です。

CloudWatchは例えるなら **工場の中央制御室**。工場には何百台もの機械が並んでいて、温度計や圧力計がそれぞれ数値を出している。中央制御室のオペレーターはその計器を見て、異常があれば警報を鳴らす。運転ログも記録する。定期点検もスケジュール実行する。CloudWatchはこれをAWSリソースに対してやってくれるサービスです。

CloudWatchには4つの主要機能があります。順番に見ていきましょう。

- **CloudWatch Metrics** … 数値を集める（計器）
- **CloudWatch Alarm** … 閾値を超えたら通知する（警報）
- **CloudWatch Logs** … ログを集める（運転日誌）
- **Amazon EventBridge** … イベントやスケジュールで自動実行（定期点検・自動化ルール）

### CloudWatch Metrics

まず **CloudWatch Metrics**。これは **"数値データを集めて時系列で記録する"** 機能です。

AWSリソースからは常にいろんな数値が出ています。EC2のCPU使用率、RDSのDB接続数、ELBのリクエスト数、S3のオブジェクト数……これらを全部CloudWatch Metricsが受け取って時系列データとして保存する。あとからグラフで見たり、次に説明するアラームと連携したりできます。

メトリクスには2種類あります。**標準メトリクス** と **カスタムメトリクス** です。

**標準メトリクス** は、AWSが勝手に送ってくれるやつ。EC2のCPU使用率とか、RDSの接続数とか、何もしなくても勝手にCloudWatchに飛んできます。

でもここで **試験頻出の落とし穴** があります。**EC2のメモリ使用率とディスク使用率は標準メトリクスに含まれません**。なぜかわかりますか？ これはOSの内部の話だからです。AWSから見えるのはハイパーバイザーの層までで、OSの中でメモリが何%使われているかはAWS側からは見えないんです。

じゃあメモリ使用率を監視したいときはどうするか？ **CloudWatch Agent** というエージェントソフトをEC2にインストールします。するとエージェントがOSの中からメモリ使用率を取って、**カスタムメトリクス**としてCloudWatchに送ってくれる。これが2種類目の **カスタムメトリクス** です。

カスタムメトリクスはアプリ独自の指標にも使えます。たとえば "注文数" "エラー数" "処理時間" みたいなアプリケーション固有の数値を、自分でCloudWatchに送り込める。

試験で **"EC2のメモリ使用率を監視したい"** と来たら、**"CloudWatch Agent"** と **"カスタムメトリクス"** の組み合わせを答えてください。

### CloudWatch Alarm

次は **CloudWatch Alarm**。メトリクスを集めるだけじゃ意味がないですよね。**異常があったら通知してほしい**。それを担当するのがAlarmです。

たとえば "EC2のCPU使用率が70%を5分以上超えたらメール通知" みたいな閾値を設定します。Metricsが値を監視していて、閾値を超えたらAlarmが発動する。

Alarmが発動したら **SNSトピック経由で通知**（メール・SMS・スマホプッシュ）するのが定番ですが、他にも Auto Scaling、EC2の停止・再起動、Lambda実行などをアクションにできます。

ハンズオン②でまさにこれを体験します。CPU閾値でアラームを作って、SNSトピック経由でスマホにメール通知を飛ばします。

セットで覚えてほしいのは **"Metricsで測る → Alarmで閾値監視 → SNSで通知"** の3点セット。これがCloudWatch監視の基本パターンです。

### CloudWatch Logs

3つ目は **CloudWatch Logs**。これはその名の通り **ログの一元管理**サービスです。

運用していると困るのが、ログがあちこちに散らばることですよね。EC2のアプリログはEC2の中、Lambdaのログは関数ごと、VPCフローログはまた別……このままだと調査するときに一々SSHで入って `tail -f` しないといけない。それをまとめて1箇所に集めるのがCloudWatch Logsです。

CloudWatch Logsで覚えてほしいのが **メトリクスフィルター** という機能。ログに対して "ERROR という文字列が出たらカウントする" みたいなパターンマッチを設定すると、ヒット数を **メトリクスに変換** してくれるんです。

ここでさっきのAlarmに繋がります。メトリクスフィルターで作ったメトリクスをCloudWatch Alarmで監視すれば、**"ログに ERROR が5回出たらメール通知"** が実現できる。運用自動化の定番パターンで、試験にもよく出ます。

### Amazon EventBridge

CloudWatchの4つ目、**Amazon EventBridge**。これは少し毛色が違います。

これまでのMetrics/Alarm/Logsは "監視する" 機能でしたが、EventBridgeは **"何かが起きたら何かをする" という自動化ルール**を作る機能です。（もともと "CloudWatch Events" という名前だったものがリブランド・機能拡張されて現在のEventBridgeになっています。古い教材や記事では旧名で書かれていることもある、程度に覚えておいてください。）

構成要素はシンプルで、**イベントソース**（きっかけ：EC2の状態変化、S3へのアップロード、API呼び出しなど）、**ターゲット**（実行先：Lambda、SNS、Step Functionsなど）、そのきっかけとなる1件1件が **イベント**（JSON形式）、そして時刻ベースのトリガーが **スケジュール** です。

一番よく使うのが **スケジュール**。cron式で "毎日午前3時にLambdaを実行" みたいな定期実行が作れる。Linuxのcronと同じ感覚です。

EventBridgeはAWS内のイベントだけでなく、**SaaSアプリ**（Zendesk、Datadogなど）からのイベントも受け取れるのが強みで、マルチクラウド・SaaS連携のハブとしても使えます。

試験では **"cronで定期実行"** も **"SaaSからのイベント連携"** も、どちらもEventBridgeを選びます。

ここで1つ混同注意。このあとCloudTrailで "イベント" という言葉がまた出てきますが、**CloudTrailのイベントは別物**です。EventBridgeのイベントは "何かが起きた" という通知、CloudTrailのイベントは "誰が何のAPIを呼んだか" の記録。名前は同じでも役割が違います。

## CloudTrail

次は **CloudTrail**。これはCloudWatchと並んで試験頻出のサービスです。

課題はこれ。**セキュリティ事故があったとき、"誰が" "いつ" "何をしたか" を後から追跡できますか？** たとえば本番のS3バケットが急に公開設定になってしまった。誰がやった？ いつ？ 意図的？ 事故？ これを後から調べられないと原因究明もコンプライアンス対応もできません。

CloudTrailは **AWSアカウント内で実行されたすべてのAPI呼び出しを記録**する監査ログサービスです。AWSの操作は全部APIを通じて行われます。マネジメントコンソールでポチポチ操作してもその裏でAPIが呼ばれている。CLIでもSDKでもAPIが呼ばれる。それをCloudTrailが全部記録してくれる。

CloudTrailはAWSアカウントを作った時点で **自動的に有効化** されていて、**過去90日分**の履歴は無料で確認できます。それ以上の長期保存や、S3へのエクスポート、組織全体での集約ログには **"証跡（Trail）"** を明示的に作成します。

CloudTrailで記録されるイベントには2種類あります。**管理イベント** と **データイベント** です。

**管理イベント**は、リソースの "管理操作" の記録。EC2を起動した、IAMユーザーを作った、S3バケットを作った、セキュリティグループを変えた……こういう **コントロールプレーン** の操作です。これは **デフォルトで記録** されています。

ここで **コントロールプレーン** という言葉を軽く補足しますね。AWSに限らずクラウドやネットワークの世界では、操作を大きく2つの層に分けて考えます。**コントロールプレーン** は "リソースを作る・消す・設定を変える" といった **管理系の操作** 、**データプレーン** は "作ったリソースに対して実際にデータを出し入れする" **利用系の操作** です。たとえばS3で言えば、"バケットを作る / 削除する / ポリシーを変更する" がコントロールプレーン、"オブジェクトをPutする / Getする" がデータプレーン。EC2で言えば、"インスタンスを起動する / 停止する" がコントロールプレーン、"インスタンスに入ってアプリを動かす" がデータプレーン、というイメージです。CloudTrailの管理イベント＝コントロールプレーンの操作、データイベント＝データプレーンの操作、と対応させて覚えるとスッキリします。

一方の **データイベント** は、量が多すぎてデフォルトでは記録されません。必要なら **明示的に有効化** して、追加料金を払います。

試験で **"S3のPutObject/GetObjectを監査したい"** と来たら、**"CloudTrailのデータイベントを有効化"** を選びます。"管理イベントは自動、データイベントは明示的に有効化" この区別は押さえてください。

それからCloudWatch Logsとの違いも大事。CloudWatch Logsは "アプリが動いた結果のログ"、CloudTrailは "人やシステムがAWSに対して行った操作のログ"。視点の向きが違います。

## AWS Config

次は **AWS Config**。CloudTrailと似ているようで役割が違うサービスです。

課題はこれ。**社内ルールで "全S3バケットは暗号化必須" "セキュリティグループで22番ポートを全開放するな" と決めたとして、全リソースが本当にルールを守っているか、誰が毎日チェックしますか？** 手作業では無理ですよね。

AWS Configは **"リソースの設定状態を記録し、望ましい状態に準拠しているかを継続的に評価する"** サービスです。コンプライアンスとガバナンスの中核ですね。

例えるなら **社内ルール遵守チェッカー**。全社員がIDカードを装着しているか、書類が所定の場所に保管されているか、を定期巡回してチェックして、違反を報告する。Configがこれをクラウド上でやってくれます。

ここで **CloudTrailとConfigの違い** を整理しておきましょう。

- **CloudTrail** は "**誰が・何をしたか**" の **操作ログ**（監査）
- **Config** は "**今の設定が正しいか**" の **評価 + 構成履歴**（コンプライアンス）

視点が違う。"操作の記録" と "状態の評価" です。試験では両者の役割分担がよく問われます。

Configを理解するには **Configルール** と **修復アクション** の2つを押さえてください。

**Configルール** が評価条件の定義です。"全S3バケットは暗号化されているか" みたいな条件を書く。Configはそれを定期的にチェックして、COMPLIANT（準拠）かNON_COMPLIANT（非準拠）かを記録する。

Configルールには **マネージドルール** と **カスタムルール** があります。

**マネージドルール** はAWSが事前定義してくれた100種類以上のルール集。"S3バケットは公開禁止" "SSHは全開放禁止" "必須タグ必須" みたいな典型的なルールは全部用意されているので、選んで有効化するだけでOK。

**カスタムルール** は自社独自のルール。マネージドルールでカバーできないときに、**Lambda関数**や**AWS Config Guard**という言語で評価ロジックを自分で書きます。

そして次が面白いところ。**修復アクション**です。

違反を検知するだけだと、結局人間が直さないといけない。"公開されたS3バケットを見つけたので誰か直してください" とメールが来ても、対応が遅れるとその間に情報漏洩するかもしれない。

そこで **"違反を検知したら自動で直す"** という仕組みがあります。これが修復アクションです。"公開設定になったS3バケットを自動で非公開に戻す" "必須タグが欠落したEC2に自動でタグを付与" といった自動修復が組める。

じゃあこの自動修復は何で動いているかというと、次のSSMにつながります。**Systems Manager Automation** を裏側で呼び出して実行しているんです。これは次のSystems Managerセクションで詳しく説明します。

試験で押さえるべきは **"Config単体では検知だけ。自動修復までやるには Config + SSM Automation の組み合わせ"** という点です。

## CloudFormation

次は **CloudFormation**。運用の頭から少し離れて、**インフラ構築の自動化** の話です。

課題はこれ。**開発環境と本番環境を同じ構成で用意したい。あるいは災害復旧用に別リージョンへ同じ環境を短時間で作り直したい。手作業でマネジメントコンソールをポチポチやりますか？**

手作業だと確実にミスが出ます。"本番だけセキュリティグループの設定が1つ違っていて事故った" みたいなことが起きる。人間の記憶とチェックリストに頼った構築は限界があるんです。

CloudFormationは **"AWSリソースをコードで定義して、自動的に構築・更新・削除できる"** サービスです。いわゆる **IaC（Infrastructure as Code）** をAWS公式に実現するツール。

ここで少し **IaC（Infrastructure as Code）** という言葉自体の意味も押さえておきましょう。直訳すると "インフラをコードとして扱う" 。VPC、サブネット、EC2、セキュリティグループ、こういった **インフラの構成をテキストファイル（コード）で記述して、ソフトウェア開発と同じやり方で管理しよう** という考え方です。

昔ながらのやり方だと、インフラ構築はマネジメントコンソールをポチポチクリック、あるいは構築手順をExcelの手順書にまとめて先輩から後輩へ口伝……というのが普通でした。でもこれだと "本番と開発で微妙に設定が違う" "誰も正しい構成を把握していない" "構築した人が辞めたらブラックボックス化" みたいな問題が山ほど起きる。

IaCはこれを根本的に変えます。**コードが唯一の構成情報源** になり、**Gitでバージョン管理・差分確認・コードレビュー** ができる。同じコードを流せば同じ環境が何度でも作れる（再現性）。CIに組み込めば変更が自動反映される。コード自体が常に最新のドキュメントになる、といったメリットがあります。

IaCツールはCloudFormationの他にも、**Terraform**（HashiCorp社製でマルチクラウド対応、業界デファクト）、**AWS CDK**（TypeScriptやPythonなどのプログラミング言語でインフラを書ける）などがあります。試験で "IaC" と来たらまず **CloudFormation** を選んでください。

例えるなら **建築設計図** です。同じ設計図を渡せば、何度でも同じ建物が建てられる。人間の手作業が入らないので構築ミスがない。設計図そのものをGitでバージョン管理すれば、"いつ・誰が・どこを変えたか" も全部追跡できる。

CloudFormationで押さえる用語は2つ。**テンプレート** と **スタック** です。

**テンプレート** は設計図そのもの。**YAML** か **JSON** で書きます（可読性からYAMLが主流）。テンプレートの中で一番重要なのが **Resources** セクション。これが必須で、ここに "VPCを作る" "EC2を作る" "S3バケットを作る" を全部書きます。他に Parameters（外部入力）や Outputs（出力値）のセクションもあります。

**スタック** は、テンプレートから実際に作られたリソースのグループです。設計図が"テンプレート"で、その設計図から建てた"1軒の建物"がスタック、と思ってください。

スタックのいいところは **削除が一発** なこと。1つのスタックを削除すれば、そのテンプレートで作った全リソースが一括で削除される。検証環境を作って、検証が終わったらスタック削除、というクリーンアップが超楽になります。

ハンズオン①でまさにこれを体験します。YAMLテンプレートを書いて、VPC + EC2を一発で構築して、最後にスタック削除で一気に片付けます。

試験では **"複数リージョンに同じ環境を展開したい"** → CloudFormation（StackSetsという発展機能もある）、**"インフラをコードで管理したい"** → CloudFormation、と覚えてください。

## Systems Manager（SSM）

### 導入

次は **Systems Manager**、略してSSMです。これは運用管理の**統合ツールボックス**です。

運用していると、EC2のパッチ当てたり、複数のサーバーに同じコマンドを打ったり、サーバーにログインしてトラブル調査したり、資産管理で "どのサーバーにどのアプリが入ってるか" を棚卸ししたり……いろんな作業が発生します。

これらを個別のツールでやると管理が煩雑になる。SSMはこうした運用作業を1つの統合サービスにまとめてくれます。

SSMには多数の機能がありますが、今日は主要4つ、**Run Command / Patch Manager / Session Manager / Inventory** を押さえます。

1つ前提があります。SSMで管理するには、EC2に **SSM Agent** がインストールされていて、IAMロールに **`AmazonSSMManagedInstanceCore`** ポリシーが付いている必要があります。Amazon Linux 2023なら Agentは標準搭載なので、IAMロールを付けるだけでOK。

### Run Command

1つ目、**Run Command**。**複数のEC2にリモートでコマンドを一括実行する** 機能です。100台のEC2に `yum update` を打ちたいときにSSHで1台ずつ入るのは現実的じゃない。Run Commandなら1つの画面から対象インスタンスを選んでコマンドを一括送信できて、結果も一元的に確認できます。ハンズオン②で実際に使います。

### Patch Manager

2つ目、**Patch Manager**。**OSパッチ適用の自動化** です。パッチベースライン（どのパッチを当てるか）とメンテナンスウィンドウ（いつ当てるか）を定義しておけば、深夜などの業務影響の少ない時間帯に自動でパッチ適用してくれる。コンプライアンス要件（"全サーバー30日以内に最新パッチ適用" など）への対応に使います。

### Session Manager

3つ目、**Session Manager**。これがSSMの中で特に重要です。

課題: EC2にSSH接続するには、セキュリティグループで22番ポートを開ける必要がある。でも22番ポートを開けると、攻撃者もそこを狙ってくる。パスワード辞書攻撃、鍵の流出リスク……セキュリティ観点で好ましくない。

Session Managerは **ブラウザから直接EC2のシェルにアクセスできる** 機能です。**SSH鍵不要、踏み台サーバー不要、22番ポートを開ける必要もない**。

仕組みは、EC2のSSM Agentが AWS側にアウトバウンド接続して待機していて、ブラウザからのアクセスをそのコネクション経由で中継する、というもの。セキュリティグループにインバウンドルールを一切書かなくてもOK。

さらに嬉しいのが **セッションログが自動でCloudWatch Logs や S3 に記録される**こと。"誰がいつサーバーに入って何をしたか" が全部ログに残るので、監査対応にも使える。

ハンズオン②でまさにこれを体験します。SSH鍵を作らずに、ブラウザでEC2にログインしてターミナル操作します。初めての人は感動するポイントですよ。

試験で **"SSHポートを開けずにEC2に安全にアクセスしたい"** と来たら、答えは **Session Manager** です。

### Inventory

4つ目、**Inventory**。**サーバーのソフトウェア構成を自動棚卸し** する機能です。インストール済みアプリ、Windows更新プログラム、ネットワーク設定などが自動収集されて、コンソールで一覧・検索できます。脆弱性対応（"このライブラリを使っているサーバーはどれか"）や資産管理・ライセンス管理に使います。

## コスト管理サービス

### 導入

最後のテーマは **コスト管理** です。

AWSは従量課金なので、"月末に請求書を見たら予想の3倍だった" という事故を防ぐには、**コストを可視化・予算管理・最適化する** 仕組みが必要です。主要4つを見ます。

- **Cost Explorer** … 過去のコスト分析（振り返り）
- **Budgets** … 予算設定とアラート（予防）
- **Compute Optimizer** … リソースサイジング最適化（機械学習）
- **Trusted Advisor** … ベストプラクティス全般チェック

例えるなら **家計管理アプリ**。先月いくら使ったか、今月の予算超えそうか、無駄な契約はないかを見える化してくれる、というイメージです。

1つ **名称の注意** をしておくと、**Cost Explorer** と **Budgets** はそれぞれ独立したサービスではなく、**"AWS Billing and Cost Management" というサービスの中の機能** です。コンソールでも「Billing and Cost Management」画面の左メニューから Cost Explorer / Budgets を開きます。ただし試験問題や実務会話では機能名である "Cost Explorer" "Budgets" で呼ばれるのが普通なので、ここでもその表記で進めます。

### Cost Explorer（Billing and Cost Managementの機能）

1つ目、**Cost Explorer**。これは **過去のAWS利用料金をサービス別・日別・タグ別に可視化**できる分析ツールです。AWS Billing and Cost Management サービスの機能の1つですね。

"先月はEC2が$500、S3が$100、RDSが$300……" みたいに内訳をグラフと表で見られる。過去12ヶ月〜14ヶ月分のデータが見られます。

使い方は **振り返り** 中心。"先月コストが急増したけど原因は何か" "このプロジェクトのタグが付いたリソースのコストを集計したい" といった用途ですね。**RI（リザーブドインスタンス）や Savings Plans の推奨事項** も出してくれるので、長期利用割引の判断にも使えます。

1つ注意として、初回は **有効化が必要** で、データが反映されるまで24時間かかります。ハンズオン③でも最初に有効化してから使います。

### Budgets（Billing and Cost Managementの機能）

2つ目、**Budgets**。こちらも Billing and Cost Management の機能の1つで、**予算を設定して、超過しそうなときにアラート通知** する機能です。

Cost Explorerが "振り返り" なら、Budgetsは "予防"。"月額予算を$100にして、$80で警告、$100到達で即通知" みたいな多段階アラートが組めます。

予算の種類はいくつかあって、**コスト予算**（ドル単位）、**使用量予算**（EC2時間数など）、**RI/Savings Plans利用率予算** などから選べます。

アラートはSNS経由でメール通知したり、Lambdaに連携して自動停止処理を呼ぶこともできる。

ハンズオン③で実際に月額$50の予算を作って、80%と100%の2段階アラートを設定します。

試験では **"コスト超過を事前に検知したい"** → Budgets、**"過去のコスト分析がしたい"** → Cost Explorer、と使い分けます。

### Compute Optimizer

3つ目、**Compute Optimizer**。これは **機械学習でリソースサイジングを最適化提案** してくれるサービスです。

課題: "このEC2、本当に t3.large 必要？" "Lambdaのメモリ割り当て過剰じゃない？" こういう判断は実際のリソース使用状況を見ないとできません。でも何十台もあるEC2を1台ずつ見て判断するのは現実的じゃない。

Compute Optimizerは EC2・EBS・Lambda・Auto Scalingグループなどの使用状況を機械学習で分析して、**"このt3.largeはCPU使用率が常時10%未満なので、t3.smallにダウンサイズすると月額$50削減できます"** みたいな具体的な推奨を出してくれる。

試験で **"EC2の過剰プロビジョニングを検出したい"** と来たら **Compute Optimizer** を選びます。

### Trusted Advisor

4つ目、**Trusted Advisor**。これは **AWSのベストプラクティスに照らしてアカウント全体を自動チェック** してくれるサービスです。

チェック項目は5カテゴリ: **コスト最適化・パフォーマンス・セキュリティ・耐障害性・サービス制限**。今日の文脈だとコスト最適化の部分が重要です。

コスト最適化での典型的な指摘例を挙げると：

- 使用率の低いEC2インスタンス
- **関連付けされていないEIP**（未使用なのに課金されている）
- 使用率の低いEBSボリューム
- アイドル状態のロードバランサー

EIPの話は覚えておいてください。EIPは "アタッチされている間は無料、アタッチされていない状態は課金" という変わった料金体系なので、付け忘れ・外し忘れで地味にコストがかかります。Trusted Advisorがこれを検出してくれる。

なおTrusted Advisorはサポートプランによって使える項目数が変わります（上位プランほど多い）。

**Compute OptimizerとTrusted Advisorの違い**: Compute Optimizerは **機械学習でリソースサイジング推奨に特化**、Trusted Advisorは **ベストプラクティス全般の幅広いチェック**（コスト以外にセキュリティ等も含む）。用途が違います。

## 全体のまとめ

今日は運用支援サービスとコスト管理を一気に見てきました。最後に整理しましょう。

- **監視** → CloudWatch（Metrics / Alarm / Logs / Events）
- **監査** → CloudTrail（操作ログ）+ AWS Config（設定評価）

作るフェーズが終わったら、次はこれらを使って安定運用していく——というのがAWSの基本的な運用の形です。試験でも実務でも、この4つの柱を意識してください。

では実際に手を動かす時間にしましょう。今日のハンズオンは3つあります。CloudFormationでIaC体験、CloudWatch + Systems Managerで監視・運用体験、最後にコスト管理ツールの操作体験です。

#### 第9回（4/18）ハンズオン①：CloudFormationでインフラ自動構築

##### 想定ユースケース
新規プロジェクトで**開発・ステージング・本番の3環境を同じ構成で用意する**シナリオです。手作業でマネジメントコンソールから3回構築すると、設定漏れや微妙な差異が混入し、"本番だけ動かない" "本番だけセキュリティ設定が違う" といった事故が起きやすい。CloudFormationを使えば、1つのYAMLテンプレートから何度でも**同じ構成を再現**できます。災害復旧時に別リージョンへ同じ環境を短時間で立ち上げる場面や、検証環境を作って終わったら一括削除する場面でも広く使われる典型パターンです。本ハンズオンでは、VPC + パブリックサブネット + EC2の最小構成をテンプレートから一発で構築し、スタック更新で安全に変更を加え、最後にスタック削除で全リソースを一括クリーンアップする流れを体験します。

##### このハンズオンで学ぶこと
- CloudFormation YAMLテンプレートの構造（Parameters/Resources/Outputs）の理解
- テンプレートによるVPC + EC2環境の自動構築と依存関係の管理
- スタック更新によるインフラ変更と変更セットのプレビュー
- スタック削除による全リソースの一括クリーンアップ

##### リソース状況
- **新規作成**: handson-cfn-stack（CloudFormationスタック）、CFn-VPC、CFn-Public-Subnet、InternetGateway、RouteTable、SecurityGroup、CFn-WebServer

##### ステップ1: テンプレートの内容を確認

以下のYAMLテンプレートを `cfn-template.yaml` として保存:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Hands-on VPC and EC2 Stack

Parameters:
  InstanceType:
    Type: String
    Default: t2.micro
    AllowedValues:
      - t2.micro
      - t2.small
    Description: EC2 instance type

Resources:
  HandsonVPC:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: 10.2.0.0/16
      EnableDnsSupport: true
      EnableDnsHostnames: true
      Tags:
        - Key: Name
          Value: CFn-VPC

  PublicSubnet:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref HandsonVPC
      CidrBlock: 10.2.1.0/24
      AvailabilityZone: !Select [0, !GetAZs '']
      MapPublicIpOnLaunch: true
      Tags:
        - Key: Name
          Value: CFn-Public-Subnet

  InternetGateway:
    Type: AWS::EC2::InternetGateway

  AttachGateway:
    Type: AWS::EC2::VPCGatewayAttachment
    Properties:
      VpcId: !Ref HandsonVPC
      InternetGatewayId: !Ref InternetGateway

  RouteTable:
    Type: AWS::EC2::RouteTable
    Properties:
      VpcId: !Ref HandsonVPC

  PublicRoute:
    Type: AWS::EC2::Route
    DependsOn: AttachGateway
    Properties:
      RouteTableId: !Ref RouteTable
      DestinationCidrBlock: 0.0.0.0/0
      GatewayId: !Ref InternetGateway

  SubnetRouteTableAssociation:
    Type: AWS::EC2::SubnetRouteTableAssociation
    Properties:
      SubnetId: !Ref PublicSubnet
      RouteTableId: !Ref RouteTable

  SecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: Allow HTTP
      VpcId: !Ref HandsonVPC
      SecurityGroupIngress:
        - IpProtocol: tcp
          FromPort: 80
          ToPort: 80
          CidrIp: 0.0.0.0/0

  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      InstanceType: !Ref InstanceType
      ImageId: ami-023e4e3bd8109d6a3
      SubnetId: !Ref PublicSubnet
      SecurityGroupIds:
        - !Ref SecurityGroup
      UserData:
        Fn::Base64: |
          #!/bin/bash
          dnf install -y nginx
          systemctl start nginx
          echo "<h1>Hello from CloudFormation!</h1>" > /usr/share/nginx/html/index.html
      Tags:
        - Key: Name
          Value: CFn-WebServer

Outputs:
  PublicIP:
    Description: Public IP of the web server
    Value: !GetAtt WebServer.PublicIp
  VpcId:
    Description: VPC ID
    Value: !Ref HandsonVPC
```

※ `ImageId` はリージョンによって異なります。講師が事前に最新のAmazon Linux 2023のAMI IDを確認してください。

5. パラメータ: InstanceType → `t2.micro`（デフォルト）→「次へ」
6. オプション: デフォルトのまま →「次へ」

##### ステップ3: スタック作成を確認

1. スタックのステータスが `CREATE_IN_PROGRESS` → `CREATE_COMPLETE` になるのを観察
2. 「イベント」タブ: 各リソースの作成順序を確認（VPC → サブネット → IGW → ... → EC2）
3. 「リソース」タブ: 作成されたリソース一覧を確認
4. 「出力」タブ: `PublicIP` をコピー
5. **動作確認（推奨）**: AWSコンソール右上の **CloudShell アイコン**（プロンプト風アイコン）を開き、以下を実行
   ```bash
   curl http://<PublicIP>
   ```
   → `<h1>Hello from CloudFormation!</h1>` が返ってくればOK
   - ⚠️ **ブラウザでアクセスする場合の注意**: 必ず `http://` を明示的に付ける。`https://` ではEC2側に証明書がないため接続できない。また、Chrome/Safariは `http://` を自動で `https://` に昇格させる設定（HTTPS-First Mode）が有効なことがあり、その場合は接続できない。確実に動作確認したい場合は上記CloudShell + curlを使う。

##### ステップ4: スタックを更新（パラメータ変更）

1. CloudFormationコンソールのスタック一覧から `handson-cfn-stack` をクリックして開く
2. 画面右上の **「スタックを更新」** プルダウンをクリック →「変更セットを作成」と「直接更新を実行」のうち、**「直接更新を実行」** を選択
   - 💡 補足: 「変更セットを作成」はレビュー用の差分を名前付きで保存してから適用する方法。「直接更新を実行」でも次のウィザード最後に変更セットのプレビューが表示されるので、今回は後者のシンプルな流れを使う
3. **「テンプレートの準備」** 画面:
   - 3つの選択肢（「既存のテンプレートを使用」「既存のテンプレートを置換」「Infrastructure Composer で編集」）のうち、**「既存のテンプレートを使用」** を選択（デフォルト）
   - 画面右下の「次へ」をクリック
4. **「スタックの詳細を指定」** 画面:
   - 「パラメータ」セクションの **InstanceType** プルダウンをクリック → 現在 `t2.micro` になっているので **`t2.small`** に変更
   - 画面右下の「次へ」をクリック
5. **「スタックオプションの設定」** 画面:
   - この画面ではタグ、IAMロール、失敗時のロールバック動作、通知などを設定できるが、今回はすべてデフォルトのまま
     - 「タグ」: 追加なし
     - 「アクセス許可 - IAM ロール」: **「既存のロールを使用」**（デフォルト・空のまま）
     - 「スタックの失敗オプション」:
       - プロビジョニング失敗時の動作 = **「すべてのスタックリソースをロールバックする」**（デフォルト）
       - ロールバック中に新しく作成されたリソースを削除する = **「削除ポリシーを使用する」**（デフォルト）
     - 「その他の設定」（スタックポリシー／ロールバック設定／通知オプション）も変更なし
   - 画面右下の「次へ」をクリック
6. **「レビュー」** 画面:
   - ステップ1〜3の設定内容が一覧表示される（テンプレート／パラメータ／スタックオプション）
   - 一番下までスクロールすると **「変更セットのプレビュー」** セクションに変更内容が表が出る
     - アクション: **Modify**
     - 論理 ID: `WebServer`
     - リソースタイプ: `AWS::EC2::Instance`
     - 置換: **Conditional**（条件付き）
   - 💡 **「Conditional（条件付き）」の意味**: EC2のInstanceType変更は、CloudFormationが内部的にインスタンスを停止 → タイプ変更 → 起動するだけで済むため、**完全な再作成（Replacement）は発生しない**。よってインスタンスIDは変わらない。別のプロパティ変更（例: AMI変更）なら完全再作成になる、という "条件付き" という意味
   - 画面右下の **「送信」** をクリック（「変更セットの表示」は詳細確認用なので今回は不要）
7. スタックのステータスが `UPDATE_IN_PROGRESS` → `UPDATE_COMPLETE` になるのを観察（2〜3分）
8. **確認**: EC2コンソールで `CFn-WebServer` のインスタンスタイプが `t2.small` に変わっていることを確認（インスタンスIDは変わらない）

#### 第9回（4/18）ハンズオン②：CloudWatchアラーム + Systems Manager

##### 想定ユースケース
ECサイトを運営する企業で、**本番EC2のCPU使用率を24時間自動監視し、異常があれば運用担当者のスマホへ即時通知する**仕組みを構築するシナリオです。さらに、セキュリティ強化のため**SSHポート(22)を開けず、踏み台サーバーも置かずに**EC2の保守ができる仕組みも必要です。本ハンズオンでは、CloudWatchアラーム → SNSトピック → スマホメール通知、という運用監視の一連の流れを実際に発火させて体験します。あわせてSystems Manager セッションマネージャーでブラウザから直接EC2にログインし、Run Commandで複数台に一括コマンド実行する運用作業も体験します。これらは実際の運用現場で日常的に使われる、AWSの定番運用パターンです。

##### このハンズオンで学ぶこと
- CloudWatchアラームによるCPU使用率の監視としきい値通知の設定
- アラーム発火 → SNSトピック → スマホでアラート通知を受信する一連の運用監視フローの体験
- Systems Manager セッションマネージャーによるSSH不要のEC2接続
- Run Commandによるリモートコマンドの一括実行
- 運用監視と管理の自動化の基本パターンの理解

##### リソース状況
- **新規作成**: EC2-SSM-Role（IAMロール）、SSM-Test（EC2インスタンス）、High-CPU-Alarm（CloudWatchアラーム）、cpu-alert（SNSトピック）

##### ステップ1: EC2インスタンスを起動（SSM用IAMロール付き）

1. IAMコンソール →「ロール」→「ロールを作成」
   - 信頼されたエンティティ: `AWSのサービス` → `EC2`
   - ポリシー: `AmazonSSMManagedInstanceCore` を検索してアタッチ
   - ロール名: `EC2-SSM-Role`
2. EC2コンソール →「インスタンスを起動」
   - 名前: `SSM-Test`
   - AMI: Amazon Linux 2023
   - インスタンスタイプ: `t2.micro`
   - **キーペア（ログイン）**: プルダウンから **「キーペアなしで続行」** を選択（このインスタンスは後のステップでセッションマネージャー経由で接続するためSSH用キーペアは不要）
   - 高度な詳細 → IAMインスタンスプロファイル: `EC2-SSM-Role`
3. 画面右側の「インスタンスを起動」ボタンをクリック
   - ⚠️ もし「キーペアが選択されていないようです…」という警告モーダルが出た場合は、**「キーペアなしで続行」** にチェックを入れて「インスタンスを起動」を再度クリック

##### ステップ2: CloudWatchアラームを作成

1. コンソール上部の検索バーに「CloudWatch」と入力 →「CloudWatch」を選択
2. 左メニュー「アラーム」をクリック（展開される場合はさらにアラーム一覧ページに遷移）→ 画面右上の **「アラームの作成」** ボタンをクリック
3. メトリクスの選択:
   - 「EC2」→「インスタンス別メトリクス」
   - インスタンス `SSM-Test` の `CPUUtilization` を選択 →「メトリクスの選択」
4. 条件:
   - しきい値の種類: `静的`
   - 条件: `以上` → `70`（%）
   - 期間: `5分`（※ t2.microは基本モニタリング＝5分間隔。期間を1分にするとデータ不足でアラームが発火しにくい）
5. アクションの設定:
   - 「条件」画面で「次へ」をクリックすると **「アクションの設定」** 画面に進む
   - **「通知」** セクション:
     - **アラーム状態トリガー**: `アラーム状態`（デフォルト選択のまま）
     - **次のSNSトピックに通知を送信**: 3択のうち **「新しいトピックの作成」** を選択
     - **トピックを作成中**: `cpu-alert` と入力
     - **通知を受け取るEメールエンドポイント**: **スマホでも受信できるメールアドレス** を入力（例: Gmail、iCloudなど。後のステップで実際にスマホで通知を受け取る体験をします）
     - **「トピックの作成」** ボタンをクリック
   - その他のアクション（Lambda、Auto Scaling、EC2、Systems Manager、調査アクション）は今回は使わないのでスキップ
   - 画面右下の「次へ」をクリック
6. **「アラームの詳細の追加」** 画面:
   - **アラーム名**: `High-CPU-Alarm` と入力
   - **アラームの説明**: 空欄のままでOK（オプション）
   - **タグ**: 何も追加しない（オプション）
   - 画面右下の「次へ」をクリック
7. **「プレビューと作成」** 画面:
   - これまでの設定内容（メトリクス／条件／通知／アラーム名）が一覧表示される → ざっと確認
   - 画面右下の **「アラームの作成」** ボタンをクリック
8. メールの受信トレイで確認メールが届く →「Confirm subscription」をクリック
   - **スマホでも確認**: スマホのメールアプリで同じ確認メールが受信できていることをチェック

##### ステップ3: Systems Manager セッションマネージャーで接続

1. コンソール上部の検索バーに「Systems Manager」と入力 →「Systems Manager」を選択
2. 左メニュー「**セッションマネージャー**」（カタカナ表記）をクリック
3. ページ右上の **「セッションを開始する」** ボタンをクリック
4. **「Specify target」** 画面:
   - 「理由」（オプション）: 空欄のままでOK
   - **ターゲットインスタンス** 一覧で `SSM-Test` の行にチェックを入れる
   - 画面右下の **「セッションを開始」** ボタンをクリック
5. **確認**: ブラウザ上でターミナルが開き、Linuxシェルのプロンプトが出る。以下のコマンドを打って動作を体感する
   ```bash
   whoami       # → ssm-user （SSMが自動的に作成する専用ユーザー）
   hostname     # → インスタンスのホスト名（例: ip-10-0-xx-xx.ap-northeast-1.compute.internal）
   cat /etc/os-release  # → Amazon Linux 2023 であることを確認
   ```
6. **このステップの学習ポイント**:
   - **SSHキーペアを配っていない**のに、インスタンスの中でコマンドが実行できている
   - **セキュリティグループの22番ポート（SSH）は開けていない**（SG設定を別タブで確認すると良い）
   - それでも接続できるのは、EC2内の **SSM Agent** が **アウトバウンド通信でAWS側にポーリング** しており、マネジメントコンソールとの通信路をSSM側が確立してくれるため
   - メリット: キーペアの管理不要、踏み台サーバー不要、22番ポートを開けなくて済む（攻撃面の削減）、接続履歴がCloudTrailに残る（監査可能）

##### ステップ4: Run Commandでコマンドを一括実行

1. 左メニュー「**Run Command**」をクリック
2. ページ右上の **「Run Command」** ボタン（オレンジ色）をクリック → **「コマンドの実行」** 画面が開く
3. **「コマンドドキュメント」** セクション:
   - 一覧にはデフォルトで `AWS-ApplyAnsiblePlaybooks` などAnsible/Chef系のドキュメントが並んでいる
   - 検索ボックスに `AWS-RunShellScript` と入力してフィルタ
   - 表示された **`AWS-RunShellScript`** の行のラジオボタンを選択
   - → 下の「コマンドのパラメータ」セクションが **Commands** という入力欄に切り替わる
4. **「コマンドのパラメータ」** セクションの **Commands** 欄に以下を入力:
   ```bash
   hostname
   uptime
   df -h
   ```
   - **各コマンドの意味**:
     - `hostname` … サーバーのホスト名（コンピューター名）を表示するLinuxコマンド
     - `uptime` … サーバーの起動からの経過時間・現在のログインユーザー数・直近1/5/15分のロードアベレージ（CPU待ち行列の平均長）を表示
     - `df -h` … ディスク（ファイルシステム）の使用量をサイズ・使用率付きで表示。`-h` はhuman-readable（人間に読みやすい単位、例: 1.2G, 500M）の意味
   - いずれも運用現場で **サーバーの基本状態を素早く確認する** ときに使う定番コマンド。Run Commandでこれらを一斉実行できる＝何十台ものサーバーに1クリックでヘルスチェックを流せる、という体験がこのステップのポイント
5. **「ターゲット」** セクション:
   - **「インスタンスを手動で選択する」** のラジオボタンを選択
   - 下に表示されるインスタンス一覧で `SSM-Test` にチェック
6. その他の項目（レート制御・出力オプション・CloudWatchアラーム・SNS通知）は変更せずそのまま
7. 画面右下（一番下までスクロール）の **「実行」** ボタンをクリック
8. コマンド実行の詳細画面に遷移 → 「全体的なステータス」が `進行中` → `成功` になるのを待つ（数秒〜十数秒）
9. **「ターゲットと出力」** セクションに移動:
   - 表内の `SSM-Test` インスタンスの行にチェックを入れる → 右上の **「出力の表示」** ボタンをクリック
   - （または、インスタンスIDの行のリンクをそのままクリックしても同じ画面に遷移できる）
10. 出力詳細画面で **「出力」** セクションを開く（折りたたまれている場合があるので矢印アイコンをクリックして展開）
11. **確認**: Standard output content（標準出力）に以下のような内容が表示される
    - `hostname` の結果（例: `ip-172-31-xx-xx.ap-northeast-1.compute.internal`）
    - `uptime` の結果（稼働時間とロードアベレージ）
    - `df -h` の結果（ディスク使用量の表）

##### ステップ5: アラームを発火させてスマホで通知を受信

ここまでで監視の仕組みは整いました。実際にCPU負荷をかけてアラームを発火させ、SNS経由でスマホに通知が届くことを確認します。

1. 再び左メニュー「**Run Command**」をクリック → ページ右上の **「Run Command」** ボタン（オレンジ色）をクリック → 「コマンドの実行」画面が開く
2. **「コマンドドキュメント」** セクションで検索ボックスに `AWS-RunShellScript` と入力 → 表示された **`AWS-RunShellScript`** のラジオボタンを選択
3. **「コマンドのパラメータ」** セクションの **Commands** 欄に以下を入力（`yes` コマンドでCPUを意図的に占有。t2.microは1vCPUなので2並列で十分100%近くまで上がる）:
   ```bash
   nohup yes > /dev/null 2>&1 &
   nohup yes > /dev/null 2>&1 &
   ```
   - **コマンドの意味** (1行ずつ分解):
     - `yes` … ひたすら `y` と文字を出力し続けるLinuxコマンド（本来はシェルスクリプトの自動応答用）。無限ループでCPUを使い続けるため、**負荷テストの定番**
     - `> /dev/null` … 標準出力を **捨てる**（`/dev/null` はLinuxの"ゴミ箱"。ここに書き込むと消える）。画面が `y` で埋め尽くされるのを防ぐ
     - `2>&1` … 標準エラー出力も標準出力と同じ先（=ここでは`/dev/null`）に流す。念のためエラーも捨てる
     - `&` … コマンドを **バックグラウンド実行**。これがないとRun Commandがコマンド完了を待ち続けてタイムアウトする
     - `nohup` … Run Commandのセッションが切れても **プロセスを生かし続ける**（no hang up）ための保険
     - これを **2行=2プロセス** 立ち上げるので、1vCPUのt2.microではCPU使用率が100%に張り付く
4. **「ターゲット」** セクション:「インスタンスを手動で選択する」→ `SSM-Test` にチェック
5. 画面右下（一番下までスクロール）の **「実行」** ボタンをクリック
6. **CPU使用率が上がっていく様子を観察**:
   - EC2コンソール → `SSM-Test` を選択 →「モニタリング」タブを開く
   - `CPU utilization (%)` のグラフが右肩上がりになる
   - ⚠️ t2.microは基本モニタリング（5分間隔）なので反映に時間がかかる。数分待ってからグラフを更新すること
7. CloudWatchコンソール →「アラーム」→ `High-CPU-Alarm` を開く
8. **確認**: アラームの状態が `OK` → `ALARM` に変わる
   - ⏱️ **発火まで約8〜10分かかる** のでゆっくり待つ
   - **なぜ即座に鳴らないか**:
     - t2.microは **基本モニタリング=5分間隔** でCloudWatchにCPU使用率を送る
     - アラームの設定は **5分間のAverage（平均）≥ 70%** 。負荷を投入した直後の5分は「前半=0%、後半=100%」と平均されて例えば24%程度にしかならず、閾値70%に届かない
     - **次の5分間のデータポイント**（負荷投入から5〜10分後）でようやく平均が90〜100%に達し、アラームが `ALARM` に遷移する
     - さらに、データポイントの時刻（例: 12:52）から **アラーム状態の更新（例: 13:02）までCloudWatch側の集計・評価ラグで数分遅れる** ことも多い
   - 💡 **ブラウザで待つときの注意**: アラーム画面は **自動更新されない**。閾値を超えても画面を **リロード（Cmd/Ctrl + R）** するまでグラフも状態表示も古いまま。定期的にリロードして状態を確認すること
   - 💡 実運用では **Statistic=Maximum** にしたり **詳細モニタリング（1分間隔）** を有効化することで、もっと早くアラームを鳴らす設計もできる（ただし追加料金発生）
9. **スマホで確認**: ステップ2で登録したメールアドレスにアラート通知メールが届く
   - 件名例: `ALARM: "High-CPU-Alarm" in Asia Pacific (Tokyo)`
   - 本文にメトリクス名・閾値・発火時刻が記載されている
   - スマホのメールアプリで通知バナーが出ることを確認
   - ⏱️ メールは **アラームが `ALARM` 状態に遷移した後に送信開始** → さらに1〜2分遅れて着信することがある
10. CPU負荷を停止する（セッションマネージャーかRun Commandから）:
    - **事前にプロセスが居ることを確認**:
      ```bash
      pgrep -a yes
      ```
      → `2815 yes` `2816 yes` のようにPIDとプロセス名が2行表示されれば動いている
    - **停止コマンドを実行**:
      - セッションマネージャー経由 → `sudo pkill yes`（`sudo` 必須）
      - Run Command経由 → `pkill yes`（Run Commandはroot権限で動くので `sudo` 不要）
    - **停止後にプロセスが消えたことを確認**:
      ```bash
      pgrep -a yes
      ```
      → 何も出力されなければ成功（終了コード1で帰ってくる）
    - ⚠️ セッションマネージャーで `sudo` なしで `pkill yes` を打つと `Operation not permitted` エラーになる。`yes` プロセスはRun Commandがroot権限で起動したため、非rootユーザー（`ssm-user`）からは停止できない
11. 数分後にアラームの状態が `ALARM` → `OK` に戻ることを確認

> **運用監視の基本パターン体験**: **Metrics（計測） → Alarm（閾値監視） → SNS（通知配信） → 受信** の一連の流れが実際に機能することを確認できました。実務でもこの基本形を起点に、必要に応じてEventBridgeやLogs Insightsなどを組み合わせて監視を強化していきます。

##### リソース削除

1. CloudWatchアラーム `High-CPU-Alarm` を削除
2. SNSトピック `cpu-alert` を削除
3. EC2インスタンス `SSM-Test` を終了
4. IAMロール `EC2-SSM-Role` を削除

#### 第9回（4/18）ハンズオン③：コスト管理ツールの操作

##### 想定ユースケース
**月末にAWS請求書を見て予算オーバーに気づく事故を防ぐため、コストを継続的に可視化・予算管理・最適化する**シナリオです。AWSは従量課金なので、リソースの停止忘れや過剰なスペック設定で気づかないうちにコストが膨らみます。スタートアップから大企業まで、AWSを使う組織すべてに必須の運用です。本ハンズオンでは、Cost Explorerで過去のコスト内訳をサービス別・日別に分析し、Budgetsで月額予算と多段階アラート（80%警告 / 100%超過通知）を設定し、Compute OptimizerとTrusted Advisorで自動的なコスト最適化推奨を確認する、という4つのコスト管理ツールを一通り体験します。実務では、Budgetsで予算アラートを設定したうえで、月次でCost Explorerを振り返り、Trusted Advisorの推奨事項を定期的に確認する、という運用プロセスにつなげていきます。

##### このハンズオンで学ぶこと
- Cost Explorerによるサービス別・日別のコスト可視化と分析
- Budgetsによる予算設定と多段階アラートの構成
- Compute OptimizerやTrusted Advisorによるコスト最適化の推奨事項の確認

##### リソース状況
- **新規作成**: monthly-cost-limit（Budget）
- **参照のみ**: Cost Explorer、Compute Optimizer、Trusted Advisor

> 💡 **Cost Explorer と Budgets は独立サービスではなく「AWS Billing and Cost Management」内の機能**です。AWSコンソール上部の検索バーから直接アクセスもできますが、通常は「Billing and Cost Management」サービスの画面を開き、左メニューから「**コストと使用状況の分析**」（Cost Explorer）や「**予算**」（Budgets）を選択します。日本語コンソールでは左メニュー表記はカタカナ／日本語化されているので注意。

##### ステップ1: Cost Explorer（Billing and Cost Managementの機能）で利用料金を確認

1. コンソール上部の検索バーに「**Billing and Cost Management**」と入力 →「Billing and Cost Management」を選択
2. 左メニューから「**Cost Explorer**」を選択（検索バーから直接「Cost Explorer」で開いても同じ画面に到達できる）
3. （初回利用時は有効化が必要 → 有効化して24時間後にデータが反映）
4. 左メニューから「**コストと使用状況の分析**」→「**新しいコストと使用状況レポート**」をクリック（初期値で直近6ヶ月のデータがサービス別月次で表示される）
5. 画面右側の **「レポートパラメータ」** パネルで以下の条件を設定:
   - **粒度**: `月別`
   - **日付範囲**: `過去3ヶ月`
   - **グループ化の条件 > ディメンション**: `サービス`
   - **フィルター**: 何も追加しない（空）
   - **データ設定**: 変更なし

   - 例: `Elastic Load Balancing $18.08`, `RDS $5.50`, `VPC $3.91` など、どのサービスが支出上位かが一目で分かる
7. **粒度を「日別」に変更** → 日単位のグラフに切り替わり、「どの日にコストが跳ねたか」を追跡できる

##### ステップ2: Budgets（Billing and Cost Managementの機能）でコストアラートを作成

3. 予算名: `monthly-cost-limit`、月額予算額: `50`（USD）
4. アラート: `80`%、`100`% の2段階で設定 → メールアドレス入力
5. 「予算を作成」

##### ステップ3: Compute Optimizerを確認

1. コンソール上部の検索バーに「Compute Optimizer」と入力 →「AWS Compute Optimizer」を選択
2. **初回利用時は "リソースを分析" 画面が表示される**:
   - 「アカウント設定の構成とオプトインを開始します。オプトインすると、Compute Optimizer が実行中の AWS コンピューティングリソースを分析し、推奨事項の生成を開始します。」と表示される
   - **「ご利用開始」** ボタンをクリック
3. **「アカウントのオプトイン」画面** に遷移:
   - 「AWS Compute Optimizer がリソースと使用状況データを分析」の説明と、「必須サービスリンクロール」の説明が表示される（CloudWatchメトリクスと設定データを最大14日分使って推奨を生成）
   - ページ下部の **「オプトイン」** ボタンをクリック → サービスリンクロールが自動作成され、Compute Optimizerが有効化される
   - ⏱️ **オプトイン後、推奨事項が生成されるまで最大12時間程度かかる**（十分な使用履歴データが必要なため）。今回は即時の推奨は出ないことが多いので、**画面とメニュー構成の把握が学習ポイント**
4. **⚠️ 新機能プロモの注意**: 画面上部または中央に **「Introducing Compute Optimizer Automation」** という新機能告知バナー／ページが表示されることがある
   - これは **未アタッチEBSボリュームの自動スナップショット＆削除** などを行うAutomation機能の紹介で、**今回のハンズオンとは無関係**
   - **「Enable Automation」ボタンは絶対にクリックしないこと** — 意図しないEBS削除や課金リスクがある
   - バナーの場合は **×ボタンまたはキャンセル** で閉じる／ページの場合は左メニューから別項目へ遷移する
5. 左メニュー「**EC2インスタンス**」を選択 → EC2インスタンスの推奨事項ページ
6. **確認**: 推奨事項が生成済みなら「過剰プロビジョニング」「適正」「プロビジョニング不足」の分類で一覧表示される。未生成なら「データを収集中」と表示される
   - 推奨事項の例: 「このt3.largeはCPU使用率が常時10%未満なので、t3.smallにダウンサイズすると月額$50削減できます」

##### ステップ4: Trusted Advisorを確認

1. コンソール上部の検索バーに「Trusted Advisor」と入力 →「Trusted Advisor」を選択
2. 左メニューまたはタブから「**コスト最適化**」カテゴリを開く
3. 画面に表示されるチェック項目一覧を確認:
   - 例: 「使用率の低い Amazon EC2 インスタンス」「関連付けられていない Elastic IP Address」「アイドル状態の Load Balancer」「S3 の不完全なマルチパートアップロードの中止設定」など、40項目以上のコスト最適化チェックが並ぶ
4. ⚠️ **サポートプランによる表示制限に注意**:
   - 今回のトレーニングアカウントのような **Basic/Developerプラン** では、コスト最適化カテゴリのチェック項目は **ほぼすべてグレーアウトされて操作不能** な状態
   - クリックしても詳細画面に進めず、実行結果（件数・対象リソース一覧）も表示されない
   - 画面上部に「**AWS Support プランをアップグレードして Trusted Advisor チェックをすべて取得する**」というバナーが表示されている
   - **Business/Enterpriseプラン** にアップグレードすると全チェック項目がクリック可能・実行結果付きで表示される（本番運用では業務アカウントは通常 Business 以上）
   - 💡 今回は **"どんなチェック項目が用意されているかを眺めて把握する"** だけが実施可能。実際の推奨内容は見られない前提で進める
5. **このステップの学習ポイント**:
   - Trusted Advisorがどのような **観点（40項目超）** でコスト最適化を自動チェックしているかを体感する
   - 「使用率の低いEC2」「未使用EIP」「アイドル状態のLoad Balancer」「不要なNAT Gateway」など、**典型的なコスト垂れ流しパターン** がチェック対象になっていることを把握
   - Compute Optimizerと違い、**機械学習ではなくベストプラクティスに基づくルールベース** の幅広いチェックであることを理解
   - 実務では Business プラン以上のアカウントで **月次のTrusted Advisor確認** を運用プロセスに組み込むのが定番

##### リソース削除

1. Budget `monthly-cost-limit` を削除

## 練習問題

### 問題1
ある企業では、EC2インスタンス上で動作するWebアプリケーションを運用しています。運用チームから「EC2インスタンスのメモリ使用率が高騰してアプリケーションが不安定になることがある」との報告を受けました。メモリ使用率を継続的に監視し、閾値を超えた場合に運用チームへ通知する仕組みを構築する必要があります。

この要件を満たすために、最も適切な対応はどれでしょうか。

#### 選択肢
A. EC2インスタンスの詳細モニタリングを有効化して、CloudWatchの標準メトリクスでメモリ使用率を監視し、CloudWatchアラームを設定する。

B. EC2インスタンスにCloudWatchエージェントをインストールして、メモリ使用率をカスタムメトリクスとしてCloudWatchに送信し、CloudWatchアラームを設定する。

C. AWS Configルールを作成して、EC2インスタンスのメモリ使用率を評価し、非準拠の場合にSNSトピックへ通知する。

D. CloudTrailでEC2インスタンスのメモリ使用率イベントを記録し、CloudWatch Logsに送信してアラームを設定する。

### 問題2
あるソリューションアーキテクトは、本番環境のEC2インスタンスが予期せず終了されたことを調査しています。「誰が・いつ・どのAPIを呼び出してインスタンスを終了したのか」を特定する必要があります。

この調査に最も適したAWSサービスはどれでしょうか。

#### 選択肢
A. Amazon CloudWatch Logsでインスタンスのシステムログを確認する。

B. AWS CloudTrailのイベント履歴でTerminateInstances APIの呼び出し記録を確認する。

C. AWS Configのリソースタイムラインで構成変更履歴を確認する。

D. Amazon EventBridgeのイベントバスでEC2状態変化イベントを確認する。

### 問題3
あなたはソリューションアーキテクトとして、AWS上でアプリケーションの開発およびテストを行っています。その際は、作業効率を高めるために、AWS環境のインフラ構成を一括にデプロイするような方法が求められています。それによって、開発環境やテスト環境を迅速にプロビジョニングしたり、容易に削除したりできるようにすることを目指しています。

この要件を満たす最も適切なAWSサービスの設定はどれでしょうか。

#### 選択肢
A. AWS CodePipelineを設定して、コードを展開するパイプラインを構成することで迅速な環境設定と削除を可能にする。

B. CloudFormationによりテスト環境用のテンプレートを作成する。このテンプレートを利用してテスト環境をデプロイできるようにする。

C. EC2インスタンスのAMIとBashスクリプトを利用して、テスト環境構築を設定する。スクリプトを実行することで、テスト環境をデプロイできるようにする。

D. Amazon ECRにイメージを作成して、テスト環境構築を設定する。このイメージを利用して、テスト環境をデプロイできるようにする。

### 問題4
ある企業では、セキュリティポリシーとして「すべてのS3バケットはサーバーサイド暗号化が有効であること」「すべてのEC2インスタンスに必須タグ（Environment、Owner）が付与されていること」を義務付けています。これらのポリシーへの準拠状況を継続的に評価し、違反があれば検出できるようにする必要があります。

この要件を満たすために、最も適切なAWSサービスはどれでしょうか。

#### 選択肢
A. AWS CloudTrailを有効化して、S3バケットとEC2インスタンスに対するAPI呼び出しを記録し、違反操作を検出する。

B. AWS Configのマネージドルールを使用して、S3バケットの暗号化設定とEC2インスタンスのタグ付けを継続的に評価する。

C. Amazon CloudWatchのカスタムメトリクスを作成して、S3バケットの暗号化状態とEC2のタグ情報を定期的に収集・監視する。

D. AWS Trusted Advisorのセキュリティチェックを利用して、S3バケットの暗号化設定とEC2のタグ付けを確認する。

### 問題5
ある企業のセキュリティチームは、EC2インスタンスへのリモートアクセス方法を見直しています。現在はSSH（ポート22）を使用していますが、以下の要件が示されました。

・SSHポート（22番）をセキュリティグループで開放しない
・踏み台サーバー（Bastion Host）を設置しない
・キーペアの管理を不要にする
・接続セッションのログを監査用に記録する

これらの要件を満たすために、最も適切なソリューションはどれでしょうか。

#### 選択肢
A. EC2 Instance Connectを使用して、一時的なSSH公開鍵をプッシュしてブラウザから接続する。

B. AWS Systems Manager Session Managerを使用して、ブラウザまたはAWS CLIからEC2インスタンスに接続する。

C. AWS Direct Connectを使用して、オンプレミスネットワークからプライベート接続でEC2にアクセスする。

D. Amazon WorkSpacesを使用して、仮想デスクトップ経由でEC2インスタンスにアクセスする。

### 問題6
ある企業は、コンプライアンス要件として「S3バケットに対するすべてのオブジェクトレベルのアクセス（GetObject、PutObject、DeleteObject）を記録し、監査証跡として保存する」ことが求められています。

この要件を満たすために、最も適切な設定はどれでしょうか。

#### 選択肢
A. S3バケットのサーバーアクセスログを有効化して、ログを別のS3バケットに保存する。

B. AWS CloudTrailの証跡を作成し、S3バケットに対するデータイベントを有効化する。

C. Amazon CloudWatch Logsエージェントを設定して、S3のアクセスログを収集する。

D. AWS Configルールを設定して、S3バケットへのアクセスを継続的に評価する。

### 問題7
あなたはAWS上のインフラ展開において、特定のアカウントから複数アカウントにまたがってインフラ展開を自動化させたいと考えています。

この要件を満たすCloudFormationの機能を選択してください。

#### 選択肢
A. CloudFormationスタックセット

B. CloudFormationテンプレート

C. CloudFormation変更セット

D. CloudFormationドリフト

### 問題8
あなたはソリューションアーキテクトとして、AWSクラウドインフラストラクチャのセットアップ状況を分析およびレビューするように依頼されました。また、使用している全てのAWSリソースに対して会社が支払う金額を事前に見積もるように要求されています。

次のうちでコスト要因となっている要素を選択してください。（2つ選択してください。）

#### 選択肢
A. 実行中のEC2インスタンス

B. 停止されたオンデマンドのEC2インスタンス

C. 停止されたEC2インスタンスにアタッチされたEBSボリューム

D. 使用中のVPC

E. 大量のインフラ設定がされているCloudFormation

### 問題9
あなたはソリューションアーキテクトとして、複数のEC2インスタンスを活用して社内システムを構築しています。このシステムの運用においては、多数のEC2インスタンスに対する中央集中ロギングによる管理が必要とされています。その際、EC2インスタンスに対するネットワークトラフィックに関するログや、内部OSに関するステータスログが必要となります。

この要件を満たすソリューションはどれでしょうか。（2つ選択）

#### 選択肢
A. EC2インスタンスのログ取得を実施するために、AWS CloudTrailによるログ取得を有効にする。複数のアクティビティログを取りまとめるために、単一のCloudWatch Logsにログを送信する。

B. EC2インスタンスのネットワークインターフェースに対してAWS Config を有効化して、リソース構成に関するログを収集する。複数インスタンスのログを取りまとめるために、単一のCloudWatch Logsにログを送信する。

C. EC2インスタンスのネットワークインターフェースに対してVPCフローログを有効化する。複数のVPCフローログを取りまとめるために、単一のCloudWatch Logsにログを送信する。

D. EC2インスタンスにCloudWatchエージェントをインストールして、インスタンスの詳細ログを取得する。これらのログを取りまとめるために、単一のCloudWatch Logsにログを送信する。

E. EC2インスタンスの詳細モニタリングを有効化して、インスタンスの詳細ログを取得する。これらのログを取りまとめるために、単一のCloudWatch Logsにログを送信する。

### 問題10
ある企業では、AWS Configルールを使用してセキュリティグループの設定を継続的に評価しています。「SSHポート（22番）が0.0.0.0/0で全開放されているセキュリティグループ」が検出された場合に、自動的にそのルールを削除して修復する仕組みを構築したいと考えています。

この要件を満たすために、最も適切なソリューションはどれでしょうか。

#### 選択肢
A. AWS Configルールの評価結果をCloudWatch Eventsで監視し、Lambda関数を呼び出してセキュリティグループを修正する。

B. AWS Configの修復アクションを設定して、Systems Manager Automationのドキュメントを使用してセキュリティグループを自動修復する。

C. AWS CloudTrailでセキュリティグループの変更イベントを検知し、SNS経由で運用チームに通知して手動で修正する。

D. AWS Trusted Advisorのセキュリティチェックで検出された結果に基づいて、手動でセキュリティグループを修正する。
