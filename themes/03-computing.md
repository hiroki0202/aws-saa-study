# 第3章: コンピューティング

## 座学

## コンピューティングサービスの全体像

### 導入（サーバーとはなにか）

> 「コンピューティングサービスというと難しく聞こえますが、要するに「**プログラムを実行するサーバーをどう用意するか**」という話です。
>
> 以前は、サーバーを用意するためにハードウェアを購入して、データセンターにラックを借りて、OSをインストールして……という作業が必要でした。調達だけで数週間かかる。
>
> AWSはこれを変えました。画面から数クリックで仮想サーバーが立ち上がる。使わないときは止められる。AWSのコンピューティングサービスは大きく3つの方向性があります。**EC2**（仮想サーバー）、**ECS/EKS**（コンテナ）、**Lambda**（サーバーレス）。この3つをどう使い分けるかを今日学びます。」

---

## EC2

### 導入（仮想サーバーとは）

> 「**EC2（Elastic Compute Cloud）**は、AWSで最も基本的なコンピューティングサービスです。仮想サーバーを必要なときに必要なだけ起動できます。
>
> "仮想"というのは、1台の物理サーバーを複数の仮想サーバーに分割して提供しているということです。お客さんはその仮想サーバーを借りて、自分のOSやアプリケーションをインストールして使います。
>
> EC2の特徴は**柔軟性**です。CPUやメモリの量（インスタンスタイプ）を自由に選べる。起動も停止も数分でできる。従来の物理サーバーと同じ感覚で使えるので、オンプレミスからの移行がしやすい。」

### 購入オプション（コスト最適化）

> 「EC2は使い方によって料金を大きく変えられます。
>
> 基本は**オンデマンドインスタンス**。使った分だけ時間単位で課金されます。いつでも起動・停止できる自由度がある。ただし、一番単価が高い。
>
> **リザーブドインスタンス**は、「1年または3年使い続けることを約束する」代わりに最大72%の割引が受けられます。本番環境で常時稼働が必要なシステムには積極的に使うべきオプションです。
>
> **スポットインスタンス**は、AWSの余剰リソースを入札形式で利用する方法です。最大90%という大幅な割引が受けられますが、AWSに余剰がなくなると**2分前通知で強制終了される**ことがあります。突然終了されても問題ない処理——バッチ処理、画像の変換、機械学習の学習ジョブなど——に向いています。
>
> 試験でよく出るのが「コスト最適化」の問題。「安定した本番ワークロード」→ リザーブドインスタンス、「突然の終了を許容できる一時的な処理」→ スポットインスタンス、という判断ができるようにしてください。」

## スケーリングと耐障害性

### 単一障害点を作らない設計

> 「AWSの試験で最も重要な設計思想の1つが「**単一障害点を作らない**」ことです。
>
> 単一障害点（SPOF: Single Point of Failure）とは、そこが壊れるとシステム全体が止まってしまう部分のことです。
>
> たとえば、EC2インスタンスが1台だけでシステムが動いているとします。そのインスタンスが故障したら？ システム全体が停止します。これが単一障害点です。
>
> 解決策は2つの方向性があります。
>
> **スケールアップ**：1台のサーバーをより高性能なものに変える。CPUを増やす、メモリを増やす。わかりやすいですが、スペックには上限があるし、その1台が壊れたら終わりです。
>
> **スケールアウト**：サーバーを複数台に増やす。1台が壊れても他のサーバーが動き続けます。しかも複数台で処理を分担するので、負荷にも強くなる。AWSが推奨する設計方向です。」

---

## ELBとAuto Scaling

### 負荷分散と自動スケーリング

> 「スケールアウト構成を実現するために欠かせないのが**ELB（Elastic Load Balancing）**と**Auto Scaling**です。
>
> 複数のEC2インスタンスが並んでいるとき、ユーザーのリクエストをどのインスタンスに送るかを振り分けるのが**ELB**の役割です。ロードバランサー、つまり「負荷分散装置」ですね。1台のサーバーに処理が集中するのを防いで、均等に分散させます。さらに、故障したインスタンスへのルーティングを自動的に止めてくれます。
>
> ELBには3種類あります。**ALB（Application Load Balancer）**はHTTP/HTTPSのリクエストをURL（パス）や HTTP ヘッダーによって振り分けられる高機能なロードバランサーで、Webアプリケーションに最適です。**NLB（Network Load Balancer）**は超低レイテンシーが必要なときや、TCPレベルでの通信をそのまま転送したいときに使います。
>
> **Auto Scaling**は、負荷に合わせてEC2インスタンスの台数を**自動的に増減**させます。昼間はアクセスが多いから10台、夜中は少ないから2台——というように、コストを最小化しながら必要な処理能力を維持できます。ELBと組み合わせることで「インスタンスを増やしたら自動的に負荷分散の対象に加える」という仕組みができます。」

## ECS

### コンテナとは

> 「次は**コンテナ**の話です。EC2はOSごと仮想化しますが、コンテナは**アプリケーションとその実行に必要な環境だけをパッケージング**する技術です。
>
> 料理の例えで考えてみましょう。EC2は「キッチン付きの部屋を丸ごと借りる」イメージ。コンテナは「お弁当箱に料理と必要な調理器具を全部詰めて持ち歩く」イメージ。お弁当箱はどこに持って行っても、同じように食べられますよね。コンテナも「開発環境では動いたけど本番で動かない」という問題を解消してくれます。
>
> **ECS（Elastic Container Service）**は、AWSでDockerコンテナを管理・実行するサービスです。
>
> ECSの用語を整理しましょう。**Task**は実際に動いているコンテナのこと。**Task Definition**は「このコンテナはCPUをどれだけ使う、環境変数はこれ」というコンテナの仕様書です。**Cluster**はTaskを実行するためのコンピューティングリソースの集まり。
>
> ECSにはインフラ管理が不要な**Fargate**モードがあります。EC2インスタンスの管理をAWSに任せることで、純粋にアプリケーションの開発に集中できます。」

---

## Lambda

### サーバーレスとは

> 「最後は**Lambda**です。「サーバーレス」と呼ばれます。
>
> サーバーがないわけではありません。サーバーの**管理を一切しなくていい**という意味です。OSの管理、スケーリング、可用性の確保——すべてAWSが自動でやってくれます。
>
> Lambdaは**イベント駆動型**です。「S3に画像がアップロードされたら」「APIが呼ばれたら」「定時刻になったら」といったイベントをトリガーに、関数が実行されます。
>
> 料金モデルも特徴的で、**実際に関数が実行された時間だけ課金**されます。誰もアクセスしていない夜中に課金されることはない。アクセスがあったときだけ動いて課金される。
>
> ただし制約もあります。1回の実行で最大**15分**しか動けません。長時間の処理は向きません。また、アクセスが来るまで関数は止まっているため、初回の起動に数百ミリ秒かかる「コールドスタート」問題があります。
>
> まとめると：**EC2**はOSから管理が必要な従来型アプリに、**ECS/Fargate**はコンテナ化されたアプリに、**Lambda**は短い処理をイベント駆動で実行したい場合に——という使い分けが基本です。」

#### 第3回（2/28）ハンズオン①：Lambda + API Gatewayで簡易API構築

> **リソース状況**
> - **作成済み**: なし（すべて新規作成）
> - **新規作成**: Lambda関数、API Gateway

##### ステップ1: Lambda関数を作成（新規作成）

1. コンソール上部の検索バーに「Lambda」と入力 →「Lambda」を選択
2. 「関数の作成」をクリック
3. 「一から作成」を選択:
   - 関数名: `handson-api-function`
   - ランタイム: `Python 3.13`
   - アーキテクチャ: `x86_64`
4. 「関数の作成」をクリック
5. コードエディタで以下に書き換え:
   ```python
   import json
   from datetime import datetime

   def lambda_handler(event, context):
       return {
           'statusCode': 200,
           'headers': {
               'Content-Type': 'application/json'
           },
           'body': json.dumps({
               'message': 'Hello from Lambda!',
               'timestamp': datetime.now().isoformat(),
               'name': '自分の名前をここに入力'
           })
       }
   ```
6. 「Deploy」をクリック
7. 「Test」をクリック → イベント名: `test` →「保存」→「テスト」
8. **確認**: 実行結果が `statusCode: 200` でJSONが返ること

##### ステップ2: API Gatewayを作成（新規作成）

1. コンソール上部の検索バーに「API Gateway」と入力 →「API Gateway」を選択
2. 画面左上の「APIを作成」をクリック
3. APIタイプの選択画面が表示される。いくつか種類があるが、**「REST API」**（Privateではない方）の「構築」をクリック
4. 以下を設定:
   - 「新しいAPI」が選択されていることを確認
   - API名: `handson-api`
   - APIエンドポイントタイプ: `リージョン`
   - 説明: 空欄のままでOK
5. 「APIを作成」をクリック

##### ステップ3: リソースとメソッドを作成

> API Gatewayでは「リソース」がURLのパス（例: `/hello`）、「メソッド」がHTTPメソッド（例: `GET`）に対応します。ここではブラウザから `GET /hello` でアクセスできるエンドポイントを作成します。

1. 左メニュー「リソース」を選択 → リソースツリーに `/` が表示されていることを確認
2. 「リソースを作成」をクリック
   - リソースパス: `/`（デフォルトのまま）
   - リソース名: `hello`
3. 「リソースを作成」をクリック
4. リソースツリーで `/hello` を選択した状態で「メソッドを作成」をクリック
5. 以下を設定:
   - メソッドタイプ: `GET`
   - 統合タイプ: `Lambda関数`
   - Lambdaプロキシ統合: **オン**
   - Lambda関数: 検索欄に `handson-api-function` と入力して選択
6. 「メソッドを作成」をクリック
7. 「API Gateway に Lambda 関数を呼び出す権限を付与しますか？」というダイアログが表示されたら「OK」をクリック

##### ステップ4: APIをデプロイ

1. 「APIをデプロイ」をクリック
2. ステージ:「新しいステージ」→ ステージ名: `prod`
3. 「デプロイ」をクリック
4. 表示される「URLを呼び出す」のURLをコピー

##### ステップ5: 動作確認

1. コピーしたURLの末尾に `/hello` を追加してブラウザで開く
   - 例: `https://xxxxxxxx.execute-api.ap-northeast-1.amazonaws.com/prod/hello`
2. **確認**: JSONレスポンスが表示される（message, timestamp, name）
3. 何度かリロード → timestampが更新されることを確認

##### リソース削除

1. API Gatewayコンソール → `handson-api` を選択 →「APIを削除」
2. Lambda コンソール → `handson-api-function` を選択 →「アクション」→「関数の削除」

#### 第3回（2/28）ハンズオン②：EC2 + ALBで負荷分散

> **リソース状況**
> - **作成済み（第1回）**: `VPC-A`、`VPC-A-Public`（ap-northeast-1a）、`IGW-A`、ルートテーブル、キーペア
> - **新規作成**: `VPC-A-Public-2`（サブネット）、`SG-ALB`、`SG-EC2`、`Web-1`、`Web-2`、ターゲットグループ、ALB

##### ステップ1: 2つ目のパブリックサブネットを追加（新規作成）

> ALBは2つ以上のAZが必要です。第1回で作成した `VPC-A` には `ap-northeast-1a` のサブネットのみ存在するため、`ap-northeast-1c` にサブネットを追加します。

1. VPCコンソール → 左メニュー「サブネット」→「サブネットを作成」
2. 以下を入力：
   - VPC: `VPC-A` を選択（**作成済み**）
   - サブネット名: `VPC-A-Public-2`
   - アベイラビリティゾーン: `ap-northeast-1c`
   - IPv4 CIDR: `10.0.2.0/24`
3. 「サブネットを作成」をクリック
4. 左メニュー「ルートテーブル」→ `VPC-A` に関連付けられたルートテーブルを選択（**作成済み**）
5. 「サブネットの関連付け」タブ →「明示的なサブネットの関連付け」セクションの「サブネットの関連付けを編集」→ `VPC-A-Public-2` にチェック →「関連付けを保存」

##### ステップ2: セキュリティグループを作成（新規作成）

1. VPCコンソール → 左メニュー「セキュリティグループ」→「セキュリティグループを作成」
2. 以下を設定:
   - 名前: `SG-ALB`
   - 説明: `ALB security group`
   - VPC: `VPC-A`（**作成済み**）
   - インバウンドルール: タイプ `HTTP`、ソース `0.0.0.0/0`
3. 「セキュリティグループを作成」
4. もう1つ作成:
   - 名前: `SG-EC2`
   - 説明: `EC2 security group`
   - VPC: `VPC-A`（**作成済み**）
   - インバウンドルール: タイプ `HTTP`、ソース → `SG-ALB` のセキュリティグループIDを指定
     > ソースにはIPアドレスだけでなく、セキュリティグループIDも指定できます。検索欄に `SG-ALB` と入力すると候補が表示されるので選択してください。これにより「`SG-ALB` が割り当てられたリソース（＝ALB）からのHTTP通信のみ許可」という意味になります。

##### ステップ3: EC2を2台起動（新規作成・ユーザーデータでWebサーバー自動構築）

1. EC2コンソール →「インスタンスを起動」
2. 以下を設定:
   - 名前: `Web-1`
   - AMI: Amazon Linux 2023
   - インスタンスタイプ: `t2.micro`
   - キーペア: 第1回で作成したものを選択（**作成済み**）
   - ネットワーク設定 →「編集」:
     - VPC: `VPC-A`（**作成済み**）
     - サブネット: `VPC-A-Public`（`ap-northeast-1a`）（**作成済み**）
     - パブリックIPの自動割り当て: `有効化`
     - セキュリティグループ: `SG-EC2`
   - 「高度な詳細」を展開 →「ユーザーデータ」に以下を入力（ユーザーデータはEC2インスタンスの初回起動時に自動実行されるスクリプトです。ここではWebサーバー（nginx）のインストールと起動を自動化しています）:
     ```bash
     #!/bin/bash
     dnf install -y nginx
     systemctl start nginx
     systemctl enable nginx
     INSTANCE_ID=$(ec2-metadata -i | cut -d ' ' -f 2)
     echo "<h1>Hello from $INSTANCE_ID</h1>" > /usr/share/nginx/html/index.html
     ```
3. 「インスタンスを起動」
4. 同様にもう1台:
   - 名前: `Web-2`
   - サブネット: `VPC-A-Public-2`（`ap-northeast-1c`）（**ステップ1で新規作成**）
   - ユーザーデータは同じ内容

##### ステップ4: ターゲットグループを作成（新規作成）

1. EC2コンソール → 左メニュー「ターゲットグループ」→「ターゲットグループの作成」
2. 以下を設定:
   - ターゲットタイプ: `インスタンス`
   - ターゲットグループ名: `handson-tg`
   - プロトコル: `HTTP`、ポート: `80`
   - VPC: `VPC-A`（**作成済み**）
   - ヘルスチェックパス: `/`
3. 「次へ」→ `Web-1` と `Web-2` にチェック →「保留中として以下を含める」をクリック
   > ターゲットはまず「保留中」として追加され、ターゲットグループの作成完了後に正式に登録されます。これはヘルスチェックの設定が確定する前にトラフィックが流れることを防ぐための仕組みです。
4. 「ターゲットグループの作成」

##### ステップ5: ALBを作成（新規作成）

1. EC2コンソール → 左メニュー「ロードバランサー」→「ロードバランサーの作成」
2. 「Application Load Balancer」の「作成」をクリック
3. 以下を設定:
   - 名前: `handson-alb`
   - スキーム: `インターネット向け`
   - ネットワークマッピング: VPC `VPC-A` → `ap-northeast-1a`（`VPC-A-Public`）と `ap-northeast-1c`（`VPC-A-Public-2`）にチェック
   - セキュリティグループ: `SG-ALB`（デフォルトのSGは外す）
   - リスナー: HTTP:80 → デフォルトアクション: `handson-tg` を選択
4. 「ロードバランサーの作成」

##### ステップ6: 動作確認

1. ロードバランサー一覧から `handson-alb` を選択
2. 状態が `active` になるまで待つ（2〜3分）
3. 「DNS名」をコピー → ブラウザのアドレスバーに `http://` を付けて開く（例: `http://handson-alb-xxxxx.ap-northeast-1.elb.amazonaws.com`）
   > ブラウザが自動的に `https://` に変換する場合がありますが、今回はHTTPのみ設定しているため、必ず `http://` で始めてください。
4. **確認**: 「Hello from i-xxxxx」と表示される
5. ブラウザを何度かリロード →  表示されるインスタンスIDが切り替わることを確認（負荷分散）

##### リソース削除

> **注意**: `VPC-A`、`VPC-A-Public`、`VPC-A-Public-2`、`IGW-A`、ルートテーブル、`EC2-A` は第4回以降のハンズオンで継続的に再利用します。**削除しないでください。**

1. ALB `handson-alb` を削除
2. ターゲットグループ `handson-tg` を削除
3. EC2インスタンス `Web-1`、`Web-2` を終了（`EC2-A` は残す）
4. セキュリティグループを削除（`SG-VPC-A` は残す）:
   - 先に `SG-EC2` を削除（`SG-ALB` を参照しているため、こちらを先に削除する必要があります）
   - 次に `SG-ALB` を削除

#### 第3回（2/28）ハンズオン③：ECS Fargate でコンテナを動かす

> **ハンズオン②との比較**
> ハンズオン②ではEC2にユーザーデータで `dnf install nginx` → `systemctl start nginx` と手動インストール相当の手順を踏みました。コンテナなら nginx の公式イメージを指定するだけで、同じWebサーバーが即座に起動します。「OS にソフトをインストールする」世界と「完成済みのコンテナを実行する」世界の違いを体感してください。

> **リソース状況**
> - **作成済み（第1回・ハンズオン②）**: `VPC-A`、`VPC-A-Public`（ap-northeast-1a）、`VPC-A-Public-2`（ap-northeast-1c）、`IGW-A`、ルートテーブル
> - **新規作成**: ECSクラスター、タスク定義、セキュリティグループ（`SG-ECS`）、ECSサービス

##### ステップ1: ECS サービスリンクロールを作成（初回のみ）

> ECS を初めて使うアカウントでは、ECS が内部的に使う IAM ロール（サービスリンクロール）が存在しないため、クラスター作成時にエラーになります。CloudShell からコマンドを1つ実行して作成します。

1. コンソール画面下部の「CloudShell」アイコン（`>_` マーク）をクリック → ターミナルが開く
2. 以下のコマンドを貼り付けて Enter:
   ```
   aws iam create-service-linked-role --aws-service-name ecs.amazonaws.com
   ```
3. JSON が返ってくれば成功。CloudShell は閉じてOK

##### ステップ2: ECS クラスターを作成（新規作成）

1. コンソール上部の検索バーに「ECS」と入力 →「Elastic Container Service」を選択
2. 左メニュー「クラスター」→「クラスターの作成」をクリック
3. 以下を設定:
   - クラスター名: `handson-cluster`
   - インフラストラクチャ: 「Fargateのみ」が選択されていることを確認（デフォルトで選択済み）
     > Fargate を使うと、EC2 インスタンスを自分で管理する必要がありません。コンテナの実行に必要なコンピューティングリソースを AWS が自動でプロビジョニングします。
4. 「作成」をクリック

##### ステップ3: タスク定義を作成（新規作成）

> タスク定義は「どのコンテナイメージを使い、どれくらいの CPU・メモリを割り当て、どのポートを開けるか」をまとめた設計図です。

1. ECSコンソール → 左メニュー「タスク定義」→「新しいタスク定義の作成」をクリック
2. 以下を設定:
   - タスク定義ファミリー: `handson-nginx`
   - 起動タイプ: `AWS Fargate`
   - オペレーティングシステム/アーキテクチャ: `Linux/X86_64`
   - CPU: `.25 vCPU`
   - メモリ: `.5 GB`
   - タスクロール: 空欄のまま
   - タスク実行ロール:「デフォルトロールを作成」のまま進める
     > タスク実行ロールは、ECS がコンテナイメージの取得やログの書き込みを行うために必要な IAM ロールです。このまま進めると `ecsTaskExecutionRole` というロールが自動的に作成されます。
3. 「コンテナ - 1」セクション:
   - 名前: `nginx`
   - イメージ URI: `public.ecr.aws/nginx/nginx:latest`
     > これは Amazon ECR Public Gallery で公開されている nginx の公式イメージです。Docker Hub（コンテナイメージの有名な配布サイト）にも同じイメージがありますが、Docker Hub は無料だと短時間のダウンロード回数に制限があり、大人数で同時にダウンロードするとエラーになる可能性があります。ECR Public（AWS提供）にはその制限がないため、こちらを使用しています。
   - コンテナポート: `80`、プロトコル: `TCP`、ポート名: 空欄、アプリケーションプロトコル: `HTTP`
4. 「作成」をクリック

##### ステップ4: セキュリティグループを作成（新規作成）

1. VPCコンソール → 左メニュー「セキュリティグループ」→「セキュリティグループを作成」
2. 以下を設定:
   - 名前: `SG-ECS`
   - 説明: `ECS security group`
   - VPC: `VPC-A`（**作成済み**）
   - インバウンドルール: タイプ `HTTP`、ソース `0.0.0.0/0`
3. 「セキュリティグループを作成」

##### ステップ5: サービスを作成して実行（新規作成）

> サービスは「タスク定義をもとに、指定した数のタスク（コンテナ）を常に維持する」仕組みです。タスクが停止しても自動的に新しいタスクが起動されます。

1. ECSコンソール → 左メニュー「クラスター」→ `handson-cluster` をクリック
2. 「サービス」タブ →「作成」をクリック
3. 以下を設定:
   - **デプロイ設定**
     - ファミリー: `handson-nginx`
     - リビジョン: `LATEST`
     - サービス名: `handson-nginx-service`
     - 必要なタスク: `1`
   - **ネットワーキング**
     - VPC: `VPC-A`（**作成済み**）
     - サブネット: `VPC-A-Public` のみ選択（他のサブネットが選択されている場合は外す）
     - セキュリティグループ: 既存のSGを選択 → `SG-ECS` を選択（デフォルトのSGは外す）
     - パブリック IP: `オン`
       > Fargate タスクにパブリック IP を付与することで、ALB なしでも直接アクセスできます。本番環境では ALB を前段に置くのが一般的ですが、今回は手軽さを優先してパブリック IP で直接アクセスします。
4. 「作成」をクリック
5. サービスのデプロイが始まります。「タスク」タブでタスクの「最後のステータス」が `RUNNING` になるまで待ちます（1〜2分）

##### ステップ6: 動作確認

1. ECSコンソール → `handson-cluster` → `handson-nginx-service` →「タスク」タブ
2. 実行中のタスクIDをクリック
3. 「設定」セクションの「パブリック IP」をコピー
4. ブラウザのアドレスバーに `http://<パブリックIP>` を入力してアクセス
   > ブラウザが自動的に `https://` に変換する場合がありますが、今回はHTTPのみ設定しているため、必ず `http://` で始めてください。
5. **確認**: nginx のデフォルトページ（「Welcome to nginx!」）が表示される

> **ハンズオン②との比較まとめ**
>
> | | ハンズオン②（EC2） | ハンズオン③（ECS Fargate） |
> |---|---|---|
> | コンピューティング | EC2インスタンス（OS管理が必要） | Fargate（OS管理不要） |
> | nginx の準備 | ユーザーデータで `dnf install` + `systemctl start` | コンテナイメージで即起動 |
> | スケーリング | 手動でインスタンス追加 or Auto Scaling設定 | タスク数を変更するだけ |
> | 課金 | インスタンス起動中は常に課金 | タスク実行中のみ課金（秒単位） |

##### リソース削除

> **注意**: `VPC-A`、サブネット、`IGW-A`、ルートテーブルは第4回以降のハンズオンで継続的に再利用します。**削除しないでください。**

1. ECSコンソール → `handson-cluster` → `handson-nginx-service` →「サービスを削除」
   - 確認画面で「強制削除」にチェックを入れて削除（実行中のタスクも同時に停止されます）
2. ECSコンソール → 左メニュー「タスク定義」→ `handson-nginx` → 該当リビジョンにチェック →「登録解除」
3. ECSコンソール → 左メニュー「クラスター」→ `handson-cluster` →「クラスターの削除」
   - 確認欄に `delete handson-cluster` と入力して削除
4. VPCコンソール → セキュリティグループ → `SG-ECS` を削除

## 練習問題

### 問題1

あなたはソリューションアーキテクトとしてAmazon DynamoDBテーブルにデータを書き込むためのAWS Lambda関数を作成しました。 このLambda関数がDynamoDBテーブルを操作できるように権限を設定する必要があります。

どのように権限を設定すればよいでしょうか。

<details>
<summary>選択肢を見る</summary>

A. Lambda関数に適切な権限を付与したIAMユーザーを設定してDynamoDBへのアクセスを許可する。

B. Lambda関数に適切な権限を付与したIAMロールを設定してDynamoDBへのアクセスを許可する。

C. Lambda関数に適切な権限を付与したIAMポリシーを設定してDynamoDBへのアクセスを許可する。

D. Lambda関数に適切な権限を付与したリソースポリシーを設定してDynamoDBへのアクセスを許可する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
B

### 解説
Lambda関数がDynamoDBなど他のAWSサービスにアクセスするには、Lambda関数にIAMロール（実行ロール）をアタッチする必要があります。

- Lambda関数の実行ロールには、DynamoDBへの書き込み権限（dynamodb:PutItemなど）を含むIAMポリシーをアタッチする
- Lambda関数が実行されると、このロールの権限を引き受けて（AssumeRole）DynamoDBにアクセスする
- AWSサービス間のアクセスにはIAMロールを使用するのがベストプラクティス

### 他の選択肢が不適切な理由
A
IAMユーザーは人間やアプリケーションが長期的な認証情報（アクセスキー等）で利用するものです。
Lambda関数のようなAWSサービスにはIAMロールを使用するのがベストプラクティスであり、IAMユーザーをLambda関数に設定する仕組みはありません。

C
IAMポリシーは権限のルールを定義するドキュメントであり、単体でLambda関数にアタッチすることはできません。
IAMポリシーはIAMロールにアタッチして使用します。

D
リソースポリシーは、Lambda関数を「誰が呼び出せるか」を制御するものです。
Lambda関数が他のサービスにアクセスするための権限とは異なります。

### まとめ
Lambda関数が他のAWSサービスにアクセスするには、IAMロール（実行ロール）を使用する。
IAMロールに適切なIAMポリシーをアタッチすることで、DynamoDBへのアクセスが可能になる。

</details>

### 問題2

ある企業では、EC2インスタンス上でバッチ処理ワークロードを運用しています。このワークロードは複数のAmazon EC2インスタンスを活用しており、ステートレスな特性を持っています。そのため、インスタンス処理を途中で停止したり再開したりすることが容易です。全体の処理時間は約1時間を要します。あなたはソリューションアーキテクトとして、コストの最適化を求められています。

コスト最適化を実現するために、ソリューションアーキテクトはどの購入オプションを選択すべきでしょうか。

<details>
<summary>選択肢を見る</summary>

A. スポットインスタンスを利用する。

B. リザーブドインスタンスを利用する。

C. オンデマンドインスタンスを利用する。

D. ベアメタルインスタンスを利用する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
A

### 解説
スポットインスタンスは、AWSの未使用キャパシティを活用できる購入オプションで、オンデマンドと比べて最大90%のコスト削減が可能です。

このワークロードがスポットインスタンスに適している理由は以下の通りです。

- ステートレスであり、中断されても再開が容易
- 処理時間が約1時間と短時間
- 複数インスタンスで分散処理しているため、一部が中断されても全体への影響が限定的

スポットインスタンスはAWSの需要状況により中断される可能性がありますが、中断耐性のあるワークロードであれば大幅なコスト削減が実現できます。

### 他の選択肢が不適切な理由
B
リザーブドインスタンスは1年または3年の長期契約で割引を受ける購入方式です。
約1時間のバッチ処理には長期契約は不要であり、コスト最適化の観点で適切ではありません。

C
オンデマンドインスタンスは利用した分だけ支払う方式で柔軟ですが、スポットインスタンスと比較するとコストが高くなります。
中断耐性のあるワークロードではスポットインスタンスの方がコスト最適です。

D
ベアメタルインスタンスはハードウェアに直接アクセスできる高性能なインスタンスです。
コスト最適化が目的の場合に選択するインスタンスタイプではありません。

### まとめ
ステートレスで中断耐性のあるバッチ処理ワークロードには、スポットインスタンスが最もコスト効率が高い。
スポットインスタンスはオンデマンドと比べて最大90%の割引が可能。

</details>

### 問題3

ある会社はAmazon EC2インスタンスにホストされているウェブサイトを運用しています。このウェブサイトのEC2インスタンスはHTTPトラフィックとHTTPSトラフィックを別々に処理するALBのターゲットグループに構成されています。この企業は、すべてのリクエストをこのウェブサイトに転送して、リクエストがHTTPSを使用するようにしたいと考えています。

この要件を満たすために、ソリューションアーキテクトはどうすればよいでしょうか。

<details>
<summary>選択肢を見る</summary>

A. HTTPSトラフィックだけを許可するようにALBのネットワークACLルールを設定する。

B. HTTPをHTTPSに変換するリスナールールをALBで設定する。

C. HTTPトラフィックをHTTPSトラフィックにリダイレクトするリスナールールをALBで設定する。

D. ALBをNLBに置き換えて、HTTPトラフィックをHTTPSトラフィックにリダイレクトするリスナールールを設定する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
C

### 解説
ALBのリスナールールでは、HTTPリクエストをHTTPSにリダイレクトするアクションを設定できます。

具体的な設定方法は以下の通りです。

- ALBのHTTP:80リスナーにリダイレクトアクションを設定する
- リダイレクト先のプロトコルをHTTPS、ポートを443に指定する
- ステータスコード301（永続的リダイレクト）でクライアントにHTTPS URLを返す
- クライアントは自動的にHTTPSで再リクエストを行う

これにより、HTTPでアクセスしたユーザーも自動的にHTTPS通信に切り替わります。

### 他の選択肢が不適切な理由
A
ネットワークACLでHTTPSだけを許可すると、HTTPでアクセスしたユーザーは接続が拒否されてしまいます。
「すべてのリクエストをウェブサイトに転送」という要件を満たしません。

B
「HTTPをHTTPSに変換する」という表現は不正確です。
ALBが行うのはリダイレクト（HTTPステータス301/302で別URLへ転送）であり、プロトコルの変換ではありません。

D
ALBをNLBに置き換える必要はありません。
ALB単体でHTTPからHTTPSへのリダイレクトが可能であり、NLBへの変更は不要な構成変更です。

### まとめ
HTTPからHTTPSへの自動切り替えは、ALBのHTTPリスナーにリダイレクトルールを設定することで実現できる。
ネットワークACLによるブロックではなく、リダイレクトで対応することがポイント。

</details>

### 問題4

あなたはソリューションアーキテクトとして、Webサーバーとデータベースサーバーから構成されるWebアプリケーションのアーキテクチャを設計しています。Webサーバーとデータベースサーバーは、それぞれ異なるサブネットに配置されたEC2インスタンス上にホストされる予定です。この際、セキュリティを強化するために、データベースサーバーはWebサーバーからのトラフィックのみを受け入れる必要があります。また、このWebサーバーにはAuto Scalingグループを設定して、負荷に応じてインスタンスの数を増減させる必要があります。

この要件を満たすための適切なトラフィック設定方法はどれでしょうか。

<details>
<summary>選択肢を見る</summary>

A. DBインスタンスのセキュリティグループのインバウンドルールにWEBサーバーインスタンスのプライベートIPアドレスを設定する。

B. DBインスタンスのセキュリティグループのインバウンドルールにWEBサーバーインスタンスのセキュリティグループIDを設定する。

C. DBインスタンスのセキュリティグループのインバウンドルールにWEBサーバーインスタンスのElastic IPアドレスを設定する。

D. DBインスタンスのセキュリティグループのインバウンドルールにWEBサーバーインスタンスのパブリックIPアドレスを設定する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
B

### 解説
セキュリティグループのインバウンドルールでは、ソースとして別のセキュリティグループIDを指定できます。

この設定方法が適切な理由は以下の通りです。

- Auto Scalingによりインスタンスが増減するため、個々のIPアドレスを指定する方法では管理が困難
- セキュリティグループIDを指定すれば、そのセキュリティグループに属するすべてのインスタンスからのトラフィックが自動的に許可される
- インスタンスが追加・削除されても、セキュリティグループの設定変更は不要
- Webサーバーのセキュリティグループに属するインスタンスのみがDBにアクセスできるため、セキュリティも確保される

### 他の選択肢が不適切な理由
A
プライベートIPアドレスを指定する方法では、Auto Scalingでインスタンスが増減するたびにセキュリティグループのルールを手動で更新する必要があります。
運用負荷が高く、設定漏れのリスクもあるため適切ではありません。

C
Elastic IPアドレスはインスタンスごとに個別に割り当てる静的IPです。
Auto Scalingで自動起動されるインスタンスにElastic IPを割り当てるのは現実的ではなく、管理が煩雑になります。

D
パブリックIPアドレスは動的に割り当てられるため、インスタンスの再起動で変わる可能性があります。
また、DBサーバーへの通信にパブリックIPを使用するのはセキュリティ上も不適切です。

### まとめ
Auto Scaling環境では、セキュリティグループのソースにセキュリティグループIDを指定するのがベストプラクティス。
インスタンスの増減に自動対応でき、運用負荷を最小限に抑えられる。

</details>

### 問題5

ある企業はEC2インスタンス上にアプリケーションを構築しました。アプリケーションのトラフィック急増時にも円滑に処理を続けられるよう、これらのインスタンスにはAuto Scalingグループが設定されています。しかし、最近、毎日17時から17時半の間にアプリケーションのパフォーマンスが急激に低下する問題が発生しています。この負荷増加が短時間であるため、Auto Scalingによるインスタンスの追加が間に合っていないようです。

ソリューションアーキテクトとして、どのようにこの問題を改善するべきでしょうか。

<details>
<summary>選択肢を見る</summary>

A. Auto Scalingが起動するインスタンス数の最大数を増加する。

B. ELBによるロードバランシングを設定する。

C. Route53によるトラフィックルーティングを設定する。

D. Auto Scalingに対して、スケジュールされたスケーリングポリシーを追加する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
D

### 解説
毎日17時から17時半という予測可能なパターンで負荷が増加しているため、スケジュールされたスケーリング（Scheduled Scaling）が最適です。

スケジュールされたスケーリングのポイントは以下の通りです。

- 事前に決められた時間にインスタンス数を増やす設定ができる
- 例えば16時50分にインスタンスを追加しておくことで、17時の負荷増加に備えられる
- 動的スケーリングでは間に合わない短時間の負荷増加に対応可能
- 負荷パターンが予測可能な場合に最も効果的

### 他の選択肢が不適切な理由
A
最大数を増やしても、Auto Scalingがスケールアウトするタイミングは変わりません。
問題の本質は「スケーリングが間に合わない」ことであり、最大数の問題ではありません。

B
ELBは既存のインスタンス間で負荷を分散しますが、インスタンス数を増やす機能はありません。
インスタンスが不足している状態では、ELBを追加しても根本的な解決にはなりません。

C
Route 53はDNSサービスであり、トラフィックルーティングはDNSレベルでの振り分けです。
短時間の負荷急増に対する即時のスケーリング対応としては不適切です。

### まとめ
予測可能な時間帯の負荷増加に対しては、スケジュールされたスケーリングを使用して事前にインスタンスを準備するのが最適。
動的スケーリングが間に合わないケースでは、時間ベースの事前スケーリングが有効。

</details>

### 問題6

ある会社は複数のAmazon EC2インスタンスとAmazon RDS DBインスタンスを利用したアプリケーションを運用しています。あなたはソリューションアーキテクトとして、このアプリケーションのコスト効率を改善することが求められています。そのために、あなたは業務時間外にEC2インスタンスとDBインスタンスを自動的に起動・停止してコストを削減することにしました。

これらの要件を満たすソリューションはどれでしょうか。

<details>
<summary>選択肢を見る</summary>

A. EC2インスタンスとDBインスタンスを起動・停止するcronジョブを設定する。このジョブをAmazon EventBridgeスケジュールに設定する。

B. EC2インスタンスとDBインスタンスを起動・停止するLambda関数を作成する。このLambda関数をAmazon EventBridgeスケジュールに設定する。

C. EC2インスタンスとDBインスタンスを起動・停止するStepFunctionsワークフローを作成する。このワークフローをAmazon EventBridgeスケジュールに設定する。

D. スケジュールドスケーリングを利用したAuto Scaling グループを使用して、EC2インスタンスのスケーリングを行う。業務時間外はDBインスタンスを０にスケールさせる。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
B

### 解説
Lambda関数とAmazon EventBridgeスケジュールの組み合わせは、定期的なタスクの自動実行に最適なサーバーレスソリューションです。

この構成のポイントは以下の通りです。

- Lambda関数でEC2のstart/stop APIとRDSのstart/stop APIを呼び出す
- EventBridgeスケジュールで業務開始時刻と終了時刻にLambda関数をトリガーする
- サーバーレスのため、Lambda関数自体の常時稼働コストは不要
- 実行時間も短いため、Lambda関数のコストはほぼ無視できる

### 他の選択肢が不適切な理由
A
cronジョブはLinuxのスケジュール実行機能であり、EventBridgeスケジュールに直接設定するものではありません。
cronジョブを実行するには常時稼働するサーバーが必要となり、コスト効率が悪くなります。

C
Step Functionsはワークフローの管理に適していますが、EC2とRDSの起動・停止という単純な処理にはオーバーエンジニアリングです。
Lambda関数で十分対応でき、よりシンプルかつコスト効率の良いソリューションです。

D
Auto Scalingのスケジュールドスケーリングはインスタンス数の増減に使えますが、RDS DBインスタンスの起動・停止には対応していません。
また、RDSインスタンスを「0にスケール」することはRDSの標準機能では直接的にはできません。

### まとめ
業務時間外のEC2・RDSの自動起動・停止には、Lambda関数とEventBridgeスケジュールの組み合わせが最適。
サーバーレスで運用コストを最小限に抑えつつ、定期的な起動・停止を自動化できる。

</details>

### 問題7

ある企業がAWS上で3層アプリケーションを開発しています。あなたはソリューションアーキテクトとして、このアプリケーションを複数のマイクロサービスに分割し、疎結合化することが求められています。その際、マイクロサービスの構築にはコンテナを利用します。また、運用要件として、AWS上でのコンテナ用コンピュートリソースの構成管理を不要にすることが求められています。

この要件を満たすために、どのようなアプローチを取るべきでしょうか。

<details>
<summary>選択肢を見る</summary>

A. Amazon ECSクラスターをEC2起動モードでプロビジョニングする。ECSクラスターのEC2ノードにAmazon EC2 Auto Scalingグループをアタッチし、コンテナにタスクを定義する。

B. AWS Lambda関数を作成して、マイクロサービスのコンポーネントを構成する。これらの関数をAmazon API Gatewayと統合して、スケーリングとトラフィック制御を実施する。

C. Amazon ECSクラスターをEC2起動モードでプロビジョニングする。コンテナにEC2タスクをデプロイする。

D. Amazon ECSクラスターをFargate起動モードでプロビジョニングする。コンテナにFargateタスクをデプロイする。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
D

### 解説
要件は「コンテナを使用する」かつ「コンピュートリソースの構成管理を不要にする」の2つです。

AWS Fargate は以下の特徴を持ちます。

- コンテナのためのサーバーレスコンピュートエンジン
- EC2インスタンスのプロビジョニング、パッチ適用、スケーリングの管理が不要
- コンテナの実行に必要なリソース（CPU、メモリ）を指定するだけで自動的にインフラが割り当てられる
- ECSと統合して、タスク定義に基づいてコンテナを実行できる

これにより、コンテナを使いながら、基盤となるコンピュートリソースの管理を完全にAWSに任せることができます。

### 他の選択肢が不適切な理由
A
EC2起動モードでは、EC2インスタンスの管理（プロビジョニング、スケーリング、パッチ適用など）が必要です。
「コンピュートリソースの構成管理を不要にする」という要件を満たしません。

B
Lambda関数はサーバーレスでコンピュートリソースの管理は不要ですが、コンテナベースのマイクロサービスアーキテクチャには適していません。
Lambdaはコンテナイメージでの実行もサポートしていますが、実行時間の制限（最大15分）やコールドスタートの特性があり、常時稼働型のマイクロサービスとして運用するにはECS/Fargateの方が適切です。

C
Aと同様にEC2起動モードのため、EC2インスタンスの管理が必要です。
コンピュートリソースの構成管理が不要という要件を満たしません。

### まとめ
コンテナを使いつつコンピュートリソースの管理を不要にしたい場合は、ECS + Fargate起動モードが最適。
Fargateはコンテナのためのサーバーレスコンピュートエンジンであり、インフラ管理をAWSに任せられる。

</details>

### 問題8

ある企業は、単一リージョン内の2つのアベイラビリティゾーン（AZ）に配置されたAmazon EC2インスタンスに対してALBおよびAuto Scalingが設定されたWebアプリケーションを運営しています。運用要件として、1つのAZがダウンし、かつAuto Scalingが残るAZで新たなインスタンスを起動できていない期間においても、アプリケーションが常に100％のピークロードが処理できるパフォーマンスを維持する必要があります。

この要件を満たすために必要なアーキテクチャの構成はどれでしょうか。

<details>
<summary>選択肢を見る</summary>

A. 3つのAZに対して、AZ毎に50％のピークロードを処理できるAuto Scaling設定を行ったEC2インスタンスを展開する。

B. 3つのAZに対して、AZ毎に30％のピークロードを処理できるAuto Scaling設定を行ったEC2インスタンスを展開する。

C. ２つのAZに対して、リージョン毎に50％のピークロードを処理できるAuto Scaling設定を行ったEC2インスタンスを展開する。

D. ２つのAZに対して、AZ毎に50％のピークロードを処理できるAuto Scaling設定を行ったEC2インスタンスを展開する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
A

### 解説
この問題のポイントは「1つのAZがダウンしても、Auto Scalingが追いつく前の時点で100%のピークロードを処理できる」ことです。

計算で考えると以下の通りです。

- 3つのAZに各50%のキャパシティ → 合計150%
- 1つのAZがダウン → 残り2つのAZで50% × 2 = 100%

これにより、AZ障害発生直後でも100%のピークロードを維持できます。

### 他の選択肢が不適切な理由
B
3つのAZに各30%のキャパシティの場合、合計90%です。
1つのAZがダウンすると残り2つで30% × 2 = 60%となり、100%のピークロードを処理できません。

C
「リージョン毎に50%」では合計50%しかなく、AZ障害時にも50%以下の処理能力しか確保できません。
100%のピークロード処理には不十分です。

D
2つのAZに各50%のキャパシティの場合、合計100%です。
1つのAZがダウンすると残り1つで50%となり、100%のピークロードを処理できません。

### まとめ
AZ障害時にも100%のピークロードを維持するには、N+1の冗長構成が必要。
3つのAZに各50%のキャパシティを配置すれば、1つのAZがダウンしても残りの2つで100%を維持できる。

</details>

### 問題9

ある企業がオンプレミス環境からAWSへの移行を決定しました。この移行を円滑に進めるため、同社はコンサルティング会社と契約しました。移行作業においては、同社のAWSアカウントにあるAmazon EC2インスタンスのAmazon マシンイメージ (AMI) をコンサルティング会社のAWSアカウントと共有する必要があります。このAMIはAmazon S3バケットにバックアップされており、AWS KMSのカスタマー管理キーを用いてスナップショットの暗号化も行われています。あなたはソリューションアーキテクトとして、このAMIの共有設定を行う役割を担っています。

この要件を満たすために、ソリューションアーキテクトはどのような手順を踏むべきでしょうか。

<details>
<summary>選択肢を見る</summary>

A. 暗号化されたAMIを公開して、コンサルティング会社のAWSアカウントがアクセスできるようにする。AWS KMSのキーポリシーを変更して、コンサルティング会社のAWSアカウントが暗号化に使用されたKMSキーを使用できるようにする。

B. AMIをコンサルティング会社のAWSアカウントのみと共有できるようにAMIのLaunchPermissionプロパティを変更する。AWS KMSのキーポリシーを変更して、コンサルティング会社のAWSアカウントが暗号化に使用されたKMSキーを使用できるようにする。

C. AMIをコンサルティング会社のAWSアカウントのみと共有できるようにAMIのLaunchPermissionプロパティを変更する。AWS KMSのキーポリシーを変更して、コンサルティング会社の所有する新しいKMSキーを許可して、このAMIに適用する。

D. 暗号化されたAMIを公開して、コンサルティング会社のAWSアカウントがアクセスできるようにする。AWS KMSのキーポリシーを変更して、コンサルティング会社の所有する新しいKMSキーを許可して、このAMIに適用する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
B

### 解説

#### AMI（Amazon Machine Image）とは
AMIはEC2インスタンスを起動するためのテンプレートです。OS、アプリケーション、設定情報などがまとめてイメージ化されており、AMIから同じ構成のEC2インスタンスを何度でも起動できます。自社で作成したカスタムAMIを別のアカウントと共有することで、同じ環境を再構築する手間を省けます。

#### LaunchPermissionとは
LaunchPermissionは、AMIを使ってEC2インスタンスを起動できるアカウントを制御するプロパティです。デフォルトではAMIの所有者のみが使用できますが、LaunchPermissionに他のAWSアカウントIDを追加することで、特定のアカウントにだけAMIの使用を許可できます。AMIを「公開（public）」にすれば全アカウントから利用可能になりますが、暗号化されたAMIは公開に設定できないというAWSの制約があります。

#### 必要な設定
暗号化されたAMIを別のAWSアカウントと共有するには、以下の2つの設定が必要です。

1. **AMIのLaunchPermissionを変更**して、対象アカウントのみにアクセスを許可する
   - LaunchPermissionに対象アカウントIDを追加する
   - これにより、指定されたアカウントのみがAMIを使用できる

2. **KMSキーポリシーを変更**して、対象アカウントに暗号化キーの使用を許可する
   - AMIのスナップショットが暗号化されているため、起動時にキーへのアクセスが必要
   - 対象アカウントに kms:DescribeKey、kms:ReEncrypt*、kms:CreateGrant、kms:Decrypt の権限を付与する

### 他の選択肢が不適切な理由
A / D
暗号化されたAMIを「公開」することはセキュリティ上のベストプラクティスに反します。
特定のアカウントとのみ共有すべきであり、LaunchPermissionで対象を限定するのが正しいアプローチです。
また、暗号化されたAMIはそもそも公開（public）に設定できないというAWSの制約もあります。

C
コンサルティング会社の所有する新しいKMSキーを使用するという選択肢は、既存のAMIの暗号化キーを別のキーに変更する必要があることを意味します。
しかし、既存のAMIに対して暗号化キーを後から変更することはできません。
正しい方法は、既存のKMSキーへのアクセス権限を対象アカウントに付与することです。

### まとめ
暗号化されたAMIをクロスアカウントで共有するには、LaunchPermissionの変更とKMSキーポリシーの変更の2つが必要。
AMIを公開するのではなく、LaunchPermissionで対象アカウントを限定し、既存のKMSキーへのアクセスを許可する。

</details>

### 問題10

ある企業は、Amazon Route 53パブリックホストゾーンで公開されている単一のAmazon EC2インスタンスにホストされたウェブアプリケーションを運用しています。また同インスタンスはMySQLデータベースも同時に実行しています。ソリューションアーキテクトは、アプリケーションをスケーラブルで高可用性なアーキテクチャに再構成して、MySQLの読み取りレイテンシーも削減することが求められています。

これらの要件を満たすソリューションの組み合わせはどれでしょうか。（2つ選択）

<details>
<summary>選択肢を見る</summary>

A. 複数のアベイラビリティーゾーンにEC2インスタンスを展開するようにALBとAuto Scalingグループを設定する。

B. 別リージョンに2つ目のEC2インスタンスを起動する。Route53のフェイルオーバールーティングポリシーを使用して、両方のリージョン内のEC2インスタンスにルーティングを構成する。

C. データベースをAmazon Aurora MySQLのグローバルデータベースに移行して、クロスリージョンリードレプリカを構成する。

D. データベースをAmazon Aurora MySQLクラスターに移行して、プライマリDBインスタンスとリーダーDBインスタンスを別々のアベイラビリティーゾーンに作成する。

E. データベースをAmazon Aurora MySQLのグローバルデータベースに移行して、マルチAZ配置を有効化する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
A
D

### 解説
この問題では、Webサーバーとデータベースの両方をスケーラブルかつ高可用なアーキテクチャにする必要があります。

**A（ALB + Auto Scaling + マルチAZ）**
- 現在の単一EC2インスタンスという単一障害点を解消する
- ALBで複数のEC2インスタンスに負荷分散する
- Auto Scalingで負荷に応じてインスタンス数を自動調整（スケーラビリティ）
- 複数AZに展開することで、AZ障害時も稼働を継続（高可用性）

**D（Aurora MySQL + マルチAZリーダー）**
- Aurora MySQLクラスターに移行することで、マネージドなデータベースサービスを活用
- プライマリDBインスタンスとリーダーDBインスタンスを別AZに配置（高可用性）
- リーダーDBインスタンスが読み取りクエリを処理するため、読み取りレイテンシーが削減される
- プライマリ障害時にはリーダーが自動昇格してフェイルオーバー

### 他の選択肢が不適切な理由
B
別リージョンにEC2インスタンスを1台追加する構成は、同一リージョン内でのスケーラビリティを提供しません。
Auto Scalingによる自動スケーリングが含まれていないため、「スケーラブル」という要件を十分に満たしません。

C
グローバルデータベースとクロスリージョンリードレプリカは、複数リージョンにまたがる大規模な構成です。
単一リージョン内での高可用性と読み取りレイテンシー削減が要件であるため、オーバーエンジニアリングになります。

E
グローバルデータベースは複数リージョン間のレプリケーション用であり、単一リージョン内の要件には過剰です。
また、グローバルデータベース自体がクロスリージョンの仕組みであるため、マルチAZ配置と組み合わせる意味が薄くなります。

### まとめ
単一EC2インスタンスのWebアプリケーションをスケーラブルかつ高可用にするには、
- Webサーバー側：ALB + Auto Scaling + マルチAZで負荷分散と冗長化
- データベース側：Aurora MySQLクラスターでマルチAZリーダーを活用して高可用性と読み取りレイテンシー削減

この2つの組み合わせが最適。

</details>
