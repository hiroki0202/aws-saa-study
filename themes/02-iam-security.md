# 第2章: IAM・ガバナンス・セキュリティ

## 座学

## AWSアカウントとIAMの全体像

### 導入（なぜ権限管理が重要なのか）

> 「まず最初に、AWSのセキュリティを考えるうえで最も大切な話をします。
>
> 皆さん、会社の社員証を想像してください。入社したばかりの新人が、社長室にも金庫にも自由に入れる——これは問題ですよね。"最小権限の原則"、つまり業務に必要な最小限の権限しか与えないことが、セキュリティの基本です。
>
> AWSでも同じです。アカウントを作った直後に存在するのが**ルートユーザー**——銀行口座を作ったときの暗証番号のような、最強の権限を持つアカウントです。このルートユーザーは、**請求情報の設定や解約など、特定の操作以外では絶対に使わない**のがベストプラクティスです。
>
> 日常の操作は**IAMユーザー**を使う。IAMユーザーには、業務に必要な最小限の権限だけを与える。これがAWSセキュリティの出発点です。」

## AWS Organizations

### 導入（複数アカウントの管理問題）

> 「さて、企業のAWS利用が拡大すると、"1つのアカウントで全部管理する" では限界が来ます。
>
> 開発環境と本番環境が同じアカウントにあったら何が起きるか。開発者が誤って本番のデータベースを削除してしまうかもしれない。コストも混在するので、どちらでいくら使っているかわからない。
>
> だからアカウントを分ける。開発用、ステージング用、本番用……と別々のAWSアカウントを使う。
>
> でも今度は逆の問題が起きます。アカウントが増えれば増えるほど、**管理が分散する**。"全アカウントのMFAは有効になっているか？" "セキュリティポリシーは全アカウントで守られているか？" を人力で確認するのは現実的ではありません。
>
> これを解決するのが**AWS Organizations**です。複数のAWSアカウントをグループ化して、**一元管理する**サービスです。」

### SCP（サービスコントロールポリシー）

> 「Organizationsで最も重要な機能が**SCP（サービスコントロールポリシー）**です。
>
> IAMポリシーと何が違うの？ と思うかもしれません。IAMポリシーは「このユーザーは何ができる/できない」を制御します。でも、**アカウント全体に絶対に守らせたいルール**を設定したいとき、IAMポリシーだけでは不十分です。管理者権限を持つIAMユーザーが、そのルールを無効にできてしまうからです。
>
> SCPは**アカウント全体への上限**を設定します。どんなに権限の強いIAMユーザーでも、SCPで禁止されていることはできません。たとえば「全アカウントで東京リージョン以外は使えない」「特定のサービスは一切使わせない」という制約をSCPで設定すると、IAMで管理者権限を与えられていても、その制約は破れません。
>
> SCPはIAMポリシーと組み合わせて機能します。**両方で許可されている操作だけが実行できる**という理解が重要です。」

## IAM

### ポリシーの種類

> 「IAMの権限管理の核になるのが**ポリシー**です。"誰が" "何を" "できるか" を JSON 形式で記述したルールブックのようなものです。
>
> ポリシーには2つの種類があります。
>
> **管理ポリシー**は、複数のユーザーやグループに使い回せるポリシーです。さらに "AWSが最初から用意してくれている **AWS管理ポリシー**" と "自分で作る **カスタマー管理ポリシー**" の2種類があります。たとえば `AdministratorAccess` や `ReadOnlyAccess` はAWSが用意したポリシーで、すぐに使えます。細かくカスタマイズしたい場合は自分で作る。
>
> **インラインポリシー**は、特定のユーザーやロールに直接埋め込まれるポリシーです。そのユーザー専用のルールを作るときに使いますが、使い回しができないので、基本的には管理ポリシーを使う方が推奨されます。」

### IAMロールとIAMアイデンティティセンター

> 「**IAMロール**は "人" ではなく "役割" に権限を付与する仕組みです。
>
> たとえばEC2インスタンスからS3にアクセスしたい場合、アクセスキーをインスタンスに埋め込むのは危険です。もしそのキーが漏れたら大変。代わりにEC2にIAMロールをアタッチすることで、**キーなしでS3へのアクセスを安全に許可**できます。
>
> 次に**IAMアイデンティティセンター（旧 AWS SSO）**。複数のAWSアカウントへのログインを1か所で管理するシングルサインオン（SSO）サービスです。
>
> 社員が入社したときに、Organizationsで管理している10個のAWSアカウントそれぞれにIAMユーザーを作るのは大変ですよね。IAMアイデンティティセンターを使えば、1つのIDで複数アカウントに権限を付与できます。Active Directory（社内のユーザー管理システム）とも連携できるので、社員の入退社管理がそのままAWSへのアクセス管理に反映できます。」

## 鍵管理

### KMSとCloudHSM

> 「AWSのデータ暗号化に欠かせないのが**鍵管理**です。
>
> 暗号化には "データを暗号化するための鍵" が必要です。でも鍵の管理も大変で、鍵が漏れたら暗号化の意味がない。
>
> AWSでの標準的な鍵管理サービスが**KMS（Key Management Service）**です。暗号化キーの生成・保管・ローテーションをAWSが管理してくれる。ほとんどのAWSサービス（S3、EBS、RDSなど）とネイティブに統合されているので、チェックボックス1つで暗号化を有効にできます。
>
> KMSは**エンベロープ暗号化**という方式を使います。データを暗号化するための「データキー」をKMSが発行し、そのデータキー自体はKMSが管理する「KMSキー」で暗号化する。2重の鍵構造です。これにより、KMSキーを安全に管理するだけで、大量のデータを効率よく保護できます。
>
> **CloudHSM**は、専用のハードウェアセキュリティモジュール（HSM）をVPC内に置いて、鍵を完全に自分たちの管理下に置くサービスです。金融業界などで "鍵はクラウドのどこにも置けない、自分たちのハードウェアで管理する" という規制要件がある場合に使います。コストが大幅に高くなるので、KMSで要件を満たせる場合は使いません。」

## セキュリティサービス

### 脅威検出から保護まで

> 「AWSには多くのセキュリティサービスがあります。似た名前が多くて混乱しやすいので、**何を守るのか**という切り口で整理しましょう。
>
> まず**脅威の検出**。**Amazon GuardDuty**はCloudTrailのAPI呼び出しログやVPCフローログを機械学習で分析して、不審なアクティビティを自動検知します。"深夜に普段アクセスしない国からログインがあった" "EC2から大量のDNSリクエストが出ている（C&Cサーバーとの通信かも）" といった異常を検出します。有効にするだけでいい——これが最大のメリットです。
>
> **Amazon Inspector**はEC2やLambdaの**ソフトウェアの脆弱性**を自動でスキャンします。"このEC2にインストールされているApacheに既知の脆弱性がある" と教えてくれます。
>
> **Amazon Macie**はS3バケットを分析して、**個人情報や機密データ**が含まれていないかをチェックします。クレジットカード番号やマイナンバーが意図せず公開バケットに置かれている——というミスを防ぎます。」

### 攻撃からの保護

> 「次に**攻撃から守る**サービスです。
>
> **AWS Shield**はDDoS攻撃（大量のリクエストを送り付けてシステムをダウンさせる攻撃）から保護します。Shield Standardは全AWSユーザーに無料で適用されています。もっと高度な保護と専任のサポートが必要なら、Shield Advancedがあります（有料）。
>
> **AWS WAF**はWebアプリケーションへの攻撃（SQLインジェクション、XSSなど）を防ぎます。HTTPリクエストの中身を検査して、不正なリクエストをブロックします。CloudFrontやALBと組み合わせて使います。
>
> **AWS Firewall Manager**は、複数のAWSアカウントにWAFやShieldの設定を一括で適用・管理するサービスです。Organizationsと組み合わせて、全アカウントに統一されたセキュリティポリシーを適用できます。」

### 証明書・認証情報の管理

> 「最後に**認証情報・証明書管理**のサービスです。
>
> **ACM（AWS Certificate Manager）**はHTTPS通信に使うSSL/TLS証明書を無料で発行・管理します。通常、SSL証明書は年間数千〜数万円かかりますが、ACMならAWSサービスで使う分は無料。しかも証明書の有効期限の更新も自動でやってくれます。
>
> **AWS Secrets Manager**はデータベースのパスワードやAPIキーなどの認証情報を安全に保管・ローテーションするサービスです。アプリケーションのコードに直接パスワードを書く——これは最悪のアンチパターンです。Secrets Managerに保管しておけば、コードからは安全に取り出せるし、パスワードの定期的な自動変更もできます。
>
> **AWS Security Hub**は、GuardDuty、Inspector、Macieなど複数のセキュリティサービスの検出結果を1か所に集約して表示するダッシュボードです。複数のアカウントのセキュリティ状況を一括して把握できます。」

#### 第2回（2/21）ハンズオン①：IAMポリシー設計と動作検証

##### このハンズオンで学ぶこと
- IAMポリシーのJSON記述と最小権限の原則の実践
- IAMロールをEC2にアタッチしてAWSリソースへアクセスする仕組み
- ロール付与前後でのアクセス権限の変化を体験
- 許可/拒否の動作検証（読み取りのみ許可、書き込みは拒否）

##### リソース状況
- **作成済み（再利用）**: EC2-A
- **新規作成**: handson-iam-test（S3バケット）、S3ReadOnlyCustom（IAMポリシー）、EC2-S3ReadOnly-Role（IAMロール）

##### ステップ1: S3バケットを作成（検証用）

1. コンソール上部の検索バーに「S3」と入力 →「S3」を選択
2. 「バケットを作成」をクリック
3. バケット名: `handson-iam-test`（グローバルで一意にする）
4. リージョン: `ap-northeast-1`
5. 他はデフォルトのまま →「バケットを作成」
6. 作成したバケットを開き、適当なテキストファイルをアップロード

##### ステップ2: IAMポリシーをJSON作成

1. コンソール上部の検索バーに「IAM」と入力 →「IAM」を選択
2. 左メニュー「ポリシー」→「ポリシーの作成」
3. 「JSON」タブをクリック → 以下を入力：
   ```json
   {
     "Version": "2012-10-17",
     "Statement": [
       {
         "Effect": "Allow",
         "Action": [
           "s3:GetObject",
           "s3:ListBucket"
         ],
         "Resource": [
           "arn:aws:s3:::handson-iam-test",
           "arn:aws:s3:::handson-iam-test/*"
         ]
       }
     ]
   }
   ```
4. 「次へ」→ ポリシー名: `S3ReadOnlyCustom` →「ポリシーの作成」

##### ステップ3: IAMロールを作成

1. 左メニュー「ロール」→「ロールを作成」
2. 信頼されたエンティティ:「AWSのサービス」→「EC2」を選択 →「次へ」
3. 許可ポリシー: 検索バーに `S3ReadOnlyCustom` と入力 → チェック →「次へ」
4. ロール名: `EC2-S3ReadOnly-Role` →「ロールを作成」

##### ステップ4: ロールアタッチ前のアクセス確認

> 第1回ハンズオンで作成した `EC2-A` を使用します。

1. EC2コンソールで `EC2-A` のパブリックIPアドレスを確認
2. SSHでログイン（Mac: ターミナル / Windows: PowerShell）:
   ```
   ssh -i ~/.ssh/handson-key.pem ec2-user@<EC2-AのパブリックIP>
   ```
   > ※ Windows PowerShellでも `~` はホームディレクトリとして使えます。
   >
   > ※ SSHが使えない場合は、EC2 Instance Connectを使用してください: EC2コンソールで `EC2-A` を選択 →「接続」→「EC2 Instance Connect」→「接続」
3. S3バケットへのアクセスを試行:
   ```
   aws s3 ls s3://handson-iam-test/
   ```
   → IAMロールが付与されていないため、`Unable to locate credentials` または `AccessDenied` エラーが表示されることを確認

##### ステップ5: IAMロールをEC2にアタッチ

1. EC2コンソール → `EC2-A` を選択
2. 「アクション」→「セキュリティ」→「IAMロールを変更」
3. IAMロール: `EC2-S3ReadOnly-Role` を選択 →「IAMロールの更新」

##### ステップ6: ロールアタッチ後の動作検証

1. `EC2-A` のターミナルに戻る
2. 許可されている操作を確認:
   ```
   aws s3 ls s3://handson-iam-test/
   aws s3 cp s3://handson-iam-test/test.txt ./
   ```
   → ステップ4では失敗したコマンドが、ロールをアタッチしたことで成功するようになったことを確認
3. 許可されていない操作を確認:
   ```
   aws s3 cp ./test.txt s3://handson-iam-test/upload.txt
   ```
   → `AccessDenied` エラーが表示されることを確認（読み取りのみ許可で書き込み権限なし）

##### リソース削除（一部）

> **注意**: S3バケット `handson-iam-test` はハンズオン②で再利用するため、ここでは削除しないでください。

1. EC2コンソール → `EC2-A` を選択 →「アクション」→「セキュリティ」→「IAMロールを変更」→「IAMロール」を空欄にして「IAMロールの更新」（ロールをデタッチ）
2. IAMロール `EC2-S3ReadOnly-Role` を削除
3. IAMポリシー `S3ReadOnlyCustom` を削除

#### 第2回（2/21）ハンズオン②：KMSカスタムキーでS3暗号化

##### このハンズオンで学ぶこと
- KMSカスタマー管理キーの作成と管理
- S3バケットへのSSE-KMS暗号化設定
- KMSキーの無効化/有効化による暗号化の効果の体験
- 保管時の暗号化（Encryption at Rest）の実践的理解

##### リソース状況
- **作成済み（再利用）**: handson-iam-test（S3バケット、ハンズオン①で作成）
- **新規作成**: handson-cmk（KMSキー）

##### ステップ1: KMSカスタマー管理キーを作成

1. コンソール上部の検索バーに「KMS」と入力 →「Key Management Service」を選択
2. 左メニュー「カスタマー管理型のキー」→「キーの作成」
3. キーのタイプ: `対称`、キーの使用法: `暗号化および復号化` →「次へ」
4. エイリアス: `handson-cmk` →「次へ」
5. キー管理者: 自分のSSOロール（`AWSReservedSSO_AWSPowerUserAccess_...`）を選択 →「次へ」
6. キーの使用アクセス許可: 自分のSSOロール（`AWSReservedSSO_AWSPowerUserAccess_...`）を選択 →「次へ」
7. 「完了」をクリック
8. **確認**: キー一覧に `handson-cmk` が表示され、ステータスが「有効」であること

##### ステップ2: ハンズオン①のS3バケットにKMS暗号化を設定

> ハンズオン①で作成した `handson-iam-test` バケットを再利用します。

1. S3コンソール → `handson-iam-test` バケットを選択
2. 「プロパティ」タブ →「デフォルトの暗号化」セクション →「編集」
3. 以下を設定:
   - 暗号化タイプ:「AWS Key Management Service キーによるサーバー側の暗号化（SSE-KMS）」を選択
   - AWS KMS キー:「AWS KMS キーから選択する」→ `handson-cmk` を選択
   - バケットキー: 有効にする
4. 「変更の保存」

##### ステップ3: 暗号化を確認

1. バケットを開き、新しいテキストファイルをアップロード（暗号化設定変更後にアップロードしたファイルにKMS暗号化が適用されます）
2. アップロードしたファイルをクリック →「プロパティ」タブ
3. 「サーバー側の暗号化設定」セクションを確認:
   - **確認**: 暗号化タイプが `SSE-KMS`、KMSキーARNに `handson-cmk` が表示されていること
4. 「ダウンロード」をクリック → ファイルが正常に復号されてダウンロードできることを確認

##### ステップ4: KMSキーを無効化して暗号化の効果を確認

> KMSキーがないと復号できない＝暗号化されていることを体験します。

1. KMSコンソール → `handson-cmk` を選択 →「キーのアクション」→「無効にする」→ 確認して「無効にする」
2. S3コンソール → `handson-iam-test` バケットで先ほどのファイルを「ダウンロード」
   - **確認**: KMSキーが無効なため、ダウンロードが失敗（AccessDenied）することを確認
3. KMSコンソールに戻り、`handson-cmk` を選択 →「キーのアクション」→「有効にする」
4. 再度S3からファイルをダウンロード
   - **確認**: KMSキーを有効に戻すと、正常にダウンロードできることを確認

##### リソース削除

> **注意**: 第1回で作成した `VPC-A`、`VPC-A-Public`、`IGW-A`、`SG-VPC-A`、`EC2-A` は第3回以降のハンズオンでも継続的に再利用します。**削除しないでください。**

1. S3バケット内のオブジェクトをすべて削除 → バケットを削除
2. KMSコンソール → `handson-cmk` を選択 →「キーのアクション」→「キーの削除をスケジュール」→ 待機期間: 7日 →「削除をスケジュール」

> **コスト節約のヒント**: `EC2-A` は次回のハンズオンまで「停止」しておくとコストを抑えられます（「終了」はしないでください）。EC2コンソール → `EC2-A` を選択 →「インスタンスの状態」→「インスタンスを停止」

## 練習問題

### 問題1

ある企業は、AWS上でホストされている一連のアプリケーションの運用を行っています。
この企業では、アプリケーション間の連携を実現するためにAPI Gatewayを導入することに決定しました。
そのため、API Gatewayにおける権限管理を行い、API呼び出し時にAPIコール元に対して適切なアクセス権限を設定する必要があります。

この要件を満たすためのAPI Gatewayの最も適切な設定方法はどれでしょうか。

### 正解
B

### 解説
API Gatewayの認可（誰が呼び出せるか）をAWSネイティブに管理する代表的な方法の1つが、IAM認証（IAM Authorization）です。

- APIの呼び出し元（IAMユーザー、IAMロール）に対してIAMポリシーで execute-api:Invoke を許可する
- 必要に応じて、API Gateway側のリソースポリシーでも呼び出し元を制限できる
- 署名付きリクエスト（SigV4）によって、呼び出し元が認証される

このため「APIコール元に対して適切なアクセス権限を設定する」要件に対して、IAMポリシーで制御するのが最も適切です。

### 他の選択肢が不適切な理由
A
STSは一時クレデンシャルを発行する仕組みであり、権限そのものの定義はIAMポリシーで行います。
STS単体を「権限付与の方法」とするのは不正確です。

C
AWS Configはリソース設定の変更履歴や評価を行うサービスであり、API呼び出し元の認可には使いません。

D
IAMアクセスキーは認証情報の一種であり、権限管理の仕組みそのものではありません。
また長期的なアクセスキーの配布は漏洩リスクがあり、ベストプラクティスではありません。

### まとめ
API GatewayでAPIコール元のアクセス権限を管理するには、
IAMポリシーで execute-api:Invoke を制御する方法が最も適切。

        {
          "Action": "ec2:\*",

C

### 解説

- Allowがあっても、Denyに該当すれば最終的に拒否される

- ec2:TerminateInstances

- リザーブドインスタンス関連操作は実行不可

TerminateInstances そのものが拒否されているため、通常のEC2インスタンスも終了できません。

このポリシーでは、EC2全操作は原則許可されるが、

D

<summary>正解と解説を見る</summary>

### 正解
A

### 解説
このポリシーのポイントは以下です。

- Effect が Deny
- Action は "*"
- Resource は "*"
- 条件が NotIpAddress で aws:SourceIp が 172.101.1.36/32

意味を整理すると、

「送信元IPアドレスが 172.101.1.36 ではない場合、すべてのアクションを拒否する」

という内容になります。

IAMの評価ルールでは、

- 明示的な Deny は常に優先される
- Allow があっても Deny に該当すれば拒否される

そのため、このポリシーは実質的に

- 172.101.1.36 からのアクセスのみ許可対象になり得る

C
条件は NotIpAddress なので、172.101.1.36 以外は拒否されます。
「以外が許可」ではありません。

D
指定されているIPは 172.101.1.36 であり、172.101.1.6 ではありません。

### まとめ
このポリシーは、172.101.1.36 以外のIPアドレスからのすべてのAWS操作を明示的に拒否する設定。
IPベースでアクセス元を厳しく制限する典型的なDenyポリシー例です。

Dの構成は以下の点で最も安全です。

- WEBサーバーもDBサーバーもプライベートサブネットへ移す
  - どちらもパブリックIPを持たず、インターネットから直接到達できない
- 管理アクセスは踏み台サーバー経由に限定できる
  - 直接SSH/RDPを内部サーバーへ通さない
- 社内ネットワークからはSite-to-Site VPNでVPCへ接続する
  - インターネット公開を避けて社内経路に閉じる

この結果、外部公開範囲を最小化しつつ、社内ユーザーのアクセス要件も満たせます。

### 他の選択肢が不適切な理由
A
NATインスタンスは主にアウトバウンド通信のための仕組みです。
サーバー配置（パブリック/プライベート）や、社内ユーザーの安全なアクセス経路の設計としては主旨がずれます。
またNATインスタンスは運用負荷も増えます。

B
WEBサーバーだけをプライベートへ移しても、DBサーバーがパブリックに残るため、重要データへのリスクが残ります。
「最もセキュリティが高い」にはなりません。

C
DBをプライベートへ移すのは重要な改善ですが、WEBサーバーがパブリックのままだと攻撃面が残ります。
要件が「最も高いセキュリティ」であれば、WEBも含めてプライベートへ寄せ、アクセスを最小化するのがより強固です。

### まとめ
最も安全な構成は、WEB/DBの両方をプライベートサブネットへ隔離し、
社内からはVPNで接続、管理は踏み台経由に限定してアクセス経路を制御すること。
この方針がアタックサーフェスを最小化し、セキュリティを最大化できる。

### 問題9

製造業のA社はAWS上にエンタープライズシステムをホストしています。最近になって、そのシステムが突如停止するという障害が発生しました。ソリューションアーキテクトが調査したところ、一人のエンジニアが本番環境のEC2インスタンスを誤って終了してしまい、サービス中断を引き起こしたことが判明しました。また、実稼働するアプリケーション向けのインフラ構成を操作できる開発者が多数存在するという事実も同時に判明しました。

この種の障害が再び発生するのを防ぐための、適切な対応はどれでしょうか？（2つ選択してください。）

<details>
<summary>選択肢を見る</summary>

A. すべてのEC2インスタンスに適切なタグ設定を行い、タグ情報に基づいて本番環境向けインスタンスの削除を拒否する権限を設定する。それによって、開発者の権限を所属するグループタグ内でのリソース操作に限定する。

B. 開発者向けのIAMポリシーを修正して、EC2インスタンスの停止権限の許可ポリシーを削除する。

C. 開発者グループのIAMグループのIAMポリシーを修正して、EC2インスタンスへの操作権限を削除する。

D. AWS Organizationsを利用して開発者グループとそれ以外を分割する組織ルールを適用して、開発者がアクセスできるインスタンスを制限する。

E. 開発環境向けのVPCを新たに設置して、開発者のアクセス権限を開発者向けVPC内に制限する。その上で、IAMポリシーによってVPCごとに権限設定を割り振ることで、開発者グループが利用できるVPC内リソースを制限する。

</details>

<summary>正解と解説を見る</summary>

### 正解
A
E

### 解説
この問題のポイントは、開発者の業務に必要な権限は維持しつつ、本番環境を保護する方法を選ぶことです。

A（タグベースのアクセス制御）
- EC2インスタンスに環境タグ（例: Environment=Production, Environment=Development）を設定する
- IAMポリシーのConditionでタグに基づいたアクセス制御を行い、本番環境タグのインスタンスに対する削除・終了操作を拒否する
- 開発者は開発環境のインスタンスは自由に操作でき、本番環境のインスタンスは保護される

E（VPCによる環境分離）
- 本番環境と開発環境をVPCレベルで分離する
- IAMポリシーでVPCごとにアクセス権限を制御し、開発者は開発用VPC内のリソースのみ操作可能にする
- ネットワークレベルでの分離により、誤操作のリスクをさらに低減できる

この2つを組み合わせることで、タグによる論理的な制御とVPCによるネットワーク的な分離の両面から本番環境を保護できます。

### 他の選択肢が不適切な理由
B
EC2インスタンスの停止権限を完全に削除すると、開発環境のインスタンスも停止できなくなります。
開発業務に必要な権限まで奪ってしまうため、適切な対応とは言えません。

C
EC2インスタンスへの操作権限をすべて削除すると、開発者は開発業務自体ができなくなります。
問題の本質は「本番環境へのアクセスを制限すること」であり、すべての権限を削除することではありません。

D
AWS Organizationsはアカウント単位の管理サービスであり、同一アカウント内の開発者グループを分割する仕組みではありません。
アカウント内のユーザー権限管理にはIAMを使用するのが適切です。

### まとめ
本番環境の誤操作を防ぐには、権限を全削除するのではなく、タグベースのアクセス制御とVPCによる環境分離を組み合わせて、開発者の業務に必要な権限を維持しつつ本番環境を保護するのが適切。

### 問題10

A社では社内アプリケーションに対するDDoS攻撃によって大規模なシステム障害が発生しました。そこで、あなたはソリューションアーキテクトとして、AWS上にホストしている自社アプリケーションを保護するためにDDoS攻撃などの外部攻撃を軽減するアーキテクチャを設定するように依頼されました。具体的に防止するべき攻撃リストは以下の通りです。

- DDoS攻撃
- SYNフラッド
- UDPリフレクション攻撃
- SQLインジェクション
- クロスサイトスクリプティング

これらの要件を満たす、AWSアーキテクチャ設計パターンを選択してください。（3つ選択してください。）

<details>
<summary>選択肢を見る</summary>

A. AWS Shield Standardを利用して高度なDDoS攻撃及びSYNフラッドやUDPリフレクション攻撃を検出して、アプリケーションレイヤー上のリアルタイム通知を実施する。

B. Amazon Route 53 のシャッフルシャーディングと anycast ルーティング機能により、DDoS 攻撃を回避して、エンドユーザーがアプリケーションにアクセスできるようにする。

C. AWS Shield Standardを利用して、SQLインジェクション、クロスサイトスクリプティング攻撃などを検出して、アプリケーションレイヤー上のリアルタイム通知を実施する。

D. AWS Shield Advancedを利用して高度なDDoS攻撃を検出して、アプリケーションレイヤー上のモニタリングを実施する。

E. AWS WAFを利用してセキュリティルールをカスタマイズして、アプリケーションに対するトラフィックを制御する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
B
D
E

### 解説
この問題では、ネットワーク層の攻撃（DDoS、SYNフラッド、UDPリフレクション）とアプリケーション層の攻撃（SQLインジェクション、XSS）の両方に対処する必要があります。それぞれのサービスの役割を理解することがポイントです。

B（Route 53のシャッフルシャーディングとanycastルーティング）
- シャッフルシャーディングにより、DDoS攻撃の影響範囲を限定できる
- anycastルーティングにより、トラフィックを最寄りのロケーションに分散し、攻撃の影響を軽減する
- これによりDDoS攻撃中でもエンドユーザーのアクセスを維持できる

D（AWS Shield Advanced）
- 高度なDDoS攻撃、SYNフラッド、UDPリフレクション攻撃を検出・防御する
- アプリケーションレイヤーのモニタリングとリアルタイムの可視化を提供する
- DDoS Response Team（DRT）による支援が受けられる

E（AWS WAF）
- SQLインジェクションやクロスサイトスクリプティングなど、アプリケーション層の攻撃を防御する
- カスタムルールにより、アプリケーション固有の脅威に対応できる
- Shieldがネットワーク層を守り、WAFがアプリケーション層を守るという役割分担

### 他の選択肢が不適切な理由
A
Shield Standardは基本的なDDoS防御を自動で提供しますが、高度な攻撃の検出やリアルタイム通知機能はありません。
高度なDDoS攻撃への対応やモニタリングにはShield Advancedが必要です。

C
Shield（StandardもAdvancedも）はネットワーク層・トランスポート層のDDoS防御サービスです。
SQLインジェクションやクロスサイトスクリプティングはアプリケーション層の攻撃であり、これらの検出・防御はAWS WAFの役割です。

### まとめ
外部攻撃からアプリケーションを総合的に保護するには、各サービスの守備範囲を理解して組み合わせることが重要。
- Route 53：DDoS攻撃時のDNS可用性確保
- Shield Advanced：ネットワーク層のDDoS防御とモニタリング
- WAF：アプリケーション層の攻撃（SQLi、XSS等）の防御
