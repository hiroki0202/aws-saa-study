# 第4章: ストレージサービス

## 座学

## ストレージの種類と選び方

### 導入（3種類のストレージ）

> 「データをどこに保存するか——これはシステム設計の基本です。AWSのストレージサービスを選ぶには、まず**ストレージの3つの種類**を理解することが重要です。
>
> **ブロックストレージ**は、データをブロック（細かい単位）に分割して保存します。HDDやSSDのような感覚で使えるストレージで、OSからは「ディスク」として見えます。EC2のディスクや、データベースの保存領域に使います。
>
> **ファイルストレージ**は、ファイルとフォルダの階層構造でデータを管理します。私たちが普段Windowsや Macで使っているファイルシステムと同じ感覚です。複数のサーバーから同時にアクセスできるのが特徴です。
>
> **オブジェクトストレージ**は、ファイルを「オブジェクト」という単位で管理します。フォルダの概念はなく、すべてフラットに保存されます。URLでアクセスできて、無制限に保存でき、非常に耐久性が高い。大量の画像や動画、ログファイルの保存に向いています。
>
> AWSでは：EBS → ブロック、EFS/FSx → ファイル、S3 → オブジェクト、という対応関係になります。」

---

## EBS

### EC2のディスク

> 「**EBS（Elastic Block Store）**はEC2インスタンスにアタッチして使うブロックストレージです。EC2のシステムドライブ（Cドライブのような存在）や追加ディスクとして使います。
>
> EBSの重要なポイントが2つあります。
>
> 1つ目は、**EC2インスタンスとは独立して存在する**こと。EC2を停止・削除してもEBSボリュームは消えません。別のEC2インスタンスにアタッチして使い続けられます。大切なデータはEBSに保存しておけば安心です。
>
> 2つ目は、**同じAZ内でしか使えない**こと。EC2がap-northeast-1aにあれば、EBSもap-northeast-1aに作る必要があります。別のAZには直接アタッチできません。
>
> EBSには複数の種類があります。**gp3**（汎用SSD）は最もよく使われる標準的なタイプ。**io2**は高IOPSが必要なデータベース向けの高性能タイプ。**st1**は大量データをシーケンシャルに読み書きするHDDタイプでコストが安い——といった使い分けです。」

## EFS

### 複数のEC2から同時アクセス

> 「**EFS（Elastic File System）**は、複数のEC2インスタンスから**同時にアクセスできる**ファイルストレージです。
>
> EBSとの違いを考えてみましょう。EBSは1つのEC2インスタンスにしかアタッチできません（マルチアタッチという機能もありますが限定的です）。でもWebサーバーが10台並んでいて、全サーバーで同じファイルを共有したい——こういう場面はよくありますよね。ユーザーがアップロードした画像ファイルを、どのWebサーバーからでも読めるようにしたい。
>
> EFSはこれを実現します。複数のEC2から同時に読み書きできる。しかも容量は使った分だけ自動拡張されるので、あらかじめサイズを決める必要がありません。
>
> EFSを使うときに注意が必要なのが**パフォーマンスモード**と**スループットモード**の選択です。パフォーマンスモードは作成後に変更できないので、事前によく検討する必要があります。選択の参考になるのがCloudWatchの`PercentIOLimit`メトリクス——これが高い値になっているなら、高IOPSに対応したMaxIOモードへの変更を検討します。」

## S3

### オブジェクトストレージの王様

> 「**S3（Simple Storage Service）**はAWSで最もよく使われるサービスの1つです。11 nines（99.999999999%）という驚異的な耐久性を持ち、容量は実質無制限。Webサイトの画像、バックアップデータ、ログファイル、静的Webサイトのホスティング……様々な用途に使われます。
>
> S3の基本構造は、**バケット**（フォルダのようなもの）の中に**オブジェクト**（ファイル）を保存するシンプルな仕組みです。バケット名はグローバルで一意（世界中で重複不可）である必要があります。
>
> **アクセス管理**は3層構造になっています。
>
> **IAMポリシー**は "このIAMユーザーはこのバケットにアクセスできる" という、ユーザー側からの制御。
>
> **バケットポリシー**は "このバケットには誰がアクセスできる" という、バケット側からの制御。CloudFrontからのアクセスだけ許可する、特定のIPアドレスからのみ許可する——といった設定ができます。
>
> **署名付きURL**は、特定のオブジェクトへの一時的なアクセスを許可するURL。期限付きで、そのURLを知っている人ならAWSアカウントを持っていなくてもアクセスできます。有料コンテンツのダウンロードリンクや、プライベートな画像の一時共有に使います。有効期限が切れると自動的にアクセスできなくなります。」

### S3のストレージクラス

> 「S3には複数の**ストレージクラス**があり、データのアクセス頻度に合わせて使い分けることでコストを最適化できます。
>
> **S3 Standard**：最も標準的なクラス。頻繁にアクセスするデータ向け。Webサイトのコンテンツや、よく参照するデータに使います。
>
> **S3 Standard-IA（Infrequent Access）**：アクセス頻度が低いデータ向け。保存コストは安いですが、取り出すときに取り出し料金がかかります。月に数回しか参照しないバックアップデータなどに。
>
> **S3 Glacier**：アーカイブ用。取り出しに数分〜数時間かかりますが、保存コストが非常に安い。法規制で長期保存が義務づけられているデータ、数年に一度しか参照しないデータ向けです。
>
> **S3 Intelligent-Tiering**：アクセスパターンを機械学習で分析して、自動的に最適なクラスに移動してくれます。アクセスパターンが予測できないデータに便利です。」

## Storage Gateway

### オンプレミスとクラウドをつなぐ

> 「**Storage Gateway**は、オンプレミスにあるシステムとAWSのストレージ（S3、S3 Glacier、EBSなど）を繋ぐ"橋渡し"サービスです。
>
> たとえばこんな場面を想像してください。社内の古いシステムがNAS（ネットワーク接続ストレージ）にデータを保存しています。このシステムをすぐには移行できないけれど、データはクラウドに上げてバックアップしたい。
>
> Storage Gatewayを使えば、オンプレミスのシステムからは「ファイルサーバーに保存している」という感覚のままで、実際のデータはS3に保存されます。既存のシステムを一切変更せずに、バックアップ先をクラウドに移せる。
>
> Storage Gatewayには3種類あります。**ファイルゲートウェイ**（NFS/SMBプロトコルでS3に保存）、**ボリュームゲートウェイ**（ISCSIプロトコルでブロックストレージとして使用）、**テープゲートウェイ**（既存の仮想テープライブラリをS3/Glacierに移行）。試験では「オンプレミスのNASをS3に移行したい、でもアプリは変えたくない」→ **ファイルゲートウェイ**という判断が重要です。」

#### 第4回（3/14）ハンズオン①：EBSボリューム操作とスナップショット復元

> **このハンズオンで学ぶこと**
> - EBSボリュームの作成・アタッチ・マウントによるデータ書き込み
> - スナップショットによるバックアップ→リカバリの一連の流れ

> **実務でのユースケース**
> - **ディスク容量の拡張**: ログやデータの増加でルートボリュームだけでは足りなくなった場合に追加する
> - **OSとデータの分離**: データ用ボリュームを分けておけば、OSに問題が起きて再構築する際もデータ用ボリュームを新しいEC2に付け替えるだけで済む
> - **用途別の性能最適化**: DB用には高IOPSのボリューム、ログ用には安価なボリュームなど、用途に応じてボリュームタイプを使い分ける

> **リソース状況**
> - **作成済み（第1回〜第3回）**: `VPC-A`、`VPC-A-Public`（ap-northeast-1a）、`VPC-A-Public-2`（ap-northeast-1c）、`IGW-A`、ルートテーブル、`SG-VPC-A`、`EC2-A`（停止中）、キーペア
> - **新規作成**: EBSボリューム、スナップショット、復元ボリューム

##### ステップ1: EC2-Aを起動

> 第1回ハンズオンで作成した `EC2-A` を再利用します。

1. EC2コンソール → `EC2-A` を選択
2. 「インスタンスの状態」→「インスタンスを開始」
3. ステータスチェックが「2/2 のチェックに合格しました」になるまで待つ

##### ステップ2: EBSボリュームを作成・アタッチ

1. EC2コンソール → 左メニュー「ボリューム」→「ボリュームを作成」
2. 以下を設定:
   - ボリュームタイプ: `gp3`
   - サイズ: `8` GiB
   - アベイラビリティゾーン: `ap-northeast-1a`（`EC2-A` と同じAZ。**異なるAZのボリュームはアタッチできません**）
3. 「ボリュームを作成」
4. 作成したボリュームを選択 →「アクション」→「ボリュームのアタッチ」
   - インスタンス: `EC2-A` を選択
   - デバイス名: `/dev/sdf`（デフォルト）
5. 「ボリュームをアタッチ」

> **補足: アタッチとマウントの違い**
> - **アタッチ** → EC2にディスクを接続すること（PCにUSBメモリを挿すイメージ）
> - **マウント** → 接続したディスクをOSのフォルダに紐づけること（USBメモリをエクスプローラーで開けるようにするイメージ）
>
> アタッチしただけではOS上でファイルの読み書きはできません。次のステップ3でマウントして初めて `/mnt/ebs` のようなパスでアクセスできるようになります。なお、ルートボリューム（OS用ディスク）はEC2起動時にアタッチもマウントも自動で行われるため普段は意識する必要はありません。

> **補足: デバイス名について**
> アタッチ時に指定する `/dev/sdf` はEC2にボリュームを認識させるための名前です。EC2内部では `xvd○` という形式に自動変換されるため、`/dev/sdf` → `/dev/xvdf` としてOS上に表示されます。末尾のアルファベットが対応しています（`sdf` → `xvdf`、`sdg` → `xvdg`）。なお `xvda` はOS用のルートボリュームとして最初から使われています。

##### ステップ3: EC2でEBSをマウント・データ書き込み

1. `EC2-A` にログイン（SSHまたはEC2 Instance Connect）:
   ```
   ssh -i ~/.ssh/handson.pem ec2-user@<EC2-AのパブリックIP>
   ```
2. 以下を実行:
   ```bash
   # 接続されているディスクの一覧を表示（アタッチしたxvdfが見えることを確認）
   lsblk
   ```
   以下のように表示されればOKです:
   ```
   NAME      MAJ:MIN RM SIZE RO TYPE MOUNTPOINTS
   xvda      202:0    0   8G  0 disk            ← OS用ルートボリューム
   ├─xvda1   202:1    0   8G  0 part /            （パーティション。/ にマウント済み）
   ├─xvda127 259:0    0   1M  0 part
   └─xvda128 259:1    0  10M  0 part /boot/efi
   xvdf      202:80   0   8G  0 disk            ← 今アタッチしたボリューム（MOUNTPOINTSが空＝まだマウントされていない）
   ```
   > **ポイント**: `xvdf` が `8G` の `disk` として表示され、`MOUNTPOINTS` が空であることを確認してください。これはディスクは認識されているが、まだOS上のフォルダに紐づいていない状態です。
   ```bash
   # 新しいディスクにXFSファイルシステムを作成（初回のみ必要。USBメモリの「フォーマット」と同じ）
   sudo mkfs -t xfs /dev/xvdf
   ```
   大量の出力が表示されますが、エラーが出なければ成功です。`bsize=4096` はブロックサイズ（ディスクの読み書きの最小単位）が4KBであることを示しています。その他の項目は内部的な設定なので気にしなくてOKです。
   ```bash
   # マウント用のディレクトリを作成し、ディスクをそこに紐づける（これでファイルの読み書きが可能になる）
   sudo mkdir -p /mnt/ebs
   sudo mount /dev/xvdf /mnt/ebs

   # テストデータを書き込み、内容を確認
   sudo sh -c 'echo "Hello from EBS!" > /mnt/ebs/test.txt'
   cat /mnt/ebs/test.txt

   # メモリ上のデータをディスクに確実に書き出す（次のステップでスナップショットを取るため）
   sudo sync
   ```
3. **確認**: 「Hello from EBS!」と表示される

##### ステップ4: スナップショットを取得

1. EC2コンソール → 左メニュー「ボリューム」→ アタッチしたボリュームを選択
2. 「アクション」→「スナップショットの作成」
3. 説明: `handson-snapshot` →「スナップショットの作成」
4. 左メニュー「スナップショット」→ ステータスが `completed` になることを確認

##### ステップ5: スナップショットからボリュームを復元

> スナップショットからEBSボリュームを復元し、データが残っていることを確認します。

1. EC2コンソール → 左メニュー「スナップショット」→ `handson-snapshot` を選択
2. 「アクション」→「スナップショットからボリュームを作成」
3. 以下を設定:
   - ボリュームタイプ: `gp3`
   - サイズ: `8` GiB
   - アベイラビリティゾーン: `ap-northeast-1a`（`EC2-A` と同じAZ）
4. 「ボリュームを作成」
5. 作成したボリュームを選択 →「アクション」→「ボリュームのアタッチ」
   - インスタンス: `EC2-A` を選択
   - デバイス名: `/dev/sdg`
6. 「ボリュームをアタッチ」
7. `EC2-A` のターミナルで以下を実行:
   ```bash
   # ディスク一覧を表示（復元したxvdgが追加されていることを確認）
   lsblk

   # マウント用ディレクトリを作成し、復元ボリュームを紐づける
   # スナップショットから復元したのでフォーマット不要（データがそのまま残っている）
   # -o nouuid: 元のボリュームとディスクの識別番号（UUID）が同じため、重複を許可するオプションが必要
   sudo mkdir -p /mnt/ebs-restored
   sudo mount -o nouuid /dev/xvdg /mnt/ebs-restored

   # スナップショット時点のデータが復元されていることを確認
   cat /mnt/ebs-restored/test.txt
   ```
8. **確認**: 「Hello from EBS!」と表示される（スナップショットからデータが復元された）

##### リソース削除

1. EBSボリューム2つ（元のボリュームとスナップショットから復元したボリューム）をそれぞれデタッチ → 削除
2. スナップショット `handson-snapshot` を削除

#### 第4回（3/14）ハンズオン②：EFS共有マウント

> **このハンズオンで学ぶこと**
> - EBSは1台のEC2にしかアタッチできないが、EFSは複数のEC2から同時にアクセスできる
> - 実際に2台のEC2から同じファイルを読み書きして、共有ファイルシステムの動作を確認する

> **実務でのユースケース**
> EFSはファイルシステムとしてマウントできる（＝普通のファイル操作で読み書きできる）のが特徴です。S3はAPI経由でのアクセスが必要なため、以下のようなケースではEFSが選ばれます。
> - **レガシーアプリのクラウド移行**: オンプレで動いていたアプリがローカルファイルパス（例: `/data/shared/`）への読み書きを前提に作られている場合、S3対応にコードを書き換えずにEFSをマウントすればそのまま動かせる
> - **Lambda/Fargateで大きなファイルを扱う**: Lambdaの一時領域（`/tmp`）は最大10GBまでだが、EFSをマウントすればMLモデル（数十GB）などの大きなファイルも扱える
>
> ただし、新規にシステムを構築する場合はS3の方が安価でスケーラブルなため、S3で代替することが多いです。EFSは「ファイルシステムとしてのマウントが必要な場面」で使うサービスと覚えておくとよいでしょう。

> **リソース状況**
> - **作成済み（第1回〜第3回）**: `VPC-A`、`VPC-A-Public`（ap-northeast-1a）、`VPC-A-Public-2`（ap-northeast-1c）、`IGW-A`、ルートテーブル、`SG-VPC-A`、`EC2-A`（起動中）、キーペア
> - **新規作成**: `SG-EFS`（セキュリティグループ）、EFSファイルシステム、`EC2-EFS`（EC2インスタンス）

##### ステップ1: VPCのDNSホスト名を有効化

> EFSをマウントする際にDNS名を使用するため、VPCの「DNSホスト名」を有効にする必要があります。

1. VPCコンソール → 左メニュー「VPC」→ `VPC-A` を選択
2. 「アクション」→「VPCの設定を編集」
3. 「DNSホスト名を有効化」にチェックを入れる
4. 「保存」をクリック

##### ステップ2: EFS用のセキュリティグループを作成（新規作成）

> EFSにはNFS（ポート2049）でアクセスします。`SG-VPC-A` からのNFS通信を許可するセキュリティグループを作成します。

1. VPCコンソール →  左メニュー「セキュリティグループ」→「セキュリティグループを作成」
2. 以下を設定:
   - 名前: `SG-EFS`
   - 説明: `EFS security group`
   - VPC: `VPC-A`（**作成済み**）
   - インバウンドルール: タイプ `NFS`、ソース → `SG-VPC-A` のセキュリティグループIDを指定
     > 検索欄に `SG-VPC-A` と入力すると候補が表示されるので選択してください。これにより「`SG-VPC-A` が割り当てられたリソース（＝EC2-AやEC2-EFS）からのNFS通信のみ許可」という意味になります。
3. 「セキュリティグループを作成」

##### ステップ3: EFSを作成（新規作成）

1. コンソール上部の検索バーに「EFS」と入力 →「EFS」を選択
2. 「ファイルシステムを作成」をクリック
   - 名前: `handson-efs`
   - VPC: `VPC-A`（**作成済み**）
3. 「作成」をクリック
4. 作成したファイルシステムを選択 →「ネットワーク」タブ →「管理」をクリック
5. 各AZのマウントターゲットのセキュリティグループを `SG-EFS`（**ステップ2で新規作成**）に変更（デフォルトのSGは外す）
6. 「保存」をクリック

##### ステップ4: 2台目のEC2を起動（新規作成）

> EFSの共有マウントを確認するために、2台目のEC2を起動します。

1. EC2コンソール →「インスタンスを起動」
2. 以下を設定:
   - 名前: `EC2-EFS`
   - AMI: Amazon Linux 2023
   - インスタンスタイプ: `t2.micro`
   - キーペア: 第1回で作成したものを選択（**作成済み**）
   - ネットワーク設定 →「編集」:
     - VPC: `VPC-A`（**作成済み**）
     - サブネット: `VPC-A-Public`（**作成済み**）
     - パブリックIPの自動割り当て: `有効化`
     - セキュリティグループ: `SG-VPC-A`（**作成済み**）
3. 「インスタンスを起動」

##### ステップ5: 2台のEC2からEFSをマウントして共有を確認

> `EC2-A` でファイルを書き込み、`EC2-EFS` からそのファイルが見えることを確認します。

1. `EC2-A` にログイン（SSHまたはEC2 Instance Connect）:
   ```
   ssh -i ~/.ssh/handson.pem ec2-user@<EC2-AのパブリックIP>
   ```

   ```bash

   # amazon-efs-utils: EFSのファイルシステムIDを指定するだけでマウントできるようにするAWS公式ツール

   sudo mkdir -p /mnt/efs
   ```

#### 第4回（3/14）ハンズオン③：S3バージョニング・ライフサイクル・署名付きURL

> **このハンズオンで学ぶこと**
> - **バージョニング**: 同名ファイルの上書きでも過去のバージョンを保持・復元できること
> - **ライフサイクルポリシー**: ストレージクラスの自動移行によるコスト最適化の設定方法
> - **署名付きURL**: 非公開オブジェクトへの一時的なアクセス許可の仕組み

> **リソース状況**
> - **作成済み**: なし（すべて新規作成）
> - **新規作成**: S3バケット

##### ステップ1: S3バケットを作成してバージョニングを有効化

1. S3コンソール →「バケットを作成」
2. 以下を設定（記載のない項目はデフォルトのままでOK）:
   - **バケット名**: `handson-s3-version-<自分の名前>`（グローバルで一意にする）
   - オブジェクト所有者: デフォルト（ACL無効）のまま
   - **「パブリックアクセスをすべてブロック」: オンのまま**（デフォルト）
   - **「バケットのバージョニング」: `有効にする`に変更**
   - デフォルトの暗号化: デフォルトのまま
3. 「バケットを作成」

##### ステップ2: バージョニングを体験

1. テキストファイル `hello.txt`（内容: `Version 1`）をアップロード
2. 同じ名前のファイル `hello.txt`（内容: `Version 2`）を再度アップロード
3. もう一度 `hello.txt`（内容: `Version 3`）をアップロード
4. バケット内のオブジェクト一覧で「バージョンの表示」を切り替え
5. **確認**: 3つのバージョンが表示される
6. 古いバージョン（Version 1）を選択 →「ダウンロード」→ 内容が `Version 1` であることを確認

##### ステップ3: 署名付きURLを生成

> ステップ4でライフサイクルポリシーを設定している間に有効期限の3分が経過するので、先に署名付きURLを生成しておきます。

1. バケット内のオブジェクト `hello.txt` を選択
2. 「アクション」→「署名付き URL で共有」
3. 有効期限: `3`（分）
4. 「署名付き URL を作成」→ 表示されたURLをコピー
5. ブラウザの**シークレットウィンドウ**でURLを開く（通常のウィンドウだとAWSコンソールのログインセッションが残っているため、署名付きURLではなくログイン権限でアクセスできてしまう。シークレットウィンドウなら未認証の状態で確認できる）
6. **確認**: ファイルの内容が表示される（認証なしでアクセス可能）

##### ステップ4: ライフサイクルポリシーを設定

1. バケットを選択 →「管理」タブ →「ライフサイクルルールを作成」
2. ルール名: `archive-rule`
3. 適用範囲: 「バケット内のすべてのオブジェクトに適用」
4. ライフサイクルルールのアクション:
   - 「現行バージョンのオブジェクトをストレージクラス間で移行する」にチェック
   - 30日後 → `Standard-IA` に移行
   - 90日後 → `Glacier Flexible Retrieval` に移行
5. 「ルールを作成」
6. **確認**: ライフサイクルルール一覧に `archive-rule` が表示される

##### ステップ5: 署名付きURLの有効期限切れを確認

> ステップ3で生成した署名付きURLの有効期限（3分）が切れているはずです。

1. ステップ3で開いたシークレットウィンドウでURLを再度開く（またはページを更新）
2. **確認**: アクセスが拒否されること（有効期限が切れたため）

##### リソース削除

1. バケット内の全オブジェクト（全バージョン含む）を削除
2. バケットを削除

> **コスト節約のヒント**: `EC2-A` は次回のハンズオンまで「停止」しておくとコストを抑えられます（「終了」はしないでください）。EC2コンソール → `EC2-A` を選択 →「インスタンスの状態」→「インスタンスを停止」

## 練習問題

### 問題1

ある企業が一般公開向けの画像データを提供するアプリケーションを開発しています。このアプリケーションはAmazon EC2インスタンス上で運用されており、画像コンテンツはAmazon S3バケットの標準ストレージクラスに保存されています。S3バケットにはパブリックアクセスブロックを有効にしてバケットポリシーによる制御が設定されています。アプリケーションは、AWS外部のユーザーのリクエストに応じて特定の画像を一時的に表示する必要があり、その際には特定のユーザーに対してのみ一時的な画像の利用許可を設定することが求められます。

この要件に応じて、どのようなソリューションを導入する必要がありますか。

<details>
<summary>選択肢を見る</summary>

A. Amazon S3の期限付きの事前署名付きURLを利用して、画像コンテンツをユーザーに配信する。

B. CloudFrontディストリビューションによって画像配信を実施することで、画像コンテンツを利用できるユーザー範囲を限定する。

C. Amazon S3にAWS KMSの暗号化キーを適用して期限付きで、画像コンテンツを暗号化する。ユーザーは復号キーを利用して、画像にアクセスする。

D. 画像保存先をEFSによるファイル共有に切り替えることで、期限付きで画像コンテンツを共有する仕組みを構成する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
A

### 解説
S3の署名付きURL（Pre-signed URL）は、非公開のS3オブジェクトに対して一時的なアクセス権を付与する仕組みです。

- 署名付きURLには有効期限が設定され、期限が切れるとアクセスできなくなる
- URLを生成したユーザーの権限でオブジェクトにアクセスするため、バケット自体を公開する必要がない
- パブリックアクセスブロックが有効なままでも、署名付きURLによるアクセスは可能
- 特定のユーザーにURLを共有することで、アクセス範囲を制御できる

### 他の選択肢が不適切な理由
B
CloudFrontはコンテンツ配信を高速化するCDNサービスですが、「特定のユーザーに対してのみ一時的にアクセスを許可する」という細かなアクセス制御には適していません。CloudFrontの署名付きURLやCookieを使えば可能ですが、選択肢の記述はディストリビューション自体でユーザー範囲を限定するとしており、不正確です。

C
KMSの暗号化キーに有効期限を設定してオブジェクトを一時的にアクセス可能にする仕組みは存在しません。KMS暗号化はデータの保護が目的であり、一時的なアクセス許可の手段ではありません。

D
EFSはEC2インスタンス間でファイルを共有するためのファイルストレージであり、AWS外部のユーザーにHTTP経由で一時的にファイルを提供する機能はありません。

### まとめ
S3の署名付きURLは、非公開オブジェクトへの一時的なアクセスを付与する最もシンプルかつ安全な方法。バケットのパブリックアクセスブロックを維持したまま、期限付きでアクセスを許可できる。

</details>

B. S3 Standardにデータを蓄積して、4カ月後にGlacier Deep Archiveに移行するライフサイクルルールを設定する。

C. S3 Standardにデータを蓄積して、4カ月後にGlacier Instant Retrievalに移行するライフサイクルルールを設定する。

<summary>正解と解説を見る</summary>

### 正解

### 解説
この要件のポイントは2つあります。

2. **4か月後にはほとんどアクセスされなくなり、アーカイブが必要** → Glacier Deep Archiveが最もコスト効率が高い

- Glacier Deep Archiveは最も低コストのアーカイブストレージで、ほとんどアクセスされないデータの長期保存に最適

C
初期ストレージ（S3 Standard）は適切ですが、移行先にGlacier Instant Retrievalを選ぶのはコスト最適ではありません。「ほとんどアクセスされなくなる」データには、より安価なGlacier Deep Archiveが適しています。

D
初期段階のS3 Standard-IAが不適切である点はAと同じです。ただし、移行先のGlacier Deep Archiveは適切です。

### まとめ
頻繁にアクセスされるデータはS3 Standardに保存し、アクセス頻度が大幅に減少するタイミングでライフサイクルルールによりGlacier Deep Archiveに自動移行するのが最もコスト効率が高い。

### 正解
B

### 解説
Amazon Data Lifecycle Manager（DLM）は、EBSスナップショットの作成・保持・削除を自動化するサービスです。

- スケジュール（例：毎日、12時間ごと）に基づいてスナップショットを自動取得できる

D. S3バケットを作成した後、プロパティ変更画面でS3オブジェクトロックのガバナンスモードを有効にする。そのバケットの中にドキュメントを保存する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
A

### 解説
S3オブジェクトロックのコンプライアンスモードは、保持期間中はルートユーザーを含むすべてのユーザーがオブジェクトの削除・上書きをできないようにする機能です。

- WORM（Write Once Read Many）モデルを実現
- コンプライアンスモードでは、保持期間中は誰もオブジェクトを削除・変更できない
- 規制要件やコンプライアンス要件に対応するための厳格な保護を提供
- オブジェクトロックはバケット作成時にのみ有効化できる

### 他の選択肢が不適切な理由
B
Glacierのボールトロックは長期アーカイブ向けの機能であり、ドキュメント共有アプリケーションのようにアップロード直後から変更・削除を防ぐ用途には適しません。Glacierへの移行プロセスが介在するため、リアルタイムな保護が実現できません。

C
ガバナンスモードでは、特別なIAM権限（s3:BypassGovernanceRetention）を持つユーザーがオブジェクトを削除・変更できてしまいます。「すべてのユーザーが変更または削除できない」という厳格な要件を満たしません。

D
S3オブジェクトロックはバケット作成時にのみ有効化できます。既存のバケットに対して後からオブジェクトロックを有効にすることはできないため、この手順は実行できません。また、ガバナンスモードである点もCと同じ理由で不適切です。

### まとめ
すべてのユーザーによるオブジェクトの変更・削除を防ぐには、S3オブジェクトロックのコンプライアンスモードを使用する。ガバナンスモードでは特権ユーザーが保護を回避できるため、厳格な要件にはコンプライアンスモードが必要。

この移行に最適な構成方法はどれでしょうか。（2つ選択）

<details>
<summary>選択肢を見る</summary>

C

### 解説
この問題では、オンプレミスからEFSへのデータ移行方法と、EC2インスタンスの配置要件の2つを理解する必要があります。

**B（AWS DataSyncエージェント）**
- AWS DataSyncはオンプレミスとAWS間のデータ転送を自動化・高速化するサービス
- オンプレミスにDataSyncエージェントをインストールすることで、NFSファイルシステムからEFSへのデータ移行が可能
- データの整合性チェックや転送の最適化を自動で行う

**C（同じVPCに起動）**
- EFSにアクセスするEC2インスタンスは、EFSファイルシステムと同じVPC内に配置する必要がある

A
EFSはリージョン内の複数のAZにまたがってアクセス可能なファイルシステムです。EC2インスタンスはEFSと同じAZである必要はなく、同じVPC内であればどのAZからでもアクセスできます。

D
CloudWatchエージェントはメトリクスやログの収集を行うモニタリングツールであり、データ移行の機能はありません。

E
AWS Server Migration Service（SMS）は仮想マシン（VM）をAWSに移行するためのサービスであり、ファイルシステムのデータ移行には適しません。ファイルデータの移行にはDataSyncが適切です。

### まとめ

### 問題8

ある企業は、3TBのボリュームデータをオンプレミスのリポジトリに保有し、大量の印刷ファイルを保存しています。このリポジトリは年間500GBの容量が増加しているものの、ローカル環境の制約により、単一の論理ボリュームとして運用する必要があります。あなたはソリューションアーキテクトとして、ローカルストレージの制約を回避するために、このリポジトリをAmazon S3バケットに拡張することが求められています。その際、ローカルストレージをメインストレージとして利用して、S3バケットをバックアップとして利用します。

この要件を満たすために、AWS Storage Gatewayの構成をどのように設定すればよいでしょうか。

<details>
<summary>選択肢を見る</summary>

A. Amazon S3への移転スケジュールが設定されたスナップショットを利用するキャッシュ型ボリュームをAWS上に構成して、キャッシュを取得する仕組みをオンプレミス環境に設定する。

B. Amazon S3への移転スケジュールが設定されたスナップショットを利用する保管型ボリュームをAWS上に構成して、保管ボリュームをオンプレミス環境に設定する。

C. Amazon S3 Glacier（迅速アクセス）への移転スケジュールが設定されたスナップショットを利用するキャッシュ型ボリュームをAWS上に構成して、スナップショットをAWSの別リージョンに保管する。

D. Amazon S3への移転スケジュールが設定されたスナップショットを利用する仮想テープライブラリーをAWS上に構成して、仮想テープをAmazon S3バケットに保存する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
B

### 解説
Storage Gatewayのボリュームゲートウェイには、保管型（Stored Volumes）とキャッシュ型（Cached Volumes）の2種類があります。

**保管型ボリューム**
- データの完全なコピーをオンプレミスのローカルストレージに保持する
- 定期的にスナップショットをS3にバックアップする
- ローカルストレージがメインストレージとなり、低レイテンシーでデータにアクセスできる

**キャッシュ型ボリューム**
- データの実体はS3に保存され、頻繁にアクセスされるデータのみローカルにキャッシュする
- S3がメインストレージとなる

要件は「ローカルストレージをメインストレージ」「S3をバックアップ」であるため、保管型ボリュームが適切です。

### 他の選択肢が不適切な理由
A
キャッシュ型ボリュームはS3をメインストレージとして使用し、ローカルにはキャッシュのみを保持します。「ローカルストレージをメインストレージとして利用」という要件に合致しません。

C
キャッシュ型ボリュームである点がAと同じ理由で不適切です。また、Glacierへの移転やスナップショットの別リージョン保管は要件に含まれていません。

D
仮想テープライブラリー（VTL）はテープバックアップの代替として設計されたソリューションです。ボリュームデータの単一論理ボリュームとしての運用には適しておらず、iSCSIブロックストレージとして使用する保管型ボリュームとは用途が異なります。

### まとめ
ローカルストレージをメインとしつつS3にバックアップを取る構成には、Storage Gatewayの保管型ボリュームが最適。キャッシュ型はS3がメインとなるため、要件の「ローカルストレージをメインストレージ」と矛盾する。

</details>

### 問題9

ある企業は、社員のユーザープロファイルおよびアクセスログをAmazon S3バケットに保存しています。これらのデータは日常的にアップロードや変更が行われるため、ユーザーが誤ってS3バケット内のオブジェクトを削除するリスクが存在します。このため、適切な予防策を講じる必要がありますが、設定変更時にダウンタイムが発生し、業務に支障をきたすことは避けなければなりません。

S3バケット内のオブジェクトの誤削除を防ぐための最適な設定手順の組み合わせはどれでしょうか。（2つ選択）

<details>
<summary>選択肢を見る</summary>

A. S3バケットにおけるバージョニング機能を有効化する。

B. S3バケットにおける暗号化を有効化する。

C. S3バケットにあるMFA Deleteを有効化する。

D. S3バケットの設定をデータ削除不可とする。

E. S3バケットにIAMロールによる削除拒否を設定する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
A
C

### 解説
S3バケットの誤削除防止には、バージョニングとMFA Deleteの組み合わせが最適です。

**A（バージョニング）**
- オブジェクトを削除しても、削除マーカーが追加されるだけで実際のデータは残る
- 誤って削除した場合でも、以前のバージョンから復元が可能
- バージョニングの有効化はダウンタイムなしで設定できる

**C（MFA Delete）**
- オブジェクトのバージョンを完全に削除する際にMFA認証を要求する
- バージョニングの無効化にもMFA認証が必要になる
- 誤操作や不正アクセスによる完全削除を防ぐ追加の保護レイヤー

### 他の選択肢が不適切な理由
B
暗号化はデータの機密性を保護する機能であり、誤削除の防止とは無関係です。暗号化されたオブジェクトでも通常通り削除できます。

D
S3バケットには「データ削除不可」という直接的な設定は存在しません。オブジェクトロックで類似の機能は実現できますが、日常的にアップロードや変更が行われるデータに対して全面的な削除不可設定は業務に支障をきたします。

E
IAMロールによる削除拒否は設定可能ですが、すべてのユーザーに対して一律に適用すると、正当な削除操作も実行できなくなります。バージョニング+MFA Deleteの方が、誤削除からの復元と意図的な削除の両方に柔軟に対応できます。

### まとめ
S3の誤削除防止にはバージョニング（削除してもデータを復元可能）とMFA Delete（完全削除にMFA認証を要求）の組み合わせが最適。ダウンタイムなしで設定でき、業務への影響を最小限に抑えられる。

</details>

### 問題10

あなたはソリューションアーキテクトとして、EC2インスタンスにホストされたWebサーバーとリレーショナルデータベースによって構成される業務アプリケーションを起動しました。リレーショナルデータベースをAmazon EC2にホストするために、1つの500 GB EBS汎用ボリュームをアタッチした汎用インスタンスクラスを使用しています。パフォーマンスを確認すると、データベースへのI/O処理を向上させる必要があることが判明しました。このサービスをうまく動作させるには、ボリュームあたりの25,000ほどのIOPSのパフォーマンスが必要となります。

費用対効果の高い方法で、この要件を満たすためソリューションを選択してください。（2つ選択）

<details>
<summary>選択肢を見る</summary>

A. EC2インスタンスクラスのサイズを大きくする。

B. 2つのEBSボリューム（汎用）のRAID0構成を設定する。

C. 2つのEBSボリューム（汎用）のRAID1構成を設定する。

D. EBSボリューム（汎用）をプロビジョンドIOPSストレージ（io2 Block Express）に変更する。

E. PV AMIを利用してECインスタンスを再起動して、拡張ネットワークを有効化する。

</details>

<details>
<summary>正解と解説を見る</summary>

### 正解
A
B

### 解説
25,000 IOPSを費用対効果の高い方法で実現するには、2つの対策を組み合わせる必要があります。

**A（インスタンスクラスのサイズを大きくする）**
- EC2インスタンスにはEBS最適化スループットの上限がある
- 汎用インスタンス（例：t2やm5）の小さいサイズでは、EBSへのI/O帯域幅がボトルネックになる場合がある
- インスタンスサイズを大きくすることで、EBSへのI/O帯域幅が増加し、ボリューム側のIOPS性能を十分に活用できるようになる

**B（RAID 0構成）**
- RAID 0は複数のボリュームにI/Oを分散させるストライピング構成
- gp3ボリュームは1ボリュームあたり最大16,000 IOPSをサポート
- 2つのgp3ボリュームをRAID 0で構成すると、合計で最大32,000 IOPSが可能
- 25,000 IOPSの要件を費用対効果高く満たせる

### 他の選択肢が不適切な理由
C
RAID 1はミラーリング構成で、2つのボリュームに同じデータを書き込みます。冗長性は向上しますが、IOPSの向上にはつながりません。書き込みIOPSは1ボリュームと同等のままです。

D
io2 Block Expressは最大256,000 IOPSを提供する高性能ボリュームですが、gp3と比較して大幅にコストが高くなります。25,000 IOPSであればgp3のRAID 0構成で実現可能であり、「費用対効果の高い方法」という要件に照らすとオーバースペックです。

E
PV（Paravirtual）AMIは旧世代の仮想化方式であり、現在はHVM（Hardware Virtual Machine）AMIが推奨されています。PV AMIへの変更はパフォーマンス向上ではなく、むしろ性能低下につながる可能性があります。また、拡張ネットワークはネットワーク性能の向上であり、EBS IOPSの直接的な改善には関係しません。

### まとめ
EBSのIOPS向上を費用対効果高く実現するには、EC2インスタンスサイズの拡大（I/O帯域幅の確保）とgp3ボリュームのRAID 0構成（IOPSの合算）の組み合わせが最適。io2 Block Expressは高性能だがコストも高く、必要なIOPSが25,000程度であればgp3のRAID 0で十分対応可能。

</details>
